## 摘要
背景。自主编码智能体在软件开发中越来越广泛使用，将部分工程流程转移到AI辅助。虽然这种自动化带来了明显的好处，但也引入了治理、可追溯性和对智能体行为控制的挑战。智能体上下文文件（ACF）已成为通过结构化指令引导智能体的实用机制，然而关于这些工件如何被维护以及它们的演化如何与代码开发相关联，我们知之甚少。目标。本文计划研究ACF的演化及其在智能体驱动开发中的作用。具体而言，我们（1）通过基于软件维护理论的分类法对ACF变更进行分类，（2）分析不同类型的变更如何与代码质量结果相关联，（3）检查它们在开发生命周期中的时间模式。方法。我们进行了一项大规模挖掘研究，结合了包含ACF的仓库和智能体生成的提交。我们在提交级别重构ACF演化，使用定性方法对变更进行分类，并分析它们与代码质量指标的相关性。采用统计分析和假设来评估不同维护类别之间的差异，为未来设计用于管理自主编码智能体的ACF提供参考。

## Abstract
Context. Autonomous coding agents are increasingly used in software development, shifting parts of the engineering process to AI assistance. While this automation brings clear benefits, it introduces challenges in governance, traceability, and control over agent behavior. Agent Context Files (ACFs) have emerged as a practical mechanism to guide agents through structured instructions, yet little is known about how these artifacts are maintained and how their evolution relates to code development. Objective. This paper plans to investigate the evolution of ACFs and their role in agent-driven development. Specifically, we (1) classify ACF changes through a taxonomy grounded in software maintenance theory, (2) analyze how different types of changes are associated with code quality outcomes, and (3) examine their temporal patterns across the development lifecycle. Method. We conduct a large-scale mining study combining repositories with ACFs and agent-generated commits. We reconstruct ACF evolution at the commit level, classify changes using a qualitative approach, and analyze their association with code quality metrics. Statistical analyses and hypotheses are used to evaluate differences across maintenance categories, to inform future design of ACFs for governing autonomous coding agents.

*   **核心问题**：在软件工程中，自主编码智能体（Autonomous Coding Agents）的使用日益普及，它们通过遵循结构化指令来完成复杂的开发任务。**Agent Context Files (ACFs)**（如 `CLAUDE.md`, `AGENTS.md`）作为一种关键机制，用于定义和指导智能体行为，类似于“行为守则”或“治理 artifact”。然而，开发者**如何在实际项目中维护和演化这些 ACFs？其演化模式与代码质量及开发周期有何关联？** 这些问题尚未被深入研究。
*   **研究动机**：尽管已有研究关注了ACF的静态内容和智能体的行为影响，但对于**这些治理 Artifact 的长期、动态演化过程**的理解存在显著空白。理解ACF的演化是：
    *   洞悉开发者控制、调整和“驯化”智能体的挑战和策略。
    *   建立有效的智能体治理实践，以保障代码质量和项目可控性的基础。

#### 2. 论文提出的方法论：核心思想、关键技术细节

*   **核心思想**：将ACF的演化视为一种**软件维护活动**，并尝试用经典的软件维护理论（如SWEBOK v4.0中的修正、增强等）对其进行分类和解释。研究结合了**定性分析**（归纳主题分析）和**定量分析**（统计假设检验）。
*   **关键技术细节与流程**：
    1.  **数据采集与整合**：结合两个公开数据集：（1）ACF数据集（包含 2,303 个ACF文件和 1,925 个仓库）；（2）AIDev数据集（包含 116,211 个拥有智能体生成代码的仓库）。通过仓库标识符（owner/repo）整合，筛选出同时包含ACF和智能体提交的仓库。
    2.  **ACF演化重构与分类（定性部分 - RQ1）**：
        *   **样本提取**：从所有ACF修改提交中随机抽取统计代表性子集。
        *   **归纳主题分析**：由至少两名标注员独立审查每个选中的提交，检查提交信息、ACF变更差异（diff）、ACF的旧版本与新版本以及同一提交中的代码变更。通过开放式编码、迭代分组和讨论，形成**ACF变更的专属分类法**。
        *   **演绎解释**：将产生的ACF变更类别映射到经典的软件维护类别（如修正、预防、适应、完善、新增）。使用Cohen's Kappa系数评估标注者间一致性，目标值≥0.70。
        *   **大规模分类**：利用LLM作为“评判者”，结合分类法定义和标记好的样例，对剩余的大量ACF变更提交进行自动分类。
    3.  **窗口式分析（定量部分 - RQ2 & RQ3）**：
        *   **ACF窗口**：定义一个开发窗口，范围为**两次连续ACF修改提交之间**的所有智能体生成的提交。该窗口内所有代码都被视为受同一个ACF版本（即窗口起点修改后的ACF）的统治。
        *   **代码质量度量**：
            *   **结构性度量**：计算窗口内每个智能体提交与其父提交之间代码文件的**圈复杂度、代码行数、耦合度**的差值（Delta），并在窗口级别汇总。
            *   **过程性度量**：计算窗口内的**修正性提交概率（CCP）**，即被分类为修正bug的提交占窗口内总提交数的比例。
        *   **时间模式度量（RQ3）**：计算ACF修改提交的相对时间（基于提交序列位置和实际时间）以及修改频率（每100次提交或每单位时间内的修改次数）。

#### 8. 不足与局限

*   **数据源限制**：
    *   **仅限公共仓库**：结果可能不适用于私有、工业级项目。
    *   **代理偏倚**：仅分析**Claude Code, OpenAI Codex, GitHub Copilot**三种代理的数据。其他代理或未来代理的行为可能不同。

    *   结果仅适用于当前时间点的ACF和代理技术，该领域发展迅速，研究结论可能随新模型（如Agentic Reasoning改进）而变得过时。
