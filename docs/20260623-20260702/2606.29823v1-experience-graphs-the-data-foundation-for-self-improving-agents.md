## 摘要
数据库社区通过认识到新工作负载需要新系统架构，不断推动技术前沿。我们认为，长周期智能体任务——代码生成、科学发现、硬件设计——正是这样一种工作负载。这些智能体进行探索：它们生成工件、执行工具、观察失败、分支并修复，历经数百个步骤。这种搜索会产生一个结构化的对象，我们称之为经验图：可执行工件、工具输出、奖励、兄弟比较以及因果谱系。然而，现有的智能体框架将这种经验视为一次性状态——JSON检查点和会话日志，崩溃后无法恢复，无法跨用户查询，也无法物化为训练数据。我们提出Trellis：一个将经验图视为一等公民、可管理、可查询的数据库状态的数据基础。核心见解是，对经验图的搜索是一种数据库访问模式。前沿选择是一种查询，跨会话复用是向量种子驱动的图检索，训练数据提取是物化视图，而重构智能体在过去任意步骤所知道的内容是一种时间旅行查询。当数据库拥有经验图时，智能体变为无状态计算，崩溃恢复、水平扩展以及闭环训练飞轮成为架构上的副产品。我们在Meta的生产加速器内核优化器KernelEvolve中验证了该设计，其中跨会话复用实现了目标加速比大约快10倍，同时令牌成本降低52%。更广泛地说，Trellis将推理时的搜索从一次性计算转变为持久的机构资产：日志使数据库可靠；经验图可能使智能体具有累积性。

## Abstract
The database community has repeatedly advanced the state of the art by recognizing that new workloads demand new system architectures. We argue that long-horizon agentic tasks -- code generation, scientific discovery, hardware design -- are such a workload. These agents explore: they generate artifacts, execute tools, observe failures, branch, and repair over hundreds of steps. This search produces a structured object we call an experience graph: executable artifacts, tool outputs, rewards, sibling comparisons, and causal lineage. Yet existing agent frameworks treat this experience as disposable state -- JSON checkpoints and session logs that cannot be recovered after a crash, queried across users, or materialized into training data. We propose Trellis: a data foundation that treats the experience graph as first-class, governed, queryable database state. The core insight is that search over experience graphs is a database access pattern. Frontier selection is a query, cross-session reuse is vector-seeded graph retrieval, training-data extraction is a materialized view, and reconstructing what an agent knew at any past step is a time-travel query. When the database owns the experience graph, agents become stateless compute, and crash recovery, horizontal scaling, and a closed-loop training flywheel emerge as architectural byproducts. We ground the design in KernelEvolve, a production accelerator-kernel optimizer at Meta, where cross-session reuse reaches a target speedup roughly 10x faster at 52% lower token cost. More broadly, Trellis turns inference-time search from disposable computation into a durable institutional asset: logs made databases reliable; experience graphs may make agents cumulative.

- **核心问题**：长周期智能体任务（如代码生成、科学发现、硬件设计）在执行过程中会产生结构化的“经验图”——包含可执行工件、工具输出、奖励、兄弟比较、因果谱系等丰富信息。然而，现有的智能体框架（如 Claude Code、KernelEvolve 早期版本）将这些经验视为一次性临时状态，储存在 Python 对象、JSON 检查点或会话日志中。这种设计导致：崩溃后无法恢复、跨会话知识无法系统复用、训练数据需要事后从日志中爬取且缺乏完整谱系、横向扩展困难。  
- **整体含义**：本文认为，智能体的自我改进（RSI）本质上是一个数据库问题。需要一个持久化、可查询、可治理的数据基础，将经验图提升为“一等数据库对象”，使搜索模式转化为标准的数据库访问模式（如查询、时间旅行、物化视图），从而将推理时搜索从一次性计算转变为持久的机构资产，实现智能体的累积学习。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：提出 **Trellis** 数据基础，将经验图作为受管、可查询的数据库状态。关键在于：智能体搜索中的每个操作都可以映射为数据库访问模式——前沿选择是查询、跨会话复用是向量种子图检索、训练数据提取是物化视图、状态回放是时间旅行查询。  
- **两循环架构**：
  - **内循环**（Agent Session）：每个节点由一次内循环执行产生，包括技能驱动 → 调用 LLM 生成 → 工具执行评估 → 持久化结果。内循环是无状态的，只消耗任务描述和上下文。
  - **外循环**（Search Orchestration）：基于策略（MCTS、进化搜索、贪心等）从当前经验图中选择前沿节点，组装上下文，调用内循环，记录结果。外循环与具体技能无关，仅读写数据接口。
- **数据基础（Trellis）设计**：
  - **逻辑模型**：四表层次结构（tasks, sessions, nodes, prompt_history），支持 SQL、Cypher 和向量检索。大工件存储在对象存储中。
  - **物理架构**：存储与计算分离，采用 Axiom 优化器 + Velox 执行引擎，可同时服务子 50ms 的前沿查询、向量检索和列式训练提取。
  - **上下文作为受管状态**：通过 `prompt_history` 表持久化对话，子节点可继承父节点上下文或通过图遍历重构，避免文本断点。
  - **多版本状态与时间旅行**：字段级变更日志（按逻辑步骤号键控），支持 AS-OF 重构节点在各步的状态，确保训练数据不引入未来信息。
  - **并发搜索**：多个智能体可同时在同一棵树中工作，通过原子化操作处理 MCTS 回传冲突，容忍最终一致性。
- **训练数据提取**：SFT 轨迹、DPO 偏好对、GRPO 组作为物化视图直接从同一存储中查询生成，无需额外管道。价值模型可在经验图状态上学习外循环控制策略。

## 3. 实验设计：使用的数据集/场景、Benchmark、对比方法

- **场景**：Meta 的生产级加速器内核优化器 **KernelEvolve**，在四种硬件平台（NVIDIA、AMD、MTIA、CPU）上优化 AI 内核。
- **Benchmark**：固定步预算（100 步）、固定模型、固定工作线程数、固定贪心搜索策略。每个配置约 100 个节点。
- **对比方法**：设置三个实验组：
  - 基线（无跨会话记忆）
  - 跨会话记忆注入率 p=0.1
  - 跨会话记忆注入率 p=0.5

## 6. 论文的主要结论与发现

- **跨会话记忆显著提升性能**：
  - buggy 节点率从基线的 55% 降至 p=0.1 时的 34% 和 p=0.5 时的 21%。
  - 有效节点中达到基线加速比的比例从 79.5% 提升至 90.8% 和 100%。
  - 达到 1.2× 加速所需的步数从基线的 51 步降至约 5 步，实现约 **10× 加速**。
  - 每有效节点的令牌成本降低 **52%**（因避免 bug 调试重试循环）。
- **探索-锚定权衡**：高记忆注入率（p=0.5）导致搜索多样化显著下降（仅 8 种策略组合 vs 基线的 20 种），最优解来自无记忆（1.49× vs 1.36×）。平衡设置 p=0.1 提供最佳折中。
- **崩溃恢复**：数据库原生状态使得崩溃后自动从最后提交节点恢复，无节点丢失。
- **通用性**：相同 Trellis 基础重定向至 MTIA 硅验证（仅更换适应度函数和技能），证实架构领域无关性。
- **训练数据提取**：无需重新 rollout，生产会话同时作为服务和训练数据生成器。
