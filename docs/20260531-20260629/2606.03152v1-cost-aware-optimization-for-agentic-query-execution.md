## 摘要
经典查询优化搜索仅在成本上不同的代数等价计划。然而，一旦引入基于LLM的操作符，这一假设便不再成立：它们的位置、顺序和粒度共同决定了货币成本和答案质量，而替代方案中的正确选择往往只在运行时才显现。我们将这一设置形式化为代理查询执行，这是一种将基于代理的规划与执行交织在一起的查询执行范式，其中代理工作流优化成为经典查询优化的对应物。然后，我们提出了EnumGRPO，一个针对此设置的自改进优化器。在学习阶段，EnumGRPO在执行范式、操作符类型、操作符位置、选择性范围和投影宽度等决策上枚举查询计划，然后通过上下文强化学习将质量-成本反馈提炼为可重用的规划启发式。在SWAN的四个数据库上，EnumGRPO以每个查询0.011美元的LLM操作符成本实现了35.4%的执行准确率，与混合查询基线相比，成本降低了约317倍，同时答案准确率相对提高了18%。

## Abstract
Classical query optimization searches over algebraically equivalent plans that differ only in cost. This assumption breaks once LLM-backed operators enter the picture: their placement, ordering, and granularity jointly determine both dollar cost and answer quality, and the right choice among the alternatives is often revealed only at runtime. We formalize this setting as agentic query execution, a query execution paradigm in which agent-based planning is interleaved with execution, and agent workflow optimization becomes the analogue of classical query optimization. We then present EnumGRPO, a self-improving optimizer for this setting. During a learning stage, EnumGRPO enumerates query plans over decisions such as execution paradigm, operator type, operator placement, selectivity scope, and projection width, then distills quality-cost feedback into reusable planning heuristics via in-context reinforcement learning. Across four databases in SWAN, EnumGRPO achieves 35.4% execution accuracy at $0.011 per query in LLM-operator cost, a ~317x cost reduction over the hybrid query baseline with an 18% relative improvement in answer accuracy.

- **经典查询优化的局限**：传统查询优化假设所有候选计划在代数上等价，仅成本不同。但当引入基于大语言模型（LLM）的语义算子（如 `llm_map`, `llm_filter`）后，这一假设被打破——这些算子的**放置、顺序和粒度**共同决定了**货币成本（dollar cost）**和**答案质量**，且最优选择往往只有在运行时（观察到中间结果后）才能确定。
- **新范式的提出**：论文将这种交织了**执行与规划**的查询执行模式定义为 **代理查询执行（Agentic Query Execution）**，其核心优化问题转变为**代理工作流优化（Agent Workflow Optimization）**，即在运行时动态决定下一步调用哪个算子（SQL或LLM）、以何种方式处理哪些数据。
- **研究目标**：在保证答案质量的前提下，最小化LLM调用带来的高昂成本，实现成本与质量之间的最佳权衡。

### 2. 论文提出的方法论：核心思想、关键技术细节
- **核心思想**：将查询执行视为一个**马尔可夫决策过程（MDP）**，通过**在上下文中强化学习（In-Context Reinforcement Learning, ICRL）** 来优化代理策略——**不更新模型参数**，仅通过向系统提示（prompt）中注入可重用的规划启发式规则（heuristics）来改善决策。
- **关键技术**：
  - **计划空间结构化**：定义了5个正交的规划轴，每个轴对应一个经典查询优化概念：
    - **执行范式**（data-driven vs. code-driven）：逐行LLM调用 vs. 一次规则合成（类比JIT编译）。
    - **算子类型**（scalar vs. aggregate）：逐行处理 vs. 批处理（类比连接算法选择）。
    - **算子放置**（pre-aggregation vs. post-aggregation）：LLM在SQL聚合前或后执行（类比谓词下推）。
    - **选择性范围**（full vs. targeted）：全部输入 vs. 经SQL过滤后的最小子集（类比访问路径/物化）。
    - **投影宽度**（narrow vs. wide）：仅传递必要列 vs. 全部列（类比晚期物化）。
  - **EnumGRPO算法**：
    - **计划枚举**：每个查询生成N个不同的轴赋值（由LLM一次提出，保证多样性）。
    - **并行执行与奖励计算**：对每个赋值执行工作流，得到输出。复合奖励 = 质量得分（加权组合精确匹配EX、元组F1、单元格F1）– 成本惩罚（组内归一化的token数和步骤数）。
    - **组相对优势（Group-Relative Advantage）**：在同一查询的N个rollout组内对奖励进行z-score归一化，消除查询难度差异。
    - **经验蒸馏（Experience Distillation）**：四阶段LLM管道将轨迹对比转化为可读的启发式规则（ADD/UPDATE/DELETE），存储在**经验池 ε**中，追加到后续的系统提示中。
  - **无梯度更新**：所有改进通过修改提示文本实现，与具体LLM模型权重解耦，可跨模型升级迁移。

### 6. 论文的主要结论与发现
- **EnumGRPO达到最优平衡**：执行准确率35.4%（相对优于Agentic Text2SQL 84%、Agentic BlendSQL 18%），同时LLM算子成本仅为**$0.011/查询**，相比BlendSQL（$3.42/查询）降低**约317倍**，相比基础代理系统降低72%。
- **学习到的启发式有效**：经验池帮助代理更早确定策略（平均步数从17.0降至15.8），减少不必要的LLM调用和token消耗（token减少64.4%）。

- **可扩展性**：Agentic Query Execution + EnumGRPO在数据规模增大时，延迟和成本保持平稳甚至下降（因自适应选择更经济的路径），而BlendSQL则线性增长。
- **经验池分布**：执行范式（28%）和选择性范围（25%）是最关键的轴，占总经验的一半以上。

### 7. 优点
- **新颖性与理论贡献**：正式定义了“代理查询执行”范式和工作流优化问题，为LLM集成数据库提供了系统化的优化框架。
- **实用设计**：
  - 采用**上下文强化学习**，无需梯度更新，可移植至任何API级LLM。
  - **枚举+蒸馏**机制：通过结构化枚举保证对比信号的强度，通过自然语言启发式保持人类可审核性。
  - 多粒度奖励（EX, TupleF1, CellF1）缓解奖励稀疏，组内归一化消除查询难度差异。
- **实验充分**：多维度评估（准确率、成本、延迟、可扩展性、消融、迁移），并提供了开源代码和数据。

### 8. 不足与局限
- **绝对准确率仍低**：35.4%的执行准确率意味着大多数查询未能完全正确回答，在要求高可靠性的生产场景中不够成熟。
- **学习依赖初始数据**：需要一批带标注的问题（40个）进行离线学习；冷启动场景下尚无在线自适应机制。
- **枚举开销**：学习阶段需要枚举N个轴赋值（实验中N=5），若轴空间更大可能缩放不佳。
- **经验池静态**：训练完成后经验池固定，无法自动适应查询分布或数据模式的变化（论文提及持续学习作为未来方向）。
- **未测试更大规模或不同领域**：仅评估了SWAN的4个数据库，未覆盖更多样化的语义推理任务或更大数据量下的工程挑战（如并发、缓存）。
- **缺乏与更多基线对比**：未与Palimpzest、LOTUS等近期优化系统直接比较（可能是因为它们不适用于完全不确定的工作流场景）。
- **黑盒API依赖**：方法依赖于特定LLM API响应，不同模型可能产生不同行为，论文未充分论证启发式的通用性（虽然提示解耦了权重）。
