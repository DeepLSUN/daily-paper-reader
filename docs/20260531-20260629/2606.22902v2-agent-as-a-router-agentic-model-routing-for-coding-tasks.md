## 摘要
现实世界的用户通常可以使用来自不同提供商的多个大型语言模型（LLM），这些LLM往往在不同领域表现出色，但没有一个在所有领域都占优。因此，将每个任务路由到最合适的模型对于性能和成本都至关重要。现有的路由器将此视为一个静态的、一次性分类问题。然而，我们发现这些路由器的性能瓶颈在于信息不足：仅仅在任务维度层面用性能统计数据增强一个普通的LLM路由器，就能获得15.3%的相对增益，超过了基于相同维度层面先验知识的启发式路由器。受此发现启发，我们提出了Agent-as-a-Router框架，该框架将路由形式化为一个C-A-F循环（上下文→行动→反馈→上下文）。它通过在部署过程中积累基于执行的经验来缩小信息差距。我们将此框架实例化为ACRouter，由编排器、验证器、记忆模块组成，并引入了CodeRouterBench，一个包含约10K个任务实例的评估环境，这些任务实例具有8个前沿LLM的验证分数，从而支持基于遗憾的流式任务路由器比较。实验表明，ACRouter在分布内任务上实现了最低的累积遗憾，并推广到了分布外的智能体编程任务，证明了我们的路由框架主动缩小了信息差距。代码和基准测试已在https://github.com/LanceZPF/agent-as-a-router上发布。

## Abstract
Real-world users typically have access to multiple Large Language Models (LLMs) from different providers, and these LLMs often excel at distinct domains, yet none dominate all. Consequently, routing each task to the most suitable model becomes critical for both performance and cost. Existing routers treat this as a static, one-off classification problem. However, we identify the performance bottleneck for these routers as information deficit: simply augmenting a vanilla LLM router with performance statistics at the task-dimension level yields a 15.3% relative gain, surpassing a heuristic router built on the same dimension-level priors. Motivated by this finding, we propose Agent-as-a-Router, a framework that formalizes routing as a C-A-F loop (Context->Action->Feedback->Context). It closes the information gap by accumulating execution-grounded experience during deployment. We instantiate this framework as ACRouter, composed of an Orchestrator, a Verifier, a Memory module, and introduce CodeRouterBench, an evaluation environment comprising ~10K task instances with verified scores from 8 frontier LLMs, enabling regret-based router comparison on streaming tasks. Experiments show that ACRouter achieves the lowest cumulative regret on in-distribution tasks and generalizes to out-of-distribution agentic-programming tasks, demonstrating that our routing framework actively closes the information gap. Codes and benchmarks are released at https://github.com/LanceZPF/agent-as-a-router.

## 1. 核心问题与整体含义（研究动机和背景）

- **研究动机**：用户通常可以访问多个LLM（如Claude、GPT、Qwen等），这些模型在不同编程任务维度（代码生成、算法设计、调试、数据科学等）上表现各异，没有一个模型在所有领域占优。手动为每个任务选择最佳模型在规模上不可行，因此自动模型路由成为提升智能体性能的关键。
- **背景**：现有路由方法（如RouteLLM、LLMRouterBench）将路由视为静态分类问题：给定任务，一次性从候选模型池中选择一个。作者通过初步实验发现，这种静态路由器的性能瓶颈并非推理能力不足，而是**信息缺失**——路由器在决策时缺乏执行层面的反馈信息。仅向普通LLM路由器补充基于任务维度的性能统计数据，就能获得15.3%的相对提升（从41.41%到47.74% AvgPerf），甚至超过了使用相同维度先验的启发式路由器（DimensionBest，47.50%）。
- **核心问题**：如何让路由器在部署过程中主动获取并积累执行验证后的信息，从而动态缩小与理论最优（oracle）的差距？

## 2. 方法论：核心思想、关键技术细节、算法流程

### 核心思想：C-A-F循环（Context→Action→Feedback→Context）

- 作者将模型路由形式化为一个**上下文多臂赌博机（contextual multi-armed bandit）**问题，其中每个任务提供上下文，路由选择模型（拉臂），然后收到验证反馈（分数和成本），反馈被记忆并用于改进后续决策。
- 循环流程：
  - **Context c_i**：当前任务提示、元数据（维度、难度、语言）以及从Memory中积累的历史经验。
  - **Action a_i**：从模型池M={m1,...,mM}中选择一个模型。
  - **Feedback f_i**：运行该模型输出后由Verifier给出的统一性能分数和货币成本。
  - **Memorize**：将反馈存储到Memory中，用于构建下一个任务的Context c_{i+1}。
- 每任务的奖励定义：r_i(a_i) = ε1·s_i(a_i) + ε2·κ_i(a_i)，其中ε1>0奖励性能，ε2<0惩罚成本。本文默认使用ε1=1, ε2=-0.1。
- **累积遗憾（Cumulative Regret）**：定义为流式任务上每任务遗憾之和，δ_i = r*_i - r_i(a_i)，其中r*_i是oracle（对每个任务选择奖励最大的模型）的奖励。累积遗憾越低，路由越接近最优。

### 关键技术细节：ACRouter实例化

ACRouter由三个核心模块组成：

1. **Orchestrator（编排器）**：负责做出路由决策。它集成多种信号：
   - DimensionBest先验（来自探测集的维度级最佳模型）。
   - Memory中检索到的top-10历史邻居（基于任务嵌入的余弦kNN）。
   - 任务元数据。
   - 核心策略是一个微调的Qwen3.5-0.8B模型，结合启发式规则通过加权投票做出最终选择。
2. **Verifier（验证器）**：在沙箱中执行模型输出，聚合多种信号（AST解析、沙箱执行、prompt内测试、LLM-as-Judge）得到统一性能分数u_i∈[0,1]。
3. **Memory（记忆模块）**：在线向量存储，键为任务嵌入（如voyage-code-3或BGE-large），值为选择的模型、性能、成本及验证轨迹。检索采用余弦kNN，容量FIFO限制为20K条。每次路由后更新。

**其他基线路由策略**：通过限制或移除上述模块，可得到不同类别：
- 单模型（无Orchestrator/Verifier/Memory）。
- 静态启发式（冻结Memory，如DimensionBest、kNN检索）。
- 静态训练策略（训练Orchestrator，无Memory和Verifier，如LogReg、RouteLLM、微调Qwen）。
- 动态在线赌博机（参数化Memory，argmax Orchestrator，仅奖励Verifier，如LinUCB、LinTS）。

## 6. 主要结论与发现

1. **信息缺失是瓶颈**：LLM路由器的性能瓶颈是信息不足而非推理能力不足。仅加入维度级性能统计，零样本路由提升15.3%，超越包含相同信息的最优启发式。
2. **无单一模型在所有编程维度占优**：模型之间存在互补性（如GLM-5在算法设计优于Opus 86%，Qwen3-Max在测试生成优于Opus 111%，Kimi-K2.5在数据科学优于Opus 30%）。成本差异巨大（最贵比最便宜>20倍）。
3. **ACRouter取得最佳性能**：
   - 在**ID测试**上，AvgPerf 49.98%（最高），CumReg 205.5（最低），优于DimensionBest（47.50%, 277.4）和所有其他路由。
   - 在**OOD测试**上，AvgPerf 62.50%（最高），CumReg 17.0（最低），显著优于最强单模型Always-Opus（57.14%, 26.7）和微调Qwen（55.36%, 27.2）。
   - 成本效率高于Always-Opus（Perf/$ 3.79 vs 1.29 ID；1.18 vs 0.64 OOD）。
4. **静态学习器泛化失败**：分布内接近DimensionBest，但OOD AvgPerf降至8.93%-21.43%，低于随机（31.25%），显示严重过拟合。
5. **在线适应性是泛化关键**：LinUCB/LinTS在OOD仍达46-50%，而静态方法崩溃，说明持续学习的重要性；ACRouter因更丰富的Memory/Orchestrator领先。

## 7. 优点

1. **问题诊断明晰**：通过简洁的消融实验，暴露信息缺失是核心瓶颈，而非推理能力，指导了方法设计。
2. **理论框架完整**：将路由形式化为C-A-F循环，与上下文赌博机理论对接，使用累积遗憾作为流式度量，提供了统一视角，可以涵盖所有现有方法。
3. **系统实现完整**：ACRouter包含Orchestrator、Verifier、Memory及工具层，模块化设计便于扩展和替换组件。
4. **基准构建严谨**：CodeRouterBench包含10维约10K任务，预计算8个模型在所有任务上的执行验证分数，支持公平的遗憾计算。OOD测试（智能体编程）验证了泛化能力。
5. **实验覆盖全面**：从诊断、规模缩放、方法族对比到OOD泛化，实验设计科学，结果清晰支持结论。
6. **实用性导向**：代码和基准开源，提供了构建自己路由器的实战指南（7步法），同时考虑了成本和部署便利性（如0.8B小模型微调即有效）。

## 8. 不足与局限

1. **成本估算局限**：货币成本基于官方API定价和实测token用量，但忽略了服务端缓存命中率（如Prompt caching）等实际优化，因此成本仅作为相对比较的参考指标，并非实际支出。
2. **OOD评估预算限制**：智能体编程任务使用40步限制（而非标准250步），虽然路由器比较的相对结论不变，但可能低估了更长时间预算下的性能差异。
3. **Memory容量与检索**：使用20K FIFO和kNN（k=10），未探索更先进的参数级别记忆技术（如神经图灵机、联想记忆），可能限制了长程泛化能力。
4. **单域聚焦**：目前仅面向编码任务，虽然作者声称框架可扩展至工具选择、策略选择等，但未提供实证。迁移到其他领域（如自然语言、多模态）的有效性未知。
5. **Verifier设计**：验证器依赖沙箱执行和LLM-as-Judge，可能受限于沙箱覆盖率和评判质量；对于无法执行的任务（如代码理解），信号质量较低。
6. **模型池固定**：论文假设模型池在部署期间不变；实际上新模型可能动态加入，框架需要适应增量扩展，但本文未讨论。
7. **训练数据细节缺失**：微调Qwen3.5的具体训练超参数、收敛状态、训练轮数等未报告，影响可复现性。
8. **随机种子影响**：虽然bandit方法报告了5种子平均，但ACRouter和其他相关实验是否多次随机运行未完全说明（可能单次），可能引入方差。
