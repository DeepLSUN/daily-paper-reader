## 摘要
搜索API是许多代理的基本检索层，也是最常用的工具。传统搜索API提供URL、标题和摘要来预览网站内容。由于整页检索令牌密集，代理检索架构越来越多地采用渐进式披露：代理先看到摘要，然后选择是否获取完整页面。在此类系统中，搜索API的性能通常主要根据答案准确性来评估。我们认为，商业搜索API更适合理解为决策面：排序的摘要、URL和元数据决定了代理是立即回答、再次搜索，还是花费令牌打开页面。我们通过一个冻结的GPT-5.4代理、两个工具（search_web和fetch_page）以及来自SEALQA-HARD的100个问题来验证这一主张，仅改变搜索提供商（Brave、Tavily、Firecrawl）。一个Kimi-K2.6神谕为代理可见的每个内容元素（URL、标题、摘要以及获取时的页面）打标签，产生6,869个每URL有效判断。我们使用一个审计过的正确答案标签——语义匹配，它在接受无害格式和命名变体的同时保留精确匹配。在该度量下，各提供商接近（25、25、26/100），但其证据经济差异显著：Brave提供富含黄金答案的摘要，Tavily将黄金支持URL集中于排名1，而在此固定代理策略下，Firecrawl与更广泛的探索相关联。我们还引入了一个表面矛盾与黄金URL比率，该比率从0.92到2.59不等。因此，提供商选择是一个检索预算和策略决策，而不仅仅是召回决策。

## Abstract
Search APIs are the fundamental retrieval layer for many agents and are often their most frequently used tool. Traditional search APIs provide URLs, titles, and snippets that preview website contents. Because full-page retrieval is token-intensive, agent retrieval architectures increasingly use progressive disclosure: the agent first sees snippets and then chooses whether to fetch full pages. In such systems, search API performance is often evaluated primarily by answer accuracy. We argue that a commercial search API is better understood as a decision surface: the ranked snippets, URLs, and metadata that determine whether an agent answers immediately, searches again, or spends tokens opening pages. We test this claim with one frozen GPT-5.4 agent, two tools (search_web and fetch_page), and 100 questions from SEALQA-HARD, varying only the search provider (Brave, Tavily, Firecrawl). A Kimi-K2.6 oracle labels every content element visible to the agent (URL, title, snippet, and fetched page, when fetched), producing 6,869 valid per-URL judgments. We use an audited correct-answer label, semantic match, which preserves exact matches while accepting harmless formatting and naming variants. Under this measure, the providers remain close (25, 25, 26 / 100), but their evidence economies differ sharply: Brave offers gold-answer-rich snippets, Tavily concentrates gold-supporting URLs at rank 1, and Firecrawl is associated with broader exploration under this fixed agent policy. We also introduce a surface contradiction-to-gold URL ratio, which varies from 0.92 to 2.59. Provider choice is therefore a retrieval-budget and policy decision, not merely a recall decision.

## 1. 论文的核心问题与整体含义（研究动机和背景）
- **研究动机**：传统搜索API评估主要依赖答案准确率，认为不同搜索提供商在准确率相近时可相互替换。然而，对于逐步披露（progressive disclosure）的tool-using agent，搜索API不仅提供URL列表，还提供了决定agent行为的“决策表面”（decision surface）：排序的摘要、URL和元数据会直接影响agent是立即回答、再次搜索还是花费token获取页面。本文旨在揭示这种“准确率相同但证据经济不同”的现象。
- **核心问题**：商业搜索API应被视为agent的决策表面，而非静态排序检索器。提供商会显著改变agent的证据暴露、矛盾引入和检索预算消耗，即便最终准确率接近。
- **整体含义**：评估搜索API不能只看召回率或准确率，必须考虑其对agent动作策略和内部推理过程的影响。提供商选择本质上是检索预算和策略决策。

## 2. 论文提出的方法论：核心思想、关键技术细节
- **核心思想**：固定agent（GPT-5.4）、工具集（search_web + fetch_page）、prompt、最大迭代次数（10轮）、judge（Kimi-K2.6）和后端（Jina Reader抓取页面），仅改变搜索提供商（Brave、Tavily、Firecrawl），然后通过“每URL oracle”标注所有agent可见的证据。
- **关键技术细节**：
  - **每URL oracle协议**：使用Kimi-K2.6对每个出现在轨迹中的URL进行独立标注，区分摘要表面（pre-fetch）和获取后页面（post-fetch）的支持情况。输出JSON字段包含：contains_gold_answer、gold_answer_in_snippets、gold_answer_in_extracted_page、supports_model_answer、contradicts_gold_answer、is_garbage等。
  - **支持拆分**：预获取表面支持（pre-fetch support）指摘要或标题/URL本身包含答案；后获取发现支持（post-fetch discovered support）指仅在页面获取后才发现支持。
  - **决策分区**（decision partition）：根据预获取支持和agent是否获取URL，将provider-query对分为四类：
    - SMART：存在预获取支持且agent获取了至少一个支持URL。
    - MISSED：存在预获取支持但agent未获取任何支持URL。
    - BLIND：无预获取支持但agent获取了URL。
    - NO-OP：无预获取支持且agent未获取任何URL。
  - **表面矛盾与黄金比率**（surface contradiction-to-gold ratio, rc:g）：摘要级别中矛盾黄金答案的URL数与包含黄金答案的URL数之比，独立于模型最终答案。
  - 答案正确性使用**语义匹配**（semantic_match）：在保留精确匹配的同时，接受无害的格式和命名变体（如大小写、冠词、数字词映射）。

## 6. 论文的主要结论与发现
- **准确率相近但证据经济迥异**：三个提供商在语义匹配下分别正确25、25、26个问题，但内部证据暴露差异巨大。
- **Brave**：预获取支持最多（30个查询），摘要富含黄金答案，但许多支持被MISSED（未获取），表面矛盾率最低（rc:g=0.92）。
- **Tavily**：预获取支持较少（16个查询），但黄金支持URL高度集中在第1位（50%），适合“取第一条”策略。矛盾率中等（rc:g=1.87）。
- **Firecrawl**：预获取支持少（16个查询），导致更多BLIND探索（81%查询至少获取一次页面），页面发现支持较多，但矛盾率最高（rc:g=2.59）。
- **正确性互补而非重叠**：仅10个问题被三个提供商同时答对，12个被两个答对，22个仅一个答对，表明不同提供商解决了不同问题。
- **可见答案文本并不保证正确回答**：即使黄金答案字符串出现在轨迹中，仍有约56%的查询回答错误，说明决策表面决定了答案是否被“注意、信任和行动”。
- **提供商选择是策略决策**：提供商应与其fetch策略匹配（如Brave适合较少获取，Tavily适合top-1获取，Firecrawl需要广泛探索）。

## 7. 优点：方法或实验设计亮点
- **概念创新**：首次将搜索API形式化为“决策表面”，区分预获取和后获取证据，提供新的分析维度。
- **细粒度oracle**：每URL独立标注支持/矛盾/无用，人验证一致率94%，为诊断提供可靠基础。
- **多维度评估**：超越单一准确率，引入支持次数、排名集中度、矛盾率、决策分区、互补性等，揭示隐藏差异。
- **严格控制**：固定所有其他变量（模型、工具、需求、后端、迭代限制），使提供商差异清晰归因。
- **实用导向**：提供可直接计算的指标（rc:g、决策分区），便于从业者在候选提供商间进行策略选择（例如判断是“摘要型”还是“探索型”提供商）。
- **诚实透明**：报告样本量限制、配对bootstrap区间、不对称性处理、人工审计细节，避免过度声称。

## 8. 不足与局限
- **单一agent和judge**：仅使用GPT-5.4和Kimi-K2.6，不同模型可能改变行为（例如更小的模型可能更依赖摘要）。人类验证仅由单标注员完成，非交叉标注。
- **样本量小**：100个问题，每个决策分区中一些单元（如SMART）仅3个样本，导致置信区间宽。作者明确强调诊断性而非结论性。
- **提供商覆盖率有限**：仅三个提供商，且配置为禁用原生页面提取，虽然保证公平，但可能掩盖某些提供商的真实特性（如Brave默认提供额外摘要片段，已合并处理但影响token量）。
- **表面不对称**：Brave返回更多原生摘要文本，部分解释其预获取优势。表4的长度归一化显示该因素部分解释但非全部。
- **后获取支持下界**：未获取的URL没有页面判断，无法评估全召回下的提供商召回能力。
- **观察性而非因果性**：agent动作未随机化，不能严格证明提供商直接导致特定行为（例如Firecrawl导致更多BLIND探索可能是因果也可能是agent适应了其特性）。
- **仅限事实问答**：结果不一定泛化到需要多步骤推理、创意生成或交互式代理场景。
- **token计量细节**：重复搜索观察被计数，影响每token支持密度的准确解释（作者已说明）。
- **伦理风险**：可能被误读为提供商排行榜，需强调本文不主张单一“最佳”提供商。
