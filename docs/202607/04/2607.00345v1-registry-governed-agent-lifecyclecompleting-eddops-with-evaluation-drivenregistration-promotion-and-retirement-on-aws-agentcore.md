## 摘要
企业采用大语言模型智能体需要兼顾质量、可靠性、安全性、延迟和成本的模型选择方法。评估驱动开发与运维（EDDOps）将评估定位为贯穿智能体生命周期的持续治理功能，而非终端检查点。本文在 AWS Bedrock AgentCore 上提出了面向实践者的 EDDOps 实例化方案，并提出了一个用于企业智能体架构中基础模型选择的成本-性能框架。我们做出了三项贡献：一是概念综合，解释为什么传统的 TDD/BDD 方法对于非确定性的大语言模型智能体是不够的；二是将 EDDOps 参考架构映射到 AgentCore 运行时、评估、智能体注册表和 CloudWatch 可观测性上；三是通过概念验证比较了两种部署路径下的三个基础模型，从而验证了一个经验性的成本-性能决策框架。利用来自六个智能体的 30 次单轮调用、9 次多轮评估以及注册表集成治理的追踪数据，我们展示了评估证据如何将模型选择从基准排名练习转变为受治理的经济决策。结果表明，当托管智能体平台提供追踪原生的可观测性、可插拔的评估框架以及基于注册表的治理发现时，它们可以支持 EDDOps。

## Abstract
Enterprise adoption of LLM agents requires model selection methods that balance quality, reliability, safety, latency, and cost. Evaluation-Driven Development and Operations (EDDOps) positions evaluation as a continuous governing function across the agent lifecycle rather than a terminal checkpoint. This paper presents a practitioner-oriented instantiation of EDDOps on AWS Bedrock AgentCore and proposes a cost-to-performance framework for selecting foundation models in enterprise agent architectures. We make three contributions: a conceptual synthesis explaining why traditional TDD/BDD methods are insufficient for non-deterministic LLM agents; an architectural mapping of the EDDOps reference architecture onto AgentCore Runtime, Evaluations, Agent Registry, and CloudWatch observability; and an empirical cost-to-performance decision framework validated through a proof-of-concept comparing three foundation models across two deployment paths. Using trace data from 30 single-turn invocations across six agents, 9 multi-turn evaluations, and registry-integrated governance, we show how evaluation evidence can convert model selection from a benchmark-ranking exercise into a governed economic decision. The results suggest that managed agent platforms can support EDDOps when they provide trace-native observability, pluggable evaluator frameworks, and governed registry-based discovery.

## 2. 论文提出的方法论

### 核心思想
- 提出一个五状态生命周期状态机（DRAFT → APPROVED → PUBLISHED → DEPRECATED → RETIRED），每个状态转换必须由评估证据门控。
- 注册表作为控制平面：智能体在DRAFT状态注册，注册表通过评估证据驱动状态转换，而非仅记录外部评估结果。
- MCP（模型上下文协议）本地发现与生命周期状态耦合：仅PUBLISHED状态的智能体出现在发现结果中，实现自治式治理。

### 关键技术细节
- 状态机形式化定义：`L = (S, T, E, δ)`，其中`S`为五状态集合，`T`为转换集合，`E`为评估证据，`δ`为转移函数，且仅当证据满足目标状态的门谓词时转换才定义。
- 资格门限：忠诚度≥0.80、正确性≥0.90、工具准确性≥0.90、帮助性≥0.70、延迟≤15秒、危害率≤5%。
- 成本-性能决策框架：
  - **Weighted Quality Score (WQS)**：`WQS = Σ(w_i * s_i)`，权重按HR/合规领域分配（任务成功0.25，忠诚度0.20，帮助性0.15，连贯性0.15，工具可靠性0.15，多轮鲁棒性0.10）。
  - **Cost-Adjusted Performance (CAP)**：`CAP = WQS / (每次交互成本) * 100`。
  - **Total Cost of Agent Ownership (TCAO)**：`TCAO = C_interact + C_eval + C_registry + C_version`，包含交互成本、评估成本、注册操作成本、版本开销。
  - **生命周期调整后CAP (L-CAP)**：`L-CAP = WQS / (TCAO/n) * 100`。
- 注册记录包含评估元数据（最后评分、时间戳、评估者、门状态、过期天数）和谱系链接（supersedes/superseded_by）。

### 算法/流程（文字描述）
1. 智能体在AWS AgentCore注册表以DRAFT状态创建。
2. 离线评估运行，通过Lambda检查门限。
3. 通过资格门限后提交审批（自动），然后更新为APPROVED，经人工签字后升级为PUBLISHED。
4. 在线评估持续监控；过期策略触发重新评估：若过期7天且重新评估失败则进入DEPRECATED。
5. DEPRECATED状态经30天宽限期后自动退役为RETIRED。
6. MCP发现查询仅返回PUBLISHED状态智能体。
