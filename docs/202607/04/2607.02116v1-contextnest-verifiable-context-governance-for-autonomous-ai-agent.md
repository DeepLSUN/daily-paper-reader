## 摘要
自主AI代理日益依赖外部知识存储，然而大多数检索管道仅提供相关性，缺乏对来源、版本标识、完整性、可追溯性或时间点重建的持久保证。我们将此形式化为上下文治理，并提出ContextNest，这是一套面向经治理的AI可消费知识库的开放规范与参考实现。ContextNest并不替代检索增强生成（RAG）；它在检索之下提供治理层，在检索系统操作之前，确定哪些工件是经批准的、当前的、可归因的且完整性经过验证的。

该规范结合了带元数据的类型化Markdown文档、确定性集代数选择器、contextnest:// URI引用、SHA-256哈希链式版本历史、图级检查点、通过模型上下文协议（MCP）实现实时数据的源节点，以及代理上下文消费的审计追踪。这些机制使组织能够重建哪些知识版本影响了代理输出，以及这些版本在被消费时是否符合AI使用资格。

我们报告了来自两项对照实验的首批实证结果。在一个隔离治理与检索失败模式的过时版本攻击中，经治理的选择严格帕累托支配BM25稀疏检索，在约三分之一的输入令牌成本下，实现了更高的答案质量通过率（97%对比93-90%）。在一个包含1060篇文档的检索确定性实验中，确定性选择器和BM25在重复相同查询时返回稳定的文档集（Jaccard指数1.0），而基于稠密+HNSW的基线在80%的查询上呈现非确定性（平均Jaccard指数0.611，最差情况0.210）。这些结果表明，上下文治理解决了单独依靠检索质量无法解决的失败模式。我们以开放许可形式发布了核心引擎、CLI和MCP服务器。

## Abstract
Autonomous AI agents increasingly depend on external knowledge stores, yet most retrieval pipelines provide relevance without durable guarantees of provenance, version identity, integrity, traceability, or point-in-time reconstruction. We formalize this as context governance and present ContextNext, an open specification and reference implementation for governed AI-consumable knowledge vaults. ContextNext does not replace Retrieval-Augmented Generation (RAG); it supplies the governance layer beneath retrieval, determining which artifacts are approved, current, attributable, and integrity-verified before retrieval systems operate over them.   The specification combines typed Markdown documents with metadata, deterministic set-algebraic selectors, contextnest:// URI references, SHA-256 hash-chained version histories, graph-level checkpoints, source nodes for live data through the Model Context Protocol (MCP), and audit traces of agent context consumption. These mechanisms let organizations reconstruct which knowledge versions informed an agent output and whether those versions were AI-eligible when consumed.   We report first empirical results from two controlled experiments. In a stale-version attack isolating the governance-versus-retrieval failure mode, governed selection strictly Pareto-dominates BM25 sparse retrieval, with higher answer-quality pass rate (97% versus 93-90%) at about one-third the input-token cost. In a retrieval-determinism experiment over a 1,060-document corpus, deterministic selectors and BM25 return stable document sets across repeated identical queries (Jaccard 1.0), while a dense+HNSW baseline is non-deterministic on 80% of queries (mean Jaccard 0.611, worst case 0.210). These results suggest that context governance addresses failure modes retrieval quality alone is not designed to resolve. We release a core engine, CLI, and MCP server under open licenses.

## 1. 核心问题与整体含义（研究动机与背景）

- **核心问题**：自主AI代理日益依赖外部知识库做决策，但现有RAG等检索管道仅关注“相关性”（relevance），缺乏对知识**出处（Provenance）**、**版本标识（Version Identity）**、**完整性（Integrity）**、**可追溯性（Traceability）** 和**时间点重建（Temporal Reconstruction）** 的可验证保证。
- **整体含义**：这导致了“上下文治理鸿沟（Context Governance Gap, CGG）”。代理可能引用过时、未经批准、被篡改的知识做出看起来自信但错误的决策，且无法事后审计。论文认为，**检索 ≠ 治理**，需要专门的治理层来确保知识消费的可靠性。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：设计一个位于RAG检索层之下的**治理层**，将“哪些文档是批准的、当前的、可归因的、完整性已验证的”从检索逻辑中分离出来。治理层通过确定性机制筛选出符合资格的文档，RAG则在该子集上进行语义检索。
- **关键技术细节**：
  - **类型化Markdown文档**：文档带有YAML frontmatter，定义了8种节点类型（如document、snippet、glossary等），以及`draft`/`published`状态。
  - **确定性选择器（Selector Grammar）**：采用集合代数式的查询语言（`+`交集，`|`并集，`-`差集），基于标签、URI、类型、状态等结构化元数据进行查询，结果完全确定（Property 4）。
  - **可寻址URI方案**：`contextnest://path@checkpoint#anchor` 支持浮动（指向最新版）和固定（指向特定检查点）引用，及跨保险库联邦引用。
  - **SHA-256哈希链版本历史**：每个文档版本形成哈希链，内容、作者、时间戳、版本号都被绑定，后验可检测篡改（Property 3）。
  - **图级检查点（Checkpoint）**：每次发布事件触发，记录所有发布文档版本哈希的原子快照，支持重建任意时刻的完整知识图状态（Property 6）。
  - **MCP源节点**：用于集成实时数据（如Jira工单），其生命周期通过“阶段性（Staged）状态”机制支持审计完整性。
  - **审计追踪**：每次上下文消费都输出包含版本号、检查点、作者等信息的结构化记录，支持追溯（Property 5）。
  - **托管模型**：分层托管（文档→文件夹→标签→保险库）和角色格（Viewer/Editor/Reviewer），在治理模式下强制**职责分离（SoD）**：作者不能批准自己的版本。

## 3. 实验设计

- **数据集/场景**：
  - **陈旧版本攻击实验**：基于一个小型合成的保险库（11篇已发布文档 + 6篇过时v2版），构建30个针对性查询。过时版在5个具体事实上与当前版矛盾。
  - **检索确定性实验**：合成一个**1,060篇文档**的语料库。基于10个模板（如部署回滚手册、API设计标准、ADR等）在106个服务变体上进行复制。构建50个查询，每个查询精确指向某个服务的特定事实。
- **基准方法（Benchmark）**：
  - **陈旧版本攻击**：对比（1）上下文选择器（`ctx resolve`）；（2）**BM25（泄漏版）**：索引`.versions/`目录中的陈旧文档；（3）**BM25（干净版）**：仅索引当前发布文档。
  - **检索确定性**：对比（1）上下文选择器；（2）**BM25**（k=3）；（3）**稠密+HNSW**（`bge-small-en-v1.5`嵌入，FAISS HNSW，参数模拟生产环境，含插入顺序随机化）。
- **评估指标**：
  - 陈旧版本攻击：LLM法官（Claude Opus 4.7）评判的`PASS/FAIL`通过率和平均输入tokens。
  - 检索确定性：每个（查询，方法）对重复20次，计算20次结果集合间的平均Jaccard相似度。Jaccard=1.0视为完全确定。

## 6. 主要结论与发现

- **陈旧版本攻击**：上下文选择器严格帕累托占优BM25，在**更低token成本**（约1/3）下取得**更高通过率**（97% vs 90-93%）。论文展示了BM25的两种失效模式：
    - **过时版本中毒**：BM25（泄漏版）检索到过时文档，模型自信地给出已被替代的错误答案。
    - **检索缺失**：BM25（干净版）因文本重叠不足，漏检精确匹配话题的文档。
- **检索确定性**：选择器和BM25在所有查询上完全确定（Jaccard=1.0）。稠密+HNSW在80%的查询上非确定，平均Jaccard仅0.611，最差情况0.210。这表明基于HNSW的稠密检索可能引入不可靠性，损害审计复盘能力。
- **核心结论**：上下文治理能解决检索质量单独无法应对的失效模式（特别是版本相关和时间相关的问题）。治理层与RAG应**互补**，共同提供可靠、可审计的上下文。
