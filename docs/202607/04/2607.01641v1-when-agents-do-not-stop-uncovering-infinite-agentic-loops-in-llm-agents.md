## 摘要
LLM Agent越来越依赖迭代执行，通过规划、工具使用、状态更新和Agent协作来解决问题。虽然这种设计实现了灵活的自动化，但也产生了一类新的故障：当反馈路径没有有效边界时，Agent可能会重复执行模型调用、工具、工作流转换或Agent交接。我们将此问题称为无限Agent循环（Infinite Agentic Loops, IALs）。IALs并非普通的编程循环；它们源于Agent逻辑、框架语义、运行时观察和终止机制之间的交互。此类故障可能将单个请求放大为长时间运行的模型和工具执行，导致成本耗尽、模型拒绝服务、上下文增长以及重复的外部副作用。我们提出IAL-Scan，一种用于检测真实世界LLM Agent项目中IAL故障的静态分析工具。IAL-Scan将异构的Agent代码抽象为框架无关的Agent IR，构建Agent循环依赖图（ALDG）以恢复显式和框架引入的反馈路径，并检查这些路径是否能在没有有效边界的情况下反复达到高成本或状态增长的操作。我们在6,549个LLM Agent仓库上评估IAL-Scan。它报告了74个潜在发现，其中人工审查确认了47个项目中的68个IAL故障，实现了91.9%的精确率。

## Abstract
LLM agents increasingly rely on iterative execution to solve tasks through planning, tool use, state updates, and agent collaboration. While this design enables flexible automation, it also creates a new class of failures: an agent may repeatedly execute model calls, tools, workflow transitions, or agent handoffs when the feedback path is not effectively bounded. We call this problem Infinite Agentic Loops (IALs). IALs are not ordinary programming loops; they arise from the interaction between agent logic, framework semantics, runtime observations, and termination mechanisms. Such failures can amplify a single request into long running model and tool execution, causing cost exhaustion, model denial of service, context growth, and repeated external side effects. We propose IAL-Scan, a static analysis tool for detecting IAL failures in real-world LLM agent projects. IAL-Scan abstracts heterogeneous agent code into a framework independent Agent IR, builds an Agentic Loop Dependence Graph (ALDG) to recover explicit and framework induced feedback paths, and checks whether these paths can repeatedly reach costly or state growing operations without an effective bound. We evaluate IAL-Scan on 6,549 LLM agent repositories. It reports 74 potential findings, among which manual review confirms 68 IAL failures across 47 projects, achieving 91.9% precision.

## 1. 核心问题与整体含义（研究动机和背景）

- **研究动机**：LLM Agent（如基于ReAct范式的系统）通过迭代执行（感知-推理-行动循环）实现自动化。然而，当反馈路径没有有效边界时，Agent可能陷入**无限Agent循环（Infinite Agentic Loops, IAL）**，反复触发LLM调用、工具执行、工作流转或Agent交接，导致成本耗尽、模型拒绝服务、上下文窗口膨胀和重复的外部副作用。
- **背景**：现有框架（LangChain、LangGraph、AutoGen等）虽提供max_iterations、recursion_limit等终止机制，但开发者常遗漏、误用或配置无效边界，且模型输出、工具结果等动态因素可绕过弱终止条件。社区已报告实际案例，但缺乏预部署检测工具。
- **论文意义**：首次系统定义IAL故障，提出专用静态分析工具IAL-Scan，填补LLM Agent安全分析在循环故障检测上的空白。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：将异构Agent代码（不同框架、自定义包装）抽象为统一的**Agent IR**，再构建**Agent循环依赖图（ALDG）**，通过检查反馈路径能否反复到达高成本或状态增长操作，并判断是否存在有效边界覆盖该路径，来识别IAL故障。
- **关键技术细节**：
  1. **Agent IR构建**：定义事实集合（ExecutionUnit, Controller, Invocation, StateUpdate, Bound, ExitRecord）和关系（owns, calls, updates, guards等），从源代码和框架行为中提取。框架行为建模通过解析API调用模式、配置字段（如max_turns）、工具注册、工作流边等生成。
  2. **ALDG构建**：从Agent IR中保留控制器、高成本调用、增长状态更新和作用域节点（共12种节点类型），添加10种有向边（控制流、调用、框架转换、工具分发、反馈等）。对每个控制器节点标注循环属性（成本种类、状态增长、出口种类、保护来源、边界来源）。
  3. **IAL检测算法**（Algorithm 1）：
     - 提取ALDG中可能参与重复执行的边（排除出口边和边界属性），计算强连通分量（SCC）。
     - 对每个SCC候选：收集属性集ΠL和拓扑集ΘL，过滤非目标循环（如流消费者、分页循环、测试代码）。
     - 选择续传控制器cL，分类其控制类型（确定性/模型控制/工具控制等）。
     - 检查边界覆盖：判断是否存在有效边界（如max_iterations, timeout, retry limit）约束控制器或其作用域。边界状态分为Covered（已验证、框架默认、配置依赖）和UncoveredOrWeak（缺失、弱、禁用、无效、绕过）。
     - 若满足：包含Agent反馈路径 + 可达高成本/状态增长 + 无有效边界覆盖，则报告IAL故障。可选LLM辅助修剪环节（仅作为负滤除）。

## 6. 主要结论与发现

- IAL-Scan在6,549个仓库中检测出68个真实IAL故障，精确率91.9%。故障分布于全部8个框架，LangGraph和AutoGen占比最高（66.2%）。
- 故障模式主要为：无边界的重试反馈（25.0%）、无边界的工具调用迭代（23.5%）、无转数限制的多Agent聊天（20.6%）。根本原因均为缺失强边界，常见于工具控制的重试、模型控制的终止、工作流循环或Agent重入。
- 主要影响：API成本耗尽和模型拒绝服务（各95.6%），部分导致上下文窗口膨胀（27.9%）。
- 消融实验表明：所有组件均重要，缺少任一都会降低召回率或增加误报/成本。LLM修剪可大幅减少人工审查工作量。
- 与LLM基线对比：IAL-Scan在覆盖68个真实故障（基线最多50个）、精度、token使用和时间上均显著优于coding assistant和纯LLM API方法。
- LLM修剪阶段存在一定不稳定性（三次运行TP数68/64/60），但静态阶段完全确定，LLM仅作为辅助。

## 7. 优点

- **创新性**：首次定义并系统检测LLM Agent的无限循环故障，填补了该领域的空白。
- **实用性**：支持8个主流框架，在6,549个真实项目中实现高精度（91.9%），可直接部署用于预开发检查。
- **方法论设计**：通过Agent IR和ALDG实现框架无关的抽象，有效处理框架诱导的反馈路径；边界覆盖推理考虑了控制器、作用域和路径，而非仅检查是否存在循环限制。
- **实验充分**：大规模真实数据、多框架覆盖、详尽的消融和基线对比、重复性和LLM敏感性分析，结论可靠。

## 8. 不足与局限

- **静态分析固有局限**：对可能执行路径的过度近似导致误报（6个假阳性源于难以静态解析的间接边界）；对自定义调度、外部状态停止逻辑、语义检查等复杂情况支持有限。
- **语言与框架覆盖**：目前仅支持Python和8个框架，其他语言或未建模的框架可能漏检。
- **LLM辅助阶段的不稳定性**：不同LLM模型、同一模型多次运行的修剪结果存在波动，可能影响最终报告数量。论文将LLM仅作为负滤除，但建议进一步稳定化。
- **基线对比的公平性**：LLM基线受超时和文件限制，可能未充分挖掘全部故障，但已尽力控制成本。实际应用中LLM方法成本远高于IAL-Scan。
