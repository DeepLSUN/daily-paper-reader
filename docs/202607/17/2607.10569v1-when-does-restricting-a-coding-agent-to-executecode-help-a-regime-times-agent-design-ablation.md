## 摘要
现代编码代理暴露了多种工具界面——IDE原语、bash和模型上下文协议（MCP）代码执行——而该领域已经提出了三个相互矛盾的关于哪种工具重要的主张。我们进行了缺失的交叉比较：一个完整性清洁的三臂消融实验（基线/仅bash/仅代码），在合成计算任务和SWE-bench Mini修改任务上，保持模型、框架和提示固定，并使用两个代理（Claude Code，OpenAI Codex CLI），以便比较跨越机制和代理设计两个轴线。在由此产生的四个（机制，代理）单元中，将代理限制为单个执行代码MCP工具在三个单元中比其最便宜的富含工具对手更便宜或统计上持平（在Artifact/Claude和SWE-bench/Codex上显著；在Artifact/Codex上方向性），每个单元内的通过率统计上持平。唯一的例外是SWE-bench/Claude，其中仅代码方向性成本更高（+14.4%，不显著）；条件成本分析将该差距定位到注定失败轨迹的失败成本，而不是成功运行的单次编辑税收。两个启示：最便宜的工具表面由任务机制和代理设计共同决定，而不是仅由任一轴线决定，并且标题成本信号存在于缓存调整成本中——而不是通过率，通过率在我们评估的模型大小上跨表面不变。基准框架、任务套件和分析代码可在https://github.com/hyang0129/onlycodes获取。

## Abstract
Modern coding agents expose multiple tool surfaces -- IDE primitives, bash, and Model Context Protocol (MCP) code-execution -- and the field has shipped three contradictory claims about which one matters. We run the missing crossed comparison: an integrity-clean three-arm ablation (baseline / bash_only / code_only) on synthetic computation tasks and SWE-bench Mini modification tasks, holding model, harness, and prompts fixed, with two agents (Claude Code, OpenAI Codex CLI) so the comparison spans both regime and agent-design axes. Across the four resulting (regime, agent) cells, restricting the agent to a single execute_code MCP tool is cheaper than -- or statistically tied with -- its cheapest tool-rich rival in three cells (significantly on Artifact/Claude and SWE-bench/Codex; directionally on Artifact/Codex), with pass rates statistically tied within each cell. The lone exception is SWE-bench/Claude, where code_only is directionally costlier (+14.4%, not significant); a conditional-cost analysis localizes that gap to failure-cost on doomed-run trajectories, not a per-edit tax on successful runs. Two implications: the cheapest tool surface is jointly determined by task regime and agent design rather than by either axis alone, and the headline cost signal lives in cache-adjusted cost -- not pass rate, which is invariant across surfaces at the model sizes we evaluate. The benchmark harness, task suite, and analysis code are available at https://github.com/hyang0129/onlycodes.

## 1. 核心问题与整体含义（研究动机与背景）

现代代码代理（coding agent）通常暴露多种工具接口，包括 IDE 原语（如 Read、Edit、Write）、Bash 以及基于模型上下文协议（MCP）的代码执行工具。然而，业界存在三种相互矛盾的主张：
- **专用 IDE 原语是必需的**（SWE-agent 论证 ACI 比 shell 好）。
- **仅 Bash 就足够**（mini-SWE-agent 仅用 Bash 取得高分）。
- **将工具面替换为代码执行**可以大幅减少 token 消耗（行业报告称 MCP execute_code 可减少 98–99% token）。

这些主张从未在同一框架、同一模型、按任务类型分层的情况下被系统比较过。本文填补了这一空白，通过严格消融实验，揭示工具面选择的最优性取决于**任务机制（计算 vs 修改）** 和**代理设计（Claude vs Codex）** 的共同作用，而非单一因素。

## 2. 方法论：核心思想、关键技术细节

### 2.1 核心思想
采用 **3 臂工具面消融 × 2 种任务机制 × 2 个代理** 的交叉设计，固定模型、框架和提示，隔离工具面对成本与通过率的影响。

### 2.2 三臂工具面定义
- **baseline**：代理默认工具面（Claude Code 含 Read、Grep、Glob、Edit、Write、Bash；Codex CLI 含文件原语、shell、structured apply_patch）。
- **bash_only**：限制为 Bash + 只读浏览（cat/grep/find），禁止所有文件编辑内置工具。Codex 上通过 prompt 前缀软强制。
- **code_only**：用单个 MCP 工具 `mcp__codebox__execute_code` 替代整个工具面，在持久 Python+Bash REPL 上执行代码，禁止所有原生内置工具。

### 2.3 关键技术细节
- **缓存调整成本（cache-adjusted cost）**：由于无法强制 API 缓存隔离，采用中位数地板调整法。将每个臂首次轮次的缓存读取输入 token 的中位数作为系统提示词+工具定义的热缓存基线，低于该值的实例补足到基线，高于的保留原值。最终成本 = 输入 token × 输入率 + 输出 token × 输出率（按非缓存费率计算）。
- **评估完整性**：SWE-bench 实例采用标准后置测试补丁协议（deferred test_patch application），并在应用前清理代理留下的文件，确保测试集不被污染。实验前剥离 git 历史，每臂之间使用 overlay 刷新。
- **条件成本分解**：将成本差异分解为路径成本（edit friction）、批处理不对称（batching & output control）和失败成本（failure-cost），并通过仅在“全体通过”子集上的条件分析局部化失败成本。

## 3. 实验设计

### 3.1 数据集/场景
- **Artifact Suite（计算任务）**：自建 93 个任务，覆盖 9 个类别（数据工程、数据科学、机器学习工程、算法、数据处理、枚举、迭代数值、状态推理、验证密集型）。每个任务包含 workspace、隐藏评分器、参考输出。评分基于实际产出结果，盲于工具面。
- **SWE-bench Mini（修改任务）**：从 SWE-bench 中抽取 100 个实例，分为两个子集：
  - `verified-mini`（50 个）：Django 25 + Sphinx 25。
  - `datasci-mini`（50 个）：sklearn 15, matplotlib 12, xarray 8, sympy 7, seaborn 5, astropy 3。
  覆盖经典修改任务和数据科学领域。

### 3.2 基准与对比方法
- **对比方法**：每个任务在每个代理上运行 baseline、bash_only、code_only 三个工具臂，每个臂 3 个种子。总共形成 4 个（任务 × 代理）单元：
  - Artifact × Claude
  - Artifact × Codex
  - SWE-bench × Claude
  - SWE-bench × Codex
- 主要对比：每个单元内，code_only 与其最便宜的对手（Artifact 上为 bash_only，SWE-bench 上为 baseline）比较成本和通过率。

### 3.3 代理与模型
- **Claude Code**：版本 2.1.139，模型 claude-sonnet-4-6。
- **Codex CLI**：模型 gpt-5.5。
- 两者均以无头模式运行，禁止审批弹窗，使用独立临时配置目录实现每臂隔离。

### 3.4 种子与统计单元
- 每(instance, arm) 3 个独立种子（独立容器快照）。
- 推理单元为 **task**，而非 seed：每个任务内对 3 个种子取均值，然后跨任务计算配对差异，使用 **配对 Wilcoxon 符号秩检验**（适合重尾分布）。通过率也使用基于种子的 Wilcoxon 式检验。

### 3.5 超参数
- 每次运行的 wall time 上限为 60 分钟（超时则发 SIGKILL 并记录 FAIL）。
- 成本汇率：按供应商公布的低速批量价格计算（claude-sonnet-4-6 和 gpt-5.5 的输入/缓存输入/输出 token 价格固定，文中未明确给出数字，但表示可从 JSONL 记录重现）。

### 5.2 充分性评估
- **交叉设计**：3 臂 × 2 任务 × 2 代理是最小可区分三个矛盾主张的设计，各单元间对比清晰。
- **统计检验**：使用配对 Wilcoxon 符号秩检验（非参数），并考虑重尾分布，确保结果稳健。
- **分解分析**：除了整体成本对比，还进行条件成本（通过 vs 失败子集）、输出 token 相关性、批处理行为分析，充分探索机制。

## 6. 主要结论与发现

1. **code_only 在 3/4 的 (机制, 代理) 单元中成本更低或统计持平**：
   - Artifact/Claude：成本降低 **24.6%**，p = 7.4×10⁻¹⁴（显著）。
   - Artifact/Codex：成本降低 6.7%，p = 0.25（方向性，不显著）。
   - SWE-bench/Codex：成本降低 **19.9%**，p = 2.0×10⁻⁹（显著）。
   - SWE-bench/Claude：成本增加 14.4%，p = 0.12（方向性，不显著）。

2. **通过率在所有单元无显著差异**（差异 < 3pp，所有对比 p > 0.05）。工具表面改变的是执行路径和成本，而非最终结果。

3. **成本来源的三种机制**：
   - **路径成本（Edit Friction）**：Claude × SWE-bench 中，code_only 必须用 Python 脚本表达每次编辑，导致输出 token 增加（+39.9%，p < 10⁻⁹），且与编辑量正相关（Spearman ρ = 0.49）。
   - **定价不对称（Tool-call Batching & Output Control）**：Codex × SWE-bench 中，code_only 将多个工具调用合并到一个 execute_code 脚本（每 LLM 步约 0.89 个工具调用 vs baseline 1.35），并程序化约束输出长度（p99 从 40154 字符降至 16736），从而节省输入 token（-24.8%，p < 10⁻⁸）。
   - **失败成本（Failure-cost）**：Claude × SWE-bench 的全部成本溢出集中在**未通过**实例上；在“全体通过”子集上，成本差距从 +14.4% 骤降至 +4.1%（不显著），表明失败运行消耗了不成比例的 compute。

4. **工具面选择由机制和代理共同决定，而非单一轴线**。同一 code_only 约束，在 SWE-bench 上对 Codex 节省 20%，对 Claude 却方向性更贵。

## 7. 优点

- **严格的交叉消融设计**：首次同时变化工具面、任务机制和代理设计，厘清三个矛盾主张的适用条件。
- **缓存调整成本方法**：针对 API 缓存不可隔离的现实，提出中位数地板调整法，使成本对比更公平、可重现。
- **多机制分解**：将成本差异分解为编辑摩擦、批处理优势和失败成本，而非仅报告整体数字。
- **通过率不变性的明确证据**：通过一致性矩阵（9/9 严格一致性和多数投票）证明任务结果对工具面不敏感，为成本工程而非能力声明提供基础。
- **完整开源**：框架、数据集、分析代码公开，结果可完全重现。
- **统计方法恰当**：使用配对 Wilcoxon 处理重尾分布，而非假设正态性的 t 检验。

## 8. 不足与局限

- **代理代表性不足**：仅测试 Claude Code 和 Codex CLI 两个代理，GPT-5、Gemini 2.5 或开源框架（如 OpenHands）未纳入，结论可能不泛化。
- **Codex 软强制限制**：bash_only 和 code_only 在 Codex 上通过 prompt 前缀软禁用，未使用硬开关。虽然显式工具泄漏 <1%，但模型可能通过 Bash 实现等效编辑而不触发泄漏事件，即行为合规未被完全保证。两个显著的 Codex 结果（SWE-bench 成本节省、批处理机制）仅在该软强制条件下成立。
- **样本量中等**：每单元 93–100 个任务，每个任务 3 种子，虽能检测大效应，但 Claude × SWE-bench 的方向性不显著差距可能需要更大规模（如 SWE-bench Verified 全量）来确认。
- **Artifact 套件为新基准，外部有效性有限**：所有任务为 Python/Linux，偏重分析和 ML 计算，未涵盖 Rust、系统工程或长周期场景。且该套件为自建，未经独立验证，一些结论依赖其与 SWE-bench 的对比。
- **成本为缓存调整而非缓存隔离**：虽然调整了系统提示词部分，但无法保证后续 token 不被跨任务共享（尽管文中认为实际风险小）。原始成本调整仅影响一个方向性非显著单元。
- **单次推理、内部工具面范围**：未研究多次提示、支架演化或外部 MCP 工具（如 Drive、Stripe）场景。
