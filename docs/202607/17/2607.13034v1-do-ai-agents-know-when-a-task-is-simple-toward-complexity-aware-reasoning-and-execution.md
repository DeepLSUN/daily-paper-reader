## 摘要
大型语言模型（LLM）Agent 越来越多地自动化多步骤工程和信息学工作流程，但它们很少询问任务实际需要多少工作。它们通常采用最大上下文优先策略——重读已经看过的文件和依赖项——将一行编辑变成一个小型代码库审计。我们认为缺失的能力是任务感知的执行范围估计：在投入预算之前判断任务的难度、真正需要的信息以及最短的可靠路径。我们形式化了最小充分执行和 Agent 认知冗余比（ACRR），并提出了 E3（估计、执行、扩展）：Agent 估计初始操作点，执行最小可行路径，仅在验证失败时扩展范围。在 MSE-Bench（一个确定性基准，包含 121 次编辑的能力控制模拟器）上，E3 匹配了最强基线的 100% 成功率，同时将成本降低 85%，标记数减少 91%，检查的文件减少 92%，并且进一步超越了强大的自适应检索基线 16%；这些增益在保留的指令措辞和几乎所有成本权重下仍然存在。一个配套的真实模型框架（LLM-Case）通过在一个真实的开源库上编辑实时 gpt-4o agent 来证实效果，每个候选补丁通过实际运行项目的真实 pytest 套件并根据测量 oracle 进行评分：过度读取虽然较轻但真实存在，并且在可比较的任务成功率下，E3 是最精简和最快的策略——其唯一缺点是提供商速率限制，而不是错误的编辑。我们将其视为执行冗余的控制探针，而不是任何已部署 Agent 的测量，并将任务感知执行定位为迈向工程基础的 AI（EGAI）的一步——Agent 的努力锚定在任务的工程现实中。我们发布了框架和基准。

## Abstract
Large language model (LLM) agents increasingly automate multi-step engineering and informatics workflows, yet they rarely ask how much effort a task actually requires. They often follow a maximum-context-first strategy--re-reading files and dependencies they have already seen--turning a one-line edit into a small code-base audit. We argue the missing capability is task-aware execution-scope estimation: judging a task's difficulty, the information it truly needs, and the shortest reliable path before committing budget. We formalize minimum-sufficient execution and the Agent Cognitive Redundancy Ratio (ACRR), and propose E3 (Estimate, Execute, Expand): the agent estimates an initial operating point, executes a minimum viable path, and expands scope only when verification fails. On MSE-Bench--a deterministic benchmark of 121 edits in a capability-controlled simulator--E3 matches the strongest baseline's 100% success while cutting cost by 85%, tokens by 91%, and inspected files by 92%, and further beats a strong adaptive retrieval baseline by 16%; the gains survive held-out instruction wording and essentially every cost weighting. A companion real-model harness (LLM-Case) corroborates the effect on a live gpt-4o agent editing a real open-source library, with every candidate patch graded by actually running the project's real pytest suite against a measured oracle: the over-reading is milder but real, and E3 is the leanest and fastest policy at comparable task success--its one shortfall a provider rate-limit, not a wrong edit. We frame this as a controlled probe of execution redundancy, not a measurement of any deployed agent, and position task-aware execution as a step toward engineering-grounded AI (EGAI)--agents whose effort is anchored in the engineering reality of the task. We release the framework and benchmark.

## 1. 论文的核心问题与整体含义

- **研究动机**：当前基于 LLM 的自主 Agent 在解决多步工程/软件任务时，普遍采用**最大上下文优先**策略——即使面对极简单的修改（如替换一行图标代码），也会重读整个项目目录、分析架构、确认依赖，将秒级工作变成小规模审计。这种过度收集上下文的行为是**冗余**而非谨慎，根源在于 Agent **缺乏对任务难度的快速判断能力**，无法预估“真正需要哪些信息”、“最短可靠路径是什么”。
- **整体含义**：效率不仅体现在解决难题的能力，更体现在**识别问题简单并据此行动**。作者将这种能力称为“任务感知的执行范围估计”，并视之为“工程接地气的 AI”（EGAI）的关键实例——Agent 的努力应锚定在任务的实际工程需求上，而非无约束搜索。

## 2. 论文提出的方法论

- **核心思想**：先估计任务的最小必要范围，执行最小可行路径，仅当验证失败时逐步扩展范围。三个阶段的抽象：
  - **Estimate（估计）**：根据查询和一次廉价环境探测，生成初始操作点 \( x_0 = (\hat{d}, \hat{s}, \hat{r}, \hat{c}) \)，分别对应估计难度、范围、风险和置信度。
  - **Execute（执行）**：基于估计的范围，执行最小路径：定位直接修改点、编辑、验证。验证开销也随风险缩放。
  - **Expand（扩展）**：验证失败则单调增加范围级别（如从直接命中到跨文件追踪导入），重试，最多扩展 K 次。
- **关键技术细节**：
  - 形式化**最小充分执行**：成本 \( C(\pi) = \alpha T_{lat} + \beta N_{tok} + \gamma N_{tool} + \delta N_{file} \)；最优路径是最低成本且满足成功率 ≥ 1-ε 的轨迹。
  - **Agent 认知冗余比（ACRR）**：\( \frac{C_{act} - C_{min}}{C_{min}} \)，归一化度量浪费比例。
  - 估计器实现为词法规则 + 一次结构探测（单次搜索命中次数字词冲突时降低置信度），故意不完美，依赖扩展阶段兜底。
- **算法流程**（文字描述）：
  1. 使用估计器 \( f \) 从查询和环境计算 \( x_0 \)（≤1 次廉价探测）。
  2. 设置范围级别 ℓ = 估计难度，使用缓存搜索命中列表。
  3. 执行 Execute(τ, ℓ, x₀, H) 并检查验证是否通过。
  4. 若失败且 ℓ < 3 且 k < K，则 ℓ++，k++，返回步骤 3；否则结束。

## 6. 论文的主要结论与发现

- **冗余在简单任务上最严重**：MCF 的 ACRR 在 Level 1 上高达 22.1，Level 3 仅为 5.4，量化了“越简单越浪费”的直观现象。
- **E3 匹配最强基线 100% 成功率，同时成本大幅降低**：对比 MCF 成本降低 85%、tokens 减少 91%、检查文件减少 92%；对比强自适应 AR 基线成本降低 16%、文件减少 67%（但工具调用略多 14%）。
- **E3 的 Estimate 和 Expand 组件互补**：去掉 Expand 成功率降至 85.1%（丢失所有欺骗性任务），去掉 Estimate 成本上升 20%（Level 3 上升 36%）。
- **估计器不必完美**：即使词法规则失效（保留措辞实验），扩展机制仍能保证 100% 成功，成本增加有限。
- **真实模型上的冗余较轻但真实存在**：在 gpt-4o 上，MCF 式“读所有文件”提示仅导致 1-4 个文件读取（而非仿真的 8+），但 E3 仍是整体最精简和最快的策略（平均 tokens 80503 vs AR 98611 vs MCF 98611；平均延迟 157.6s vs 165.9s vs 192.3s）；欺骗性仓库级任务中，重读型轨迹因速率限制或步骤耗尽而失败，E3 成功。

## 7. 优点

- **问题定义清晰**：首次将“任务难度感知”作为 Agent 效率的核心缺失能力，形式化冗余度量 ACRR。
- **方法论有工程直觉**：借用电力系统潮流计算的“初始操作点”概念，直观且具说服力。
- **实验设计巧妙**：通过能力不变仿真器隔离轨迹形状因素，使冗余测量精确可复现；基线选择包括强自适应方法，避免稻草人。
- **鲁棒性检验充分**：大幅改变成本权重、破坏估计器词汇线索，结论依然成立，证明核心发现来自 Estimate-Expand 架构本身。
- **真实模型验证**：提供了 LLM-Case 框架并进行了初步实证，消除了纯仿真可能的外部效度质疑（虽然规模小）。

## 8. 不足与局限

- **仿真器无法捕捉真实模型的错误与范围交互**：仿真假设“一旦看到正确文件就能正确编辑”，但真实模型可能需要更多上下文来避免错误（如误读局部文件）。这可能导致仿真中 E3 的最优点在真实环境中需要偏移。
- **估计器过于简单**：仅为词法规则 + 单次探测，泛化能力有限。作者承认学得的或 LLM 支撑的估计器可能更好。
- **MSE-Bench 的隐藏复杂度类型有限**：仅模拟了别名/重导出导致的间接依赖，未覆盖动态分派、配置耦合、运行时反射等。
- **真实模型验证规模小**：仅 5 个任务 × 3 次运行在单个模型（gpt-4o）上，且受速率限制干扰，统计效力不足。作者自称为“案例研究”。
- **真实模型下 E3 的改进幅度温和**：在简单任务上 E3 与其他基线接近，非统一优势；在欺骗性仓库任务上 E3 虽更可靠，但所有策略都昂贵且随机性大。
- **应用限制**：E3 的渐进扩展假设验证成本相对低，若验证本身昂贵（如大规模集成测试），则最优策略可能不同。
- **伦理/安全性**：未讨论故意误导估计器可能导致的安全问题（如恶意任务伪装成简单）。
