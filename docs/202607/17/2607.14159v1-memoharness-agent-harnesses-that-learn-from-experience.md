## 摘要
智能体框架（agent harness）是外部控制层，通过管理上下文、工具、编排、记忆、解码和输出处理，将基础LLM转化为可执行的智能体。虽然框架设计对智能体行为影响很大，但大多数自动改进方法优化的都是更狭窄的构件，如提示词、流水线或工作流，且部署的智能体通常对所有情况复用单一的全局框架。我们提出MemoHarness，一种自适应的框架优化框架，它从自身的执行中学习。MemoHarness将框架分解为六个可编辑的控制维度，将每个案例的诊断和提取的全局模式存储在双层经验库中，并利用检索到的经验（无需测试时标注、反馈或额外搜索）将学到的框架适应到每个测试案例。在我们针对shell智能体、代码生成和分析推理基准的评估中，MemoHarness相比于我们比较的固定框架有所改进，并显示出对未见过的套件和基础模型的选择性迁移。当大量检索到的经验可缓存时，其额外上下文也能保持成本竞争力。这些结果表明，执行经验是构建比单一静态配置更具适应性的智能体框架的实用基础，而关于统计稳健性和组件归因的更广泛结论则留待未来工作。

## Abstract
An agent harness is the external control layer that turns a base LLM into an executable agent by managing context, tools, orchestration, memory, decoding, and output handling. While harness design strongly affects agent behavior, most automatic improvement methods optimize narrower artifacts such as prompts, pipelines, or workflows, and deployed agents usually reuse a single global harness for all cases. We introduce MemoHarness, an adaptive harness optimization framework that learns from its own executions. MemoHarness decomposes the harness into six editable control dimensions, stores per-case diagnoses and distilled global patterns in a dual-layer experience bank, and adapts the learned harness to each test case using retrieved experience without test-time labels, feedback, or additional search. In our evaluation across shell-agent, code-generation, and analytical-reasoning benchmarks, MemoHarness improves over the fixed harnesses we compare against and shows selective transfer to unseen suites and base models. Its additional context can also remain cost-competitive when much of the retrieved experience is cacheable. These results provide evidence that execution experience is a practical substrate for building agent harnesses that are more adaptive than a single static configuration, while leaving broader claims about statistical robustness and component attribution to future work.

- **核心问题**：当前大型语言模型（LLM）智能体的性能不仅依赖于基础模型，还高度依赖于其“智能体框架”（agent harness）——即控制上下文构建、工具调用、推理编排、记忆管理、解码和输出处理的外部层。然而，现有的自动化优化方法大多集中于优化提示词、流水线或工作流等较窄的构件，而将框架本身视为固定的一次性配置。这导致部署后的智能体缺乏针对不同任务场景的适应性，尤其是在测试时无法根据具体案例动态调整框架。
- **背景**：实践表明，即使使用相同的基础模型和工具，框架设计本身就能使端到端任务成功率波动数十个百分点（例如 OpenAI、LangChain 的工程博客）。然而，现有工作（如 Meta-Harness）虽然将框架作为搜索对象，但结果仍是训练时固定的全局工件，缺乏测试时自适应能力。
- **研究动机**：为了解决“框架高维耦合、基准分数作为弱监督信号、测试时自适应必须避免泄漏”三大挑战，作者提出 MemoHarness——一种能从自身执行历史中学习并实现测试时案例级自适应的框架优化方法。

### 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

#### 2.1 核心思想
- **结构化可编辑框架空间**：将框架沿推理时间流分解为6个可编辑控制维度（上下文、工具、生成、编排、记忆、输出），将框架搜索转化为结构化编辑问题。
- **双层经验库**：在搜索过程中积累每案例的执行条目和全局蒸馏模式（记录“什么有效”、“什么失败”、“维度间如何交互”），使每个迭代不仅依赖得分，还能诊断原因。
- **测试时案例自适应**：在测试时，基于检索到的相似案例和全局模式，将搜索得到的全局框架调整为针对当前案例的特定框架，无需测试时反馈、梯度更新或额外搜索轮次。

#### 2.2 关键技术细节

- **六维框架空间**（W = W^{(1)} × ... × W^{(6)}）：
  - D1：上下文组装（prompt结构、添示例、压缩）
  - D2：工具交互（启用检索、设置top-k、重排证据）
  - D3：生成控制（max tokens、temperature、采样数）
  - D4：编排（单次调用→计划/执行/精炼）
  - D5：记忆管理（保留状态、总结轨迹、丢弃过时上下文）
  - D6：输出处理（提取答案、验证模式、选择回退）

- **双层经验库** B_t = (E_t, G_t)：
  - E_t：每案例执行条目，包含案例特征、配置、轨迹、奖励、成本、诊断（成功/失败、主失败维度、副维度、自然语言分析）。
  - G_t：全局模式，每N轮从失败集群中蒸馏得到跨案例规律。

- **搜索过程**（训练时优化）：
  - 从最小化框架 W_0 开始（无示例、无工具、单次调用等），确保最终配置由经验证据驱动而非默认值。
  - 每次迭代，控制器基于当前框架和经验库生成查询 q_t，检索相关证据，提出新配置 W_t，然后在所有搜索案例上执行并记录。
  - **“正确性优先”选择**：使用词序排序，先最大化平均奖励，若奖励相同则选择平均 token 成本更低的配置。

- **测试时案例自适应**：
  - 对于测试案例 x = (u, φ)，通过指令嵌入的余弦相似度从 E_T 中检索最相似的 K 个成功和失败案例，再结合全局模式和特征条件检索，形成测试时证据 S_test(x)。
  - 控制器基于此证据调整全局框架 W* 为案例特定框架 W(x)，然后执行产生最终输出。

### 7. 优点

- **结构化框架分解**：将复杂的整个框架分解为六个可编辑维度，使搜索和诊断具有可解释性，并允许针对特定维度进行修复。
- **双层经验库设计**：不仅存储每案例执行记录，还蒸馏全局模式，使经验可重用，支持更智能的搜索引导。
- **测试时案例自适应**：无需任何测试时标签、反馈或额外搜索，即可根据当前案例特点调整框架，实现了真正的零样本自适应。
- **正确性优先的选择机制**：使用词序排序（奖励优先，成本仅作为二次判决条件），避免搜索偏向低成本但低精度的配置。
- **跨模型迁移有效性**：展示了在不同模型系列上都能提升性能，说明框架优化具有普遍性，而不仅仅是过度拟合基础模型。
- **成本竞争力**：通过缓存机制，额外检索带来的输入成本得到有效控制，实际推理成本可与更简单的基线竞争。
