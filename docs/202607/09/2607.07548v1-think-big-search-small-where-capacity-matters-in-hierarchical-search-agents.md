## 摘要
基于大语言模型的搜索智能体日益采用多智能体架构，其中主智能体将复杂问题分解为子查询并分发给并行子智能体。然而，现有系统均从同一规模的单一模型实例化所有角色，未明确模型容量应如何在各角色间分配。我们将分层搜索分解为三个角色：负责任务分解的委派角色、负责检索和证据提取的执行角色，以及固定作为混淆控制的答案生成角色。然后在五个多跳问答基准上沿委派和执行轴进行受控容量扫描。实验得出三项发现：第一，角色分解始终优于单一智能体基线，在六个模型规模上将精确匹配率提升4.5至8.6个百分点。第二，容量敏感性不对称：扩展委派骨干将精确匹配率提升约11个百分点，而扩展执行子智能体仅带来约2.6个百分点的提升，表明分解能力是瓶颈。第三，通过质量过滤轨迹蒸馏训练的1.7B参数执行器在准确率上可比肩前沿子智能体，同时减少37%的子智能体令牌消耗，推进了帕累托前沿。这些结果表明构建分层搜索智能体的具体方案：将容量集中在委派角色上，同时缩减执行角色而不牺牲准确率。我们的代码已开源在https://github.com/QinnanCai0115/role-factorized-search。

## Abstract
Large language model based search agents increasingly adopt multi-agent architectures in which a main agent decomposes a complex question into sub-queries and dispatches them to parallel sub-agents. However, existing systems instantiate all roles from a single model of identical scale, leaving open how model capacity should be distributed across roles. We factorize hierarchical search into three roles: a delegation role responsible for task decomposition, an execution role responsible for retrieval and evidence extraction, and an answer generation role held fixed as a confound control. We then conduct controlled capacity sweeps along the delegation and execution axes on five multi-hop QA benchmarks. The experiments yield three findings. First, role factorization consistently outperforms a single-agent baseline, improving exact match from 4.5 to 8.6 points across six model scales. Second, capacity sensitivity is asymmetric: scaling the delegation backbone improves EM by ~11 points, whereas scaling the execution sub-agent moves EM by only ~2.6 points, identifying decomposition as the capability bottleneck. Third, a 1.7B-parameter executor trained via quality-filtered trajectory distillation matches a frontier sub-agent in accuracy while consuming 37% fewer sub-agent tokens, advancing the Pareto frontier. These results suggest a concrete recipe for building hierarchical search agents: concentrate capacity at delegation and downsize execution without sacrificing accuracy. Our code is available at https://github.com/QinnanCai0115/role-factorized-search.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **研究动机**：当前基于大语言模型（LLM）的搜索智能体越来越多地采用多智能体架构，即主智能体（backbone）将复杂问题分解为子查询，分发给并行子智能体（sub-agent）去执行检索。然而，现有系统无论主智能体还是子智能体，都使用同一规模的单一模型（如相同参数量），没有探究模型容量在不同角色之间应该如何合理分配。
- **核心问题**：在分层搜索智能体中，委派角色（任务分解）和执行角色（检索与证据提取）对模型容量的需求是否对称？如何分配容量才能实现效果与效率的最佳平衡？
- **整体含义**：本文首次系统性地研究了多智能体搜索中的容量分配问题，揭示了“委派是瓶颈、执行可压缩”的关键不对称性，为构建高效的分层搜索智能体提供了具体策略：将计算资源集中在委派角色上，执行角色可以大幅缩小而不显著损失准确率。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
- **角色分解（Role Factorization）**：将分层搜索任务显式分解为三个独立的角色策略：
  - **委派策略 π_D**（主智能体）：负责任务分解、规划和分发子查询。
  - **执行策略 π_E**（子智能体）：负责实际检索、阅读文档和提取证据，返回压缩报告。
  - **答案生成策略 π_A**（固定模块）：基于用户问题、子查询和报告合成最终答案，不接触任何智能体的推理轨迹。
- 核心设计：将 π_A 固定不变，只变化 π_D 和 π_E 的模型规模（容量），从而隔离委派和执行两个变量对性能的影响。

### 关键技术细节
- **委派流程**：主智能体最多进行 T=4 轮委派，每轮可分发子查询或终止。子查询是自包含的原子子问题，不需要访问上下文即可理解。
- **执行流程**：每个子查询由独立子智能体在隔离上下文中执行，最多 K=3 个助理回合（ReAct风格搜索）。子智能体必须基于检索文档回答，而非依赖参数化记忆。
- **答案生成**：主智能体停止后，将问题、子查询-报告对传递给固定 π_A（Qwen3-32B）生成最终答案。π_A 看不到主智能体推理和子智能体轨迹，确保仅反映委派和执行质量。
- **容量变量**：唯一自变量为 (c_D, c_E) —— 委派和执行模型容量（参数量），其他所有因素（检索器、语料库、提示词、搜索预算、解码协议、评估集）均固定。
- **干预协议**：
  - 干预1：对比单智能体 vs. 角色分解（同一模型分别扮演委派和执行）。
  - 干预2：固定执行为最小模型（Qwen3-1.7B），变化委派模型规模（从1.7B到DeepSeek-V4-Flash）。
  - 干预3：固定委派为DeepSeek-V4-Flash，变化执行模型规模（包括SFT训练的紧凑执行器）。
- **紧凑执行器训练**：针对小模型（Qwen3-1.7B）缺乏多轮检索修正能力的短板，通过质量过滤轨迹蒸馏进行监督微调（SFT）：
  - 使用教师模型（DeepSeek-V4-Pro）和固定骨干（DeepSeek-V4-Flash）在3000个训练问题上滚动，记录完整执行轨迹。
  - 质量过滤：保留“正确单次搜索”轨迹（既来自教师也来自学生自身正确结果）和“教师多次搜索且信息增益”轨迹（仅当单次搜索失败而两次搜索成功时才保留），避免冗余学习。
  - 训练时仅计算模型生成token（推理、检索动作、报告）的损失，掩码掉检索段落。

## 6. 论文的主要结论与发现

1. **角色分解本身有收益**：在所有6个模型规模上，将委派和执行分离的多智能体架构始终优于单智能体基线，EM从+4.52到+8.63点不等。前沿模型收益最小，表明角色分离在单个模型容量不足时价值最大。
2. **委派是能力瓶颈，执行收益递减**：
   - 委派容量扫描：将骨干从1.7B扩大到DeepSeek-V4-Flash，EM提升约11点（+11.3），LLM-Judge提升约16点。
   - 执行容量扫描：将子智能体从1.7B扩大到前沿模型，EM仅提升约2.6点，LLM-Judge仅约3.8点。
   - 两者差距超过4倍，说明分解质量决定性能上限，执行能力对规模不敏感。
3. **紧凑SFT执行器可推进帕累托前沿**：
   - Qwen3-1.7B-SFT（质量过滤轨迹蒸馏）在DeepSeek-V4-Flash骨干下达到41.81 EM和61.69 Judge，超过DeepSeek-V4-Flash子智能体（41.77 EM, 61.21 Judge），同时token消耗减少37%（4758 vs. 7577），参数规模小两个数量级。
   - 该结果在GLM-5.1骨干上同样成立。
4. **案例研究**：弱骨干（Qwen3-1.7B）在委派阶段错误解析亲属关系（问母亲而非父亲），导致子智能体虽然正确检索但最终答案错误。强骨干（DeepSeek-V4-Flash）正确分解，答案正确。这直观展示了委派错误会传播并导致失败。
