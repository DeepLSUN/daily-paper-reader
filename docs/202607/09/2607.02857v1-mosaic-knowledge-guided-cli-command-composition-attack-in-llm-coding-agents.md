## 摘要
LLM编码智能体通过发出普通CLI命令来完成开发任务。遵循Unix设计，这些命令通过共享操作系统状态进行协作：一个命令可能写入状态，供后续命令读取。虽然这种组合是良性的且有意为之，但它创造了一个被忽视的利用面。现有攻击和防御主要针对指令层，恶意意图表现为敌意文本。相比之下，我们观察到单独良性的命令可以在命令轨迹中形成危险的生产者-消费者状态关系，暴露了我们称之为CLI命令组合风险（CCR）。鉴于这一新的攻击面，系统地揭示和描述CCR在真实编码智能体中的影响至关重要。然而，系统理解这一风险相当具有挑战性，因为简单的命令枚举和端到端LLM生成会产生大部分无效的工作流。我们提出了MOSAIC，一个基于知识引导的框架，将从CVE、安全公告和研究人员PoC中验证的命令-状态行为提炼成可复用的摘要，将它们组合成利用路径，并实例化为逼真的开发者工作流，用于黑盒智能体评估。在五个真实CLI编码智能体和五个后端LLM的2525次试验中，MOSAIC在良性开发者任务下实现了96.59%的攻击成功率。

## Abstract
LLM coding agents increasingly complete development tasks by issuing ordinary CLI commands. Following the Unix design, these commands cooperate through shared operating-system state: one command may write state that a later command reads. While this composition is benign and intended, it creates an overlooked exploit surface. Existing attacks and defenses mainly target the instruction layer, where malicious intent appears as hostile text. In contrast, we observe that individually benign commands can form a dangerous producer-consumer state relation across the command trace, exposing what we call CLI command-composition risk (CCR).   Given this new attack surface, it is critical to systematically uncover and characterize the impact of CCR in real-world coding agents. However, systematically understanding this risk is quite challenging, because naive command enumeration and end-to-end LLM generation produce mostly invalid workflows. We present MOSAIC, a knowledge-guided framework that distills validated command-state behaviors from CVEs, advisories, and researcher PoCs into reusable summaries, composes them into exploit paths, and instantiates them as realistic developer workflows for black-box agent evaluation. Across five real-world CLI coding agents and five backend LLMs over 2,525 trials, MOSAIC achieves a 96.59% attack success rate under benign developer tasks.

## 1. 核心问题与整体含义（研究动机和背景）
- **研究动机**：LLM编码智能体（如Claude Code、Codex CLI）通过执行普通CLI命令完成开发任务。这些命令遵循Unix哲学，通过共享操作系统状态（如环境变量、文件系统、Git配置、包管理脚本等）协作。这种设计本意是良性的，但产生了被忽视的攻击面——**CLI命令组合风险（CCR）**：多个单独良性命令通过共享状态组合成一条利用链，最终产生超出用户任务范围的安全敏感能力（如远程代码执行、凭据泄露）。
- **现有工作局限**：现有攻击和防御主要聚焦于指令层（如直接提示注入、间接提示注入），假设攻击信号在单个指令、响应或操作中可见。但CCR中攻击信号隐藏在跨命令的状态依赖关系中，现有防御无法检测。
- **整体含义**：揭示了一个操作系统层的新攻击范式，LLM编码智能体面临严重安全风险，需要系统性评估和防御。

## 2. 方法论：核心思想、关键技术细节
- **核心思想**：利用现有CLI安全知识（CVE、安全公告、研究人员PoC）作为可靠知识源，从中提取已验证的命令-状态行为，组合成利用路径，并实例化为逼真的开发者工作流进行黑盒评估。
- **关键技术细节**：MOSAIC框架包含三个模块：
  - **(A) CLI安全知识库构建**：
    - **知识获取**：从NVD、GHSA、Exploit-DB、CISA KEV、安全研究博客收集30,180条原始记录，去重后得21,149个候选，通过CLI相关性过滤（白名单命令、黑名单CWE）保留5,583条，再经LLM辅助评分和人工验证得到454条确认条目。
    - **知识蒸馏**：从每条条目中提取**命令-状态摘要**，形式化表示为：`r = (τ_r, φ_r, P_r, Q_r, κ_r, E_r)`，其中τ_r是命令模板，φ_r是触发条件，P_r是产生状态集，Q_r是消费状态集，κ_r是可达到的能力，E_r是证据。摘要聚类为13个命令-状态族（如环境传播、特权帮手调用、包管理器生命周期、仓库元数据信任等）。
  - **(B) 命令组合实例生成**：
    - **候选链构建**：从知识库中提取生产者/消费者对，枚举2~4步的利用路径Π = (r1, ..., rn)，满足rℓ⇝rℓ+1（即P_rℓ ∩ Q_rℓ+1 ≠ ∅）。
    - **可行性检查**：LLM辅助检查每个候选链是否可实例化，要求每条依赖转换都已被证据支持。
    - **实例生成与验证**：对每个可行链，LLM生成攻击实例I = (q, R, C, m)，其中q是良性用户提示，R是攻击者控制的资源，C是期望命令序列，m是期望外部效果（标记）。经过模式完整性、证据基础、状态依赖一致性、局部良性、资源一致性验证后，才用于评估。
  - **(C) 运行时执行与风险验证**：
    - 在黑盒环境中，向目标智能体发出q，智能体在完成良性任务时可能处理R。记录实际命令轨迹T和外部效果E。验证条件为`TraceOK(T, I) ∧ EffectOK(E, m)`，即命令轨迹包含关键生产者-消费者步骤，且外部效果确认终端能力已触发。

## 3. 实验设计：数据集/场景、benchmark、对比方法
- **目标智能体**：五个真实LLM编码智能体——Claude Code、Codex CLI、Gemini CLI、GitHub Copilot CLI、Trae Agent。
- **后端LLM**：五个模型——GPT-5.1、Gemini-2.5 Flash、Claude Haiku-4.5、DeepSeek V4 Flash、Qwen3.7+。
- **攻击实例**：从知识库构建101个CCR利用路径（长度2:21个，长度3:35个，长度4:45个），每个路径实例化为一个攻击实例，覆盖6种终端能力类（代码执行、容器/主机逃逸、chroot/jail逃逸、特权帮手/通道访问、持久化、输出/渲染欺骗）。
- **总试验数**：101实例 × 5智能体 × 5LLM = 2,525次攻击试验。
- **对比基线**：两个指令层攻击基线（与论文威胁模型一致）：
  - **AIShellJack**：使用特定包装风格放置恶意指令。
  - **通用间接提示注入（IPI）**：使用五种包装风格（Naive、Escape-Separation、Context-Ignoring、Fake-Completion、Important-Instructions），来自现有基准。
- **评估指标**：端到端攻击成功率（ASR）、实例有效性率（IVR）、后IVR的ASR。

## 6. 主要结论与发现
- **RQ1（CCR在真实世界中的影响）**：MOSAIC端到端ASR高达96.59%，所有5个智能体均>95.8%，所有后端LLM均>95%。相比之下，AIShellJack ASR仅2.18%，通用IPI仅0.79%。表明LLM编码智能体普遍受到CCR攻击，且现有指令层防御已趋成熟但完全无法应对CCR。
- **RQ2（知识库贡献）**：结构化知识库是生成高质量攻击实例的关键。无知识库时IVR仅8.91%，端到端ASR仅0.59%；全MOSAIC达100% IVR和96.63% ASR。且在不同构建LLM下性能稳定（ASR 96%~97%），说明知识库弥补了LLM能力差距。
- **RQ3（防御分析）**：五种代表防御均无法有效阻止CCR。PromptGuard 2、Progent、CaMeL ASR下降0%；Semgrep下降9.90个百分点（残留86.73%）；AlignmentCheck下降14.06个百分点（残留82.57%）。表明现有防御缺失对跨命令状态依赖的检测能力。
- **总体发现**：LLM编码智能体在操作系统层存在严重安全盲区，CCR攻击高效、隐蔽且通用于当前生态。

## 7. 优点
- **创新性**：首次系统定义和评估CLI命令组合风险（CCR），开辟了指令层之外的新攻击面。

- **实验全面**：覆盖主流智能体和后端LLM，包含消融、对比、防御分析，结论有说服力。
- **可推广性**：知识库可帮助开发者自检产品；提出的“起源感知”防御方向具有指导意义。

- **攻击实例长度限制**：仅考虑长度≤4的命令链，更长链可能更隐蔽但未被评估，存在遗漏。

- **生态演化风险**：随智能体和LLM更新，CCR可能变化，知识库需要持续维护。
