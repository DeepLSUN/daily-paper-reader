## 摘要
自主AI系统越来越多地通过异构运行时执行操作：本地编码钩子、SDK工具、浏览器自动化、托管代理追踪、API网关和工作流引擎。因此，单个操作行为（如发布代码、更改身份状态、转移资金或导出数据）可能由多个不兼容的运行时记录表示。这使得一个基本的治理问题难以回答：实际批准了什么动作？什么证据将批准与执行绑定？独立的验证者能否在之后重新生成相同的动作身份？
本文提出了规范动作验证与证明（CAVA），这是一个运行时语义层，用于将异构代理活动转换为规范运行时动作对象。CAVA位于带有证明的代理动作（PCAA）之下：PCAA定义了部署者拥有的路线-审查-证明治理过程，而CAVA定义了该过程所治理的稳定动作对象。本文形式化了规范动作身份、语义模式检测、批准绑定、收据完整性、运行时可移植投影以及可选的证明基础。我们通过一个包含96个种子、384个变体的基准测试研究了参考实现，涵盖了语义等价、语义分离、封装绕过、误报控制、批准绑定、收据可复现性、证明篡改检测、运行时可移植性、语义模式检测、策略降级以及Azure部署演练。本文的贡献在于将动作级规范化和策略可寻址语义模式作为部署者侧AI治理的必要基础进行了系统化表述。

## Abstract
Agentic AI systems increasingly act through heterogeneous runtimes: local coding hooks, SDK tools, browser automation, managed-agent traces, API gateways, and workflow engines. A single operational act such as publishing code, changing identity state, moving money, or exporting data may therefore be represented by many incompatible runtime records. This makes a basic governance question difficult to answer: what action was actually approved, what evidence binds the approval to execution, and can an independent verifier reproduce the same action identity later?   This paper presents Canonical Action Verification and Attestation (CAVA), a runtime-semantics layer for converting heterogeneous agent activity into canonical runtime action objects. CAVA is positioned below Proof-Carrying Agent Actions (PCAA): PCAA defines the deployer-owned route-review-prove governance process, while CAVA defines the stable action object that process governs. The paper formalizes canonical action identity, semantic pattern detection, approval binding, receipt integrity, runtime-portable projection, and optional attestation substrates. We study a reference implementation through a 96-seed, 384-variant benchmark covering semantic equivalence, semantic separation, wrapper bypass, false-positive control, approval binding, receipt reproducibility, attestation tamper detection, runtime portability, semantic pattern detection, policy degradation, and Azure deployment drills. The contribution is a systems formulation of action-level canonicalization and policy-addressable semantic patterns as a necessary substrate for deployer-side AI governance.

# CAVA: Canonical Action Verification and Attestation 论文总结

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：Agentic AI 系统在异构运行时中执行操作（shell命令、SDK方法、浏览器自动化、API网关、工作流引擎等），同一业务动作可能产生多种不兼容的运行时记录，导致基本的治理问题难以回答：实际批准了什么动作？批准与执行之间的证据如何绑定？独立验证者能否在之后重新生成相同的动作身份？
- **研究动机**：企业治理不仅需要模型侧策略、聊天日志或事后追踪，还需要一个稳定的对象来标识正在被决策的动作。当前这个对象不稳定：同一高影响动作在shell、SDK、工具调用、浏览器事件等不同运行时中的表示差异巨大，若批准绑定于原始文本，等价重写即可绕过批准；若审计绑定于原生记录，不同运行时会使同一动作无法比较；若策略绑定于首令牌，封装（env、sudo、bash -c等）可以改变表面形式而不改变后果。
- **整体含义**：CAVA提出了一个运行时语义层，将异构代理活动转换为**规范运行时动作对象**，作为治理的前提和基础。它位于PCAA（带证明的代理动作）之下，PCAA定义部署者拥有的路由-审查-证明治理流程，而CAVA定义该流程所治理的稳定动作对象。这一工作将动作级规范化和策略可寻址语义模式系统化，视为部署者侧AI治理的必要基础。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：通过一个规范器（canonicalizer）将原始运行时事件转换为规范运行时动作C(a)，再计算确定性指纹F(a)=H(canon(C(a)))，将策略决策、批准收据和审计证据绑定到指纹而非原始文本。
- **规范动作对象**：包含字段（schema_version, runtime, adapter, executable, operation, category, systems_touched, reversible, target/subject, bounded adapter metadata），详见Table 1。
- **6阶段协议**：
  1. **捕获(Capture)**：收集原始运行时事件和执行上下文。
  2. **规范化(Normalize)**：映射为规范运行时动作对象。
  3. **解释(Interpret)**：从规范动作、边界、来源、数据信号中检测策略可寻址的语义模式。
  4. **指纹(Fingerprint)**：对规范动作语义计算确定性哈希。
  5. **绑定(Bind)**：将策略结果、批准/拒绝等附加到指纹上。
  6. **关闭(Close)**：附加结果、证据、异常、副作用摘要。
  7. **证明(Attest)**：可选签名、凭据、账本锚定或外部验证收据材料。
- **语义模式层(Semantic Pattern Layer)**：在规范化之后策略路由之前，将规范动作、边界上下文(B(a))、数据上下文(D(a))映射为可复用的策略可寻址模式（如：公共持久性出口、安全控制削弱、凭证暴露、代理权限错配、工作流接收器风险、提示或规则篡改等），避免变成客户特定规则的堆积。
- **关系**：CAVA与PCAA、有界动作防火墙(BAF)、运行时暴露图(AREG)协同工作，CAVA负责动作语义，PCAA负责最终权威和证明。
- **证明基础**：本地SHA-256收据、签名、可验证凭据、in-toto/Sigstore式证明、许可/公共账本锚定、智能账户策略。Web3和账本作为可选附件而非CAVA定义。

## 3. 实验设计：数据集/场景、benchmark、对比方法

- **数据集/场景**：96个代表性种子场景，扩展为384个运行时变体，涵盖4种运行时：shell钩子、MCP风格工具、浏览器自动化、托管代理追踪。

- 原始文本治理缺乏稳定的语义等价概念；首令牌规则无法应对封装且容易误报；规范收据允许原始字符串无法提供的批准和证明检查。

- CAVA不是最终的安全保证，而是提供动作级规范化和策略可寻址语义模式的必要基础。

## 7. 优点

- **系统性**：将“动作身份”问题从产品特性提升为可检验的科学组件，包含清晰的形式化定义、协议、威胁模型和属性。
- **开放核心设计**：公开模式、哈希、收据和适配器接口，允许第三方审查和适配，同时保留商业化理由，降低锁定风险。
- **语义模式层**：将策略从客户特定规则提升为可复用的风险模式（如公共出口、安全控制削弱等），促进企业政策可移植性。
- **实验设计严谨**：分离得分检查与系统卡材料，包含消融、红队案例和部署演练，明确承认剩余风险和偏差。
- **威胁模型全面**：考虑语义绕过、封装绕过、批准漂移、证据洗白、解析器捕获等，且明确不假设所有运行时都能提供内联拦截。
- **透明性**：包含系统卡、误用分析、外部验证路线图、评估者检查清单，鼓励第三方挑战。

## 8. 不足与局限

- **语料库局限性**：基准是代表性公开语料库，非完整企业动作分布，未来需第三方轨迹扩展。
- **解析器覆盖**：规范化强依赖适配器覆盖，私有企业系统可能需要自定义解析器包，解析失败可能导致漏报或误报。
- **商业边界导致不可完全复现**：生产解析器包、客户连接器规则等被保留，公开骨架无法完全复现OSuite的管理行为。这限制了第三方完全复现。
- **证明范围**：收据哈希和签名证明完整性和绑定，而非业务智慧。一个正确证明的动作仍可能是糟糕的决策。
- **运行时深度差异**：某些运行时支持内联拦截，其他仅提供观察模式。CAVA必须披露此差异而非声称统一执行。
- **基准可能过拟合**：固定公开语料库可能被过度拟合，需要持续扩展。
- **外部验证未完成**：第三方轨迹、独立解析器挑战、买家证明审查等尚未实现，仅处于规划阶段。
