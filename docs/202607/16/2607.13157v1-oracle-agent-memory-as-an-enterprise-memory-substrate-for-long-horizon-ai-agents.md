## 摘要
代理记忆是长周期代理面临的一个系统性问题。实际部署需要在扩展对话中保留任务状态，跨会话恢复特定于用户的事实和偏好，以及累积来自先前结果的过程性知识。这些需求超出了文档检索的范畴：记忆层必须确定哪些交互成为持久状态，如何划定该状态的范围，如何在延迟约束下检索它，以及如何随时间修正或移除它。本报告研究了基于Oracle数据库构建的Oracle Agent Memory作为数据库原生记忆基板。讨论围绕三个主题展开：记忆作为生命周期，涵盖摄取、提取、整合、检索、总结以及修正或移除；分层架构，将活动记忆核心与被动记忆存储接口分离，并跨用户、代理和线程进行显式范围控制；评估方法，其中下游任务准确性辅以以记忆为中心的度量，如证据检索、召回率、延迟和估计的令牌使用量。本报告总结了LongMemEval结果，达到了93.8%的准确率，将Oracle Agent Memory与平面历史基线进行比较，使用的令牌数减少了约10.7倍，并在可用时发布了或报告了外部基线，最后以面向实现的附录材料结束，涵盖设置、线程生命周期和搜索语义。

## Abstract
Agent memory is a systems problem for long-horizon agents. Practical deployments require retention of task state across extended conversations, recovery of user-specific facts and preferences across sessions, and accumulation of procedural knowledge from prior outcomes. These requirements extend beyond document retrieval: a memory layer must determine which interactions become durable state, how that state is scoped, how it is retrieved under latency constraints, and how it is revised or removed over time. This report studies Oracle Agent Memory as a database-native memory substrate built on Oracle Database. Three themes organize the discussion: memory as a lifecycle spanning ingestion, extraction, consolidation, retrieval, summarization, and revision or removal; a layered architecture that separates an active memory core from a passive memory-store interface with explicit scope control across users, agents, and threads; and evaluation methodology in which downstream task accuracy is complemented by memory-centric measures such as evidence retrieval, recall, latency, and estimated token use. The report summarizes LongMemEval results, reaching 93.8% accuracy, compares Oracle Agent Memory against flat-history baselines, using about 10.7x fewer tokens, and published or reported external baselines where available, and closes with implementation-oriented appendix material covering setup, thread lifecycle, and search semantics.

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：长周期（long-horizon）AI 代理在真实企业部署中面临持久记忆管理的系统性问题。代理需要跨扩展对话保留任务状态、跨会话恢复用户特定事实与偏好、从先前结果中积累程序性知识。简单的文档检索或扁平对话历史无法满足这些需求——记忆层必须决定哪些交互成为持久状态、如何划定作用域、如何在延迟约束下检索、如何随时间修订或移除。
- **整体含义**：将代理记忆视为一个受管理的生命周期，而非单一的检索接口。企业环境要求记忆系统支持治理、安全性、数据驻留、可靠性和与代理循环应用逻辑的集成。论文提出将 Oracle 数据库作为数据库原生记忆基板，以统一处理持久化、索引、作用域感知检索以及与结构化/半结构化企业数据的兼容性。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：基于 Oracle 数据库构建分层记忆架构，分离“主动记忆核心”与“被动记忆存储接口”。主动层负责线程同步、提取、总结和搜索编排；存储层提供持久化、作用域检索和兼容第三方记忆提供者的合同。
- **关键技术细节**：
  - **记忆分类**：短期/工作记忆（线程总结、上下文卡片）、长期事实记忆（用户偏好、规则）、长期经验/程序性记忆（从先前执行中学习到的策略）。
  - **生命周期**：摄取→提取→整合→检索→总结→修订/移除。系统必须处理选择（并非所有消息都应成为持久记忆）、表示（文本、配置、消息历史等）、作用域（用户、代理、线程级）、演化（陈旧、冲突）。
  - **存储模型**：使用关系表、JSON、向量索引和事务。核心实体：`THREAD`、`MESSAGE`、`MEMORY`、`ACTOR_PROFILE`、`RECORD_CHUNKS`。作用域标识符（user_id, agent_id, thread_id）作为一等存储和检索维度。
  - **工作流**：初始化客户端→创建/重开线程→添加消息（触发提取和总结）→检索上下文（通过搜索、上下文卡片、总结）。
  - **集成模式**：中间件模式（自动同步）、工具模式（代理显式调用）、混合模式。
- **安全与治理**：作用域是检索控制，但生产需要数据库级授权（Oracle Deep Data Security），而非仅仅应用层过滤。

## 6. 主要结论与发现

- **准确率**：在 LongMemEval 上达到 93.8%（469/500）。多会话推理最低（88.0%）。
- **令牌效率**：相比扁平历史基线，在 80 轮对话中每请求输入令牌减少约 10.7 倍（1300 vs 13900）。
- **答案质量**：成对比较中，Agent Memory 赢得 48/80 轮，扁平历史赢得 13/80，3.7 倍胜率。
- **BEAM 性能**：在 1M 数据集上达到 68.0%（事件存在评分），超过 Mem0 报告基线（64.1%）。顺序敏感评分较低（63.0%）。10M 数据集降至 51.0%，检索诊断显示事件排序和总结回答仍是瓶颈。
- **阈值可调性**：总结触发器控制令牌使用与上下文保真度的权衡。
- **提示缓存**：需要平衡令牌节省与缓存命中率，建议将静态内容放在前缀，动态记忆放在后面。
