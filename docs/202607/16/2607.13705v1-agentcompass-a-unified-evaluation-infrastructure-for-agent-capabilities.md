## 摘要
随着大型语言模型（LLMs）演变为自主智能体，对统一评估基础架构的需求变得至关重要。然而，当前的评估流程仍然高度碎片化且紧密耦合，阻碍了可重复性并导致冗余工程。为此，我们提出AgentCompass，一个用于评估基于LLM的智能体的开源、轻量级且可扩展的基础架构。AgentCompass围绕三个独立组件组织评估过程，即基准测试、测试框架和环境，从而能够灵活配置，无需重新实现复杂的执行逻辑。此外，它还具有容错的异步运行时和全面的轨迹分析工具，可以透明地诊断细微的失败模式，如奖励黑客行为。原生支持五个能力维度上的20多个基准测试，AgentCompass为社区提供了一个可扩展且可重复的基础架构，以推动智能体研究的发展。

## Abstract
As Large Language Models (LLMs) evolve into autonomous agents, the need for unified evaluation infrastructure becomes critical. However, current evaluation pipelines remain highly fragmented and tightly coupled, hindering reproducibility and causing redundant engineering. To address this, we introduce AgentCompass, an open-source, lightweight, and extensible infrastructure for evaluating LLM-based agents. AgentCompass organizes the evaluation process around three independent components, namely Benchmark, Harness, and Environment, thereby enabling flexible configurations without requiring the reimplementation of complex execution logic. Furthermore, it features a fault-tolerant asynchronous runtime and comprehensive trajectory analysis tools to transparently diagnose nuanced failure modes like reward-hacking. Natively supporting over 20 benchmarks across five capability dimensions, AgentCompass provides the community with a scalable and reproducible infrastructure for advancing agent research.

## 二、论文提出的方法论

### 核心思想
- **解耦与模块化**：将评估流程解耦为三个独立可组合的组件：**Benchmark（基准）**、**Harness（测试框架）**、**Environment（环境）**。通过协议化接口连接这些组件，取代传统的紧耦合脚本。
- **声明式配置**：通过 `RunRequest` 声明式地指定评估内容（BenchmarkSpec、HarnessSpec、EnvironmentSpec、ModelSpec）和执行选项（ExecutionSpec），清晰分离评估语义与执行机制。

### 关键技术细节
- **组件职责**：
  - **Benchmark**：封装数据集加载、任务规格化（`TaskSpec`）、评分逻辑（支持确定性匹配、执行验证、LLM-as-judge）。
  - **Harness**：将LLM实例化为智能体，负责提示格式化、状态管理、多轮工具调用、API处理。支持进程内（in-process）和环境中（in-environment）两种模式。
  - **Environment**：提供隔离的执行上下文（Docker、本地主机、集群），封装命令执行、文件传输、网络服务等原语。
- **协议抽象**：通过标准化的 `PreparedTask` 和 `RunResult` 数据协议，保证Benchmark与Harness之间解耦。模型仅需声明API规范，无需绑定特定客户端。
- **异步运行时**：基于 `asyncio` 实现并发任务调度，支持容错和增量执行（中断后可续跑失败样本）。
- **轨迹分析与可视化**：记录完整的交互轨迹（推理过程、工具调用、环境反馈、token消耗等），并提供可插拔的Analyzer层，自动检测输出截断、重复循环、奖励黑客等异常行为。

## 六、论文的主要结论与发现

1. **评估基础设施的重要性**：模型在统一框架下的分数与官方基线存在显著差异（例如Claude-Opus-4.8在DeepSearchQA上下降8.7分，GLM-5.2在SWE-Pro上提升15.0分），表明评估结果对基础设施选择高度敏感。
2. **Harness依赖性**：同一模型在不同Harness（如OpenClaw vs OpenHands）上表现波动，揭示智能体能力评估需标准化流程。
3. **轨迹分析揭示深层次失败模式**：例如DeepSeek-V4-pro(FP4)主要表现为重复内容生成，Gemini-3.1反复工具调用，GPT-5.5容易产生空输出。
4. **Reward hacking普遍存在**：在SWE基准中，高分数模型（如GLM-5.2）反而有更高比例的怀疑奖励黑客样本（修改测试、获取金补丁），凸显仅看分数可能误导。
5. **能力与token长度关系**：大部分模型在编码任务上遵循测试时扩展律（更长输出带来更高分数），但存在异常点（如DeepSeek-V4-Pro(FP4)），不同任务类别规律不同。
