## 摘要
AI编码代理是黑盒：我们无法检查它们如何生成代码，但可以检查它们改变了什么。这种区别对于基于搜索的软件工程（SBSE）很重要，其中诸如遗传改进（在我们研究的性能优化应用中）之类的技术依赖于反映代码实际如何转换的突变算子。在AIDev-pop中33,596个代理PR中，只有不到1%针对性能，使得每个案例成为原本不透明的代理行为的罕见窗口。我们使用双LLM交集流水线，根据Even-Mendoza等人（2025）的18类语法突变分类法，对来自216个此类PR的1,254个与性能相关的差异块进行分类。三大类别占主导地位：名称修改（37.0%）、对象创建（26.4%）和类型更改（22.7%），这一分布与之前的GI语料库显著不同，其中无变化占84%。每个代理的部署系统都采用独特的突变词汇表，每个性能策略激活大部分不相交的类别子集。因此，代理身份和目标任务策略是信息丰富的先验，可以缩小有效的SBSE算子空间。复现包：https://github.com/5uper6rain/ssbse-challenge-2026

## Abstract
AI coding agents are black boxes: we cannot inspect how they generate code, but we can inspect what they change. This distinction matters for search-based software engineering (SBSE), where techniques such as genetic improvement (in the performance-optimisation application we study) depend on mutation operators that reflect how code is actually transformed. Fewer than 1% of the 33,596 agent PRs in AIDev-pop target performance, making each case a rare window into otherwise opaque agent behaviour. We classify 1,254 performance-relevant diff hunks from 216 of these PRs, spanning five agent systems, against the 18-category syntactic mutation taxonomy of Even-Mendoza et al. (2025) using a dual-LLM intersection pipeline. Three categories dominate: name modification (37.0%), object creation (26.4%), and type change (22.7%), a profile markedly different from prior GI corpora where no change accounted for 84%. Each agent's deployed system commits to a distinctive mutation vocabulary, and each performance strategy activates a largely disjoint category subset. Agent identity and target strategy are therefore informative priors that narrow the effective SBSE operator space. Replication package: https://github.com/5uper6rain/ssbse-challenge-2026

## 6. 主要结论与发现

- **RQ1（与GI语料库对比）**：AI代理性能PR的突变模式与GI语料库显著不同。
  - 三大主导类别：名称修改（37.0%）、对象创建（26.4%）、类型更改（22.7%）；而GI语料库中“无变化”占84%。
  - 名称修改在78%的案例中体现为库导入替换（如`chalk`→`picocolors`），本质上是一种更高层次的API替换，建议未来分类法将其细分。
  - 高风险类别（同步、内联、异常处理）极少被使用（<2%）。
- **RQ2（按代理和策略变化）**：
  - **按性能策略**：类型更改有89%集中于数据结构优化策略；名称修改有75%集中于构建与基础设施策略；对象创建跨多个策略；控制流与对象创建在36.6%的PR中共现。
  - **按代理系统**：四个代理显示出截然不同的突变词汇：
    - Devin（TypeScript主导）：名称修改占54%（库替换为主）。
    - GitHub Copilot（Rust主导）：类型更改占38%（数据结构升级）。
    - OpenAI Codex（Go主导）：控制流（24%）和对象创建（18%）占比最高，分布最广。
    - Cursor（样本最小）：注释修改比例较高（22%），但噪声较大。
  - **关键启示**：代理身份和目标策略可作为有效先验，将18类操作空间缩小至约5类，指导SBSE算子选择。

## 8. 不足与局限

- **构造效度**：
  - 突变分类法源于Java LLM补丁，可能无法完全覆盖多语言（TypeScript/Rust/Go）中的特有突变类型。
  - “对象创建”类别扩展了import变更，导致其26.4%的占比无法与原始Java语料直接比较。
  - 性能策略标签来自外部论文，存在误标风险。
- **内部效度**：
  - LLM分类准确率仅67.5%（精确匹配），部分一致案例采用交集可能导致有效信息丢失或低估某些类别。
  - 未进行人类专家验证或交叉标注，LLM可能系统性地偏好检测某些突变（如显式导入/重命名）而遗漏其他。
- **外部效度**：
  - 代理与编程语言高度相关（Devin→TypeScript，Copilot→Rust，Codex→Go），无法区分代理特性与语言特性的影响；结论不能直接推广到其他代理或语言组合。
  - 性能PR仅占不到1%，样本量有限（216个PR），且比例是hunk加权的（多hunk的PR贡献更大权重），可能受少数长PR主导。
  - 代理和LLM版本均为快照（2026年7月），随着系统更新结果可能变化。
  - 仅研究已标注“performance”的PR，未覆盖代理可能隐含的性能优化（如混合在bug fix中）。
- **统计充分性**：
  - 缺乏推论统计（如p值、置信区间）和效应量；仅提供描述性计数和百分比。
  - 未对代理间差异进行统计检验（如Fisher精确检验），部分差异可能由样本量不平衡（Devin 670 hunk vs Cursor 184 hunk）导致。
- **应用限制**：结论仅为描述性而非规范性，不能直接作为代理行为的最优模式；未来工作需要验证这些先验能否实际提升SBSE工具效果。
