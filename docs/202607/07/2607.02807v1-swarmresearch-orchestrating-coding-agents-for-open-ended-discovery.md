## 摘要
诸如autoresearch之类的长期运行的编码智能体可以持续发现开放式问题的优化方法。然而，它们往往收敛到单一的高级方法，然后进行低级编辑，同时错过了其他更优的问题解决途径。我们假设两种框架级设计选择导致了这种行为：在单个长期运行的智能体中积累上下文，以及仅暴露单个程序状态进行编辑。我们引入SwarmResearch，一种编排器-子智能体框架，其中Shepherd智能体使用全局上下文来引导一组Search智能体，每个智能体在其各自的git分支中使用局部上下文操作。在开放式优化任务中，与最先进的LLM引导进化和多智能体技术相比，SwarmResearch在13/15个任务上发现了更好或可比的解决方案，这是由更高级别的探索驱动的。与串行和并行智能体的固定缩放相比，SwarmResearch的编排器引导的缩放通过在不同搜索深度调整并行性，发现了性能更优的解决方案。

## Abstract
Long-running coding agents such as autoresearch can persistently discover optimizations for open-ended problems. However, they tend to converge onto a single high-level approach, then proceed with low-level edits while missing other superior approaches to the problem. We hypothesize two harness-level design choices contribute to this behavior: accumulating context in a single long-running agent and only exposing a single program state to edit. We introduce SwarmResearch, an orchestrator-subagent harness in which a Shepherd Agent uses global context to steer a population of Search Agents, each operating with local context in their respective git branch. On open-ended optimization tasks, SwarmResearch discovers better or comparable solutions to state-of-the-art LLM-guided evolution and multi-agent techniques on 13/15 tasks, driven by higher-level exploration. Compared with fixed scaling of serial and parallel agents, SwarmResearch's orchestrator-guided scaling discovers better-performing solutions by adapting parallelism at different search depths.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **问题**：长期运行的编码智能体（如 autoresearch、Claude Code）在开放式优化任务中，会快速收敛到单一高级方法，之后只进行微小的低级编辑（如参数调优），从而错过其他可能更优的解题方向。这种“局部贪婪搜索”行为严重限制了发现能力。
- **根源**：作者假设两个框架级设计缺陷导致该行为：
  - **上下文积累**：单个智能体长期运行时，其对话历史中充满了对当前方法的增量改进，使得它难以进行大规模方向转换。
  - **单一程序状态**：所有改进都提交到同一个可编辑程序，回归被回退，没有保存多个竞争方向，导致搜索路径变成贪婪的局部优化。
- **含义**：开放式发现（如算法设计、系统优化）需要持续探索多样的高维方案空间。本文提出一种新的编排框架，通过更好的上下文管理和版本控制来提升智能体的搜索广度与探索深度。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **整体框架**：SwarmResearch 采用“编排器-子智能体”架构（Orchestrator-Subagent），由一个 **Shepherd Agent（牧羊人智能体）** 和多个 **Search Agents（搜索智能体）** 组成。
- **核心机制**：
  - **Shepherd Agent**：持有全局上下文（所有搜索智能体的摘要、评估分数），负责战略性地选择和 spawn 新的搜索智能体。它通过三个 steering 机制引导搜索：
    1. **父节点选择（Parent Selection）**：为每个新搜索智能体设定一个 git 分支，指定其工作起点（可以是 baseline 或已完成智能体的 commit）。据此控制搜索邻域。
    2. **搜索智能体类型**：
       - **Explorer（探索者）**：拥有新鲜的上下文窗口，只读取工作区内容，不受历史束缚，适合尝试全新方法。
       - **Optimizer（优化者）**：继承父智能体的完整对话历史，适合在已有方法上进行深度连续改进。
    3. **提示（Prompts）**：Shepherd 可以向搜索智能体提供最小、非指令性的上下文，例如：已探索过的想法列表（避免重复）、当前最佳方法的优缺点总结（帮助突破平台期）。**关键约束**：Shepherd 不能指定具体要尝试的想法，只能提供背景信息。
  - **上下文分离**：
    - 搜索智能体仅拥有局部上下文（工作区 + 可选的父历史），不直接访问其他智能体的结果，避免早期收敛到共享最优。
    - Shepherd 拥有全局上下文（各搜索智能体的摘要），用于调整种群方向。
  - **Git 分支管理**：每个搜索智能体被分配一个新 git 分支（从 master 或某个父 commit 创建），所有修改版本化。这样可以保留竞争方案，防止因回退而丢弃有潜力的方法。分支也支持多种搜索模式（扇形展开、串行链、合并等）。
- **算法流程**（文字描述）：
  1. Shepherd 根据当前全局状态，决定 spawn 一个 Explorer 或 Optimizer，选定父分支，编写最小提示，创建新分支。
  2. 搜索智能体在其分支内读取 prompt.md 和 findings.md，提出一个想法，实现并评估（使用标准评估器），循环迭代若干次（Explorer 通常只做一次，Optimizer 做 5 次）。
  3. 每次迭代后，智能体提交变更，记录评估分数到 findings.md 并提交。
  4. Shepherd 接收完成智能体的摘要（包括 commit 消息和分数），更新全局状态，重复以上过程直到预算耗尽。
  5. Shepherd 自动调整并行/串行策略：初期大量 Explorer 扇形展开以积累多样性；后期将资源集中在最有前途的线上进行串行优化。

## 6. 论文的主要结论与发现

1. **性能优越**：SwarmResearch 在 13/15 个任务上达到或超过 EvoX 和 CORAL 的表现，在 11 个任务上严格超过 CORAL（2 个持平），在 13 个任务上超过 EvoX（1 个持平）。仅在后两个系统/启发式任务上不如 CORAL。
2. **更高级的探索**：SwarmResearch 每次 commit 的代码变更中位数为 CORAL 的 3.2 倍、EvoX 的 1.7 倍，表明它进行了更多的高层次方法变换而非微调。
3. **自适应伸缩优越**：编排器引导的缩放（动态调整不同搜索深度上的并行度）在 5 个测试任务中的 4 个上优于固定最优的并行/串行配置。例如在 Signal Processing 中，SwarmResearch 实现了更深的串行链以打磨最优解。
4. **案例研究**：在推测解码任务上，SwarmResearch 达到 **4.58×** 加速（相对于原始解码），优于 autoresearch（1.80×）和 CORAL（2.26×），准确率从 65.8% 降至 60.6%（主要是由于 token 限制）。其最佳方案包括目标模型批处理、自适应 MoE 专家数、持久化线程池等系统级优化。
5. **新颖尝试的局限性**：SwarmResearch 提出的一些完全新颖的“comonotone sampling”等想法效果很差，且智能体的理由错误，说明当前 LLM 的创造力仍有限，输出需要谨慎审查。

## 7. 优点

- **简单而有效的设计**：通过两个核心改变（上下文分离 + git 分支版本控制）显著提升了搜索多样性，未引入复杂组件。
- **兼容现有工具**：实现为三个 skill（Shepherd, Explorer, Optimizer），可直接集成到 Claude Code 等流行编码代理工具中。
- **无需超参数**：Shepherd 自适应调整并行度/串行度，避免手动调参（如固定缩放需要搜索最优配置）。
- **高度可扩展性**：框架天然支持大规模并发探索，并能在资源预算内高效分配。
- **开源可用**：代码和细节在 GitHub 上公开，便于复现和扩展。
- **新颖分析**：引入代码变更粒度作为衡量探索“高级性”的代理指标，并系统对比编排器引导缩放与固定缩放，提供了关于测试时计算分配的新见解。
- **案例研究的深度**：通过推测解码作为案例，展示了 SwarmResearch 的实际搜索轨迹和优化过程，包括对失败尝试的诚实报告，体现了研究严谨性。

## 8. 不足与局限

- **单次运行问题**：主要实验结果基于每个任务一次运行，统计稳定性不足。作者虽就此给出解释（成本高、使用指南），但削弱了结论的可靠性。不同运行时方差可能显著。
- **模型依赖**：SwarmResearch 和 CORAL 均依赖于 Claude Code 等前沿模型；EvoX 在 GPT 系列上表现更好，本文使用 Opus 可能对 EvoX 不利。此外，Shepherd 的行为高度依赖提示工程，未能表现出理想的复杂策略（如根据瓶颈合成具体问题）。
- **基本创造力未增强**：方法仍偏向于已有方法的变体，未能产生真正的根本性新见解（案例中所有成功优化均为已知技术的变体）。
- **成本高昂**：一次完整任务预算 $50 (Claude Code)，对于大规模应用仍昂贵。作者提到 $1700 运行 15 个任务的所有方法。
- **缺乏内部消融**：未单独评估 Explorer 和 Optimizer 各自贡献，也未分析提示引入信息的具体影响。固定缩放实验使用了不同的模型（Minimax）与主要实验不一致，可能引入混淆变量。
- **潜在误导风险**：SwarmResearch 生成的高吞吐量想法中可能包含有吸引力但错误的方案（如 comonotone sampling），需要人工审查，增加使用成本。
- **应用局限**：基准任务均为优化问题（算法/系统），未涉及纯科学发现（如定理证明、分子设计），通用性待验证。
