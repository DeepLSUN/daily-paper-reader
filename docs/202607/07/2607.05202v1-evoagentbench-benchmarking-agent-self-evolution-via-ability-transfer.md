## 摘要
在长周期LLM系统中，智能体的自我演进主要是程序性的：有用的经验不仅仅是存储的信息，而是用于搜索、调试和验证的可重用程序。然而，当前的评估并未孤立这种形式的迁移。智能体基准测试测试单次任务解决；记忆基准测试针对信息保留而非程序复用。我们引入了EvoAgentBench，这是一个通过能力引导迁移实现智能体自我演进的基准测试，涵盖四个智能体领域：网络研究、算法推理、软件工程和知识工作。EvoAgentBench从智能体执行中提取基于轨迹的能力，将其规范化操作单元，并构建特定领域的能力图，连接具有程序重叠的任务。通过设计，每个测试任务都有经过验证的训练侧能力支持。在528/267的训练/测试划分、两个支架和三个骨干模型上，精心策划的能力内容在不同模型系列间可靠迁移，但目前没有自动方法能在所有设置中保持正向增益。EvoAgentBench将自我演进评估从总体准确性比较转变为对经验编码、路由和吸收的细粒度诊断。该基准测试公开发布于https://huggingface.co/datasets/EverMind-AI/EvoAgentBench。

## Abstract
Agent self-evolution in long-horizon LLM systems is largely procedural: useful experience is not merely stored information, but reusable procedures for searching, debugging, and verification. Yet current evaluations do not isolate this form of transfer. Agent benchmarks test single-episode task solving; memory benchmarks target information retention rather than procedural reuse. We introduce EvoAgentBench, a benchmark for agent self-evolution via Ability-guided transfer across four agentic domains: web research, algorithmic reasoning, software engineering, and knowledge work. EvoAgentBench extracts trace-grounded Abilities from agent executions, canonicalizes them into operational units, and builds domain-specific Ability Graphs linking tasks that share procedural overlap. By design, every test task is backed by verified training-side Ability support. Across a 528/267 train/test split, two scaffolds, and three backbones, curated Ability content transfers reliably across model families, but no current automatic method sustains positive gain in all settings. EvoAgentBench shifts self-evolution evaluation from aggregate accuracy comparison to fine-grained diagnosis of experience encoding, routing, and uptake. The benchmark is publicly available at https://huggingface.co/datasets/EverMind-AI/EvoAgentBench.

## 论文总结：EvoAgentBench：通过能力迁移实现智能体自我演进的基准测试

### 1. 论文的核心问题与整体含义

*   **研究动机**：当前的大型语言模型（LLM）智能体（Agent）在解决长周期任务方面取得了显著进展，但评价体系存在关键盲点。现有的基准测试主要聚焦于：
    *   **单轮任务求解**：测试智能体是否能在一次交互中解决问题。
    *   **信息记忆**：测试模型对过往信息的回忆能力。
    *   **在线/持续学习**：评估技能库如何随任务序列增长，但混淆了技能质量、任务顺序和检索策略等因素。
*   **核心问题**：**没有一个基准能有效且专一地评估智能体将过去执行轨迹中的经验，转化为可重用的过程性知识，并成功应用到新的、但具有程序性重叠的任务上的能力。** 这种“轨迹到程序”的迁移是智能体自我演进的关键，但一直被忽视。
*   **整体含义**：论文提出了一个新的评估范式，强调智能体的价值不在于每次从零开始解决问题，而在于能否从经验中提炼出可复用的“程序”（如搜索策略、调试流程、验证方法），并在未来的任务中有效利用这些程序。

### 2. 方法论：核心思想与关键技术细节

*   **核心思想**：引入“**Ability（能力）**”作为评估迁移的基本单元。Ability不是数据集标签或话题名称，而是**从智能体执行轨迹中提炼出的、可重用的操作单元**（如一个搜索策略、一个调试步骤）。通过构建“能力图（Ability Graph）”来量化任务间的程序性关联，并以此为基础设计评估拆分，确保每个测试任务都有来自训练侧的Ability支持。

*   **关键技术细节：三个阶段的构建流程**：
    1.  **阶段一：多模型轨迹收集与能力提取**：
        *   使用多个不同的骨干模型（如Kimi-K2.5, GLM-5.1, DeepSeek-V3.2）对源任务执行，收集成功和失败的轨迹。
        *   基于这些轨迹，一个**提取器**（使用Claude Sonnet 4.6）为每个任务生成原始的“能力卡片（Ability Card）”。一张卡片是一个四元组：`(触发条件, 可重用程序, 支持证据, 边界条件, 角色)`。角色分为三类：**Method**（任务求解主程序）、**Guard**（纠正错误行为）、**Workflow**（执行控制流程）。

    2.  **阶段二：能力规范化**：
        *   不同任务产生的原始能力卡片可能在描述上不同，因此进行规范的合并。采用“嵌入相似性（进行阻塞召回）+ 三模型LLM裁判（判断操作等价性）+ 领域专家审查（解决分歧）”的流程，将等价的卡片合并为**规范化能力单元**，形成统一的“能力词汇表”。

    3.  **阶段三：能力图构建与评估拆分的生成**：
        *   根据每个任务拥有的规范化能力单元，构建**无向能力图**。如果两个任务共享至少一个能力单元，它们之间就有一条边。边存储了共享能力单元的信息。
        *   使用社区发现算法（Louvain）对图进行分区，找到程序性重叠的局部区域。
        *   **关键拆分约束**：受限于“能力支持”和“实例不重叠”原则，从每个社区中抽样测试任务。**核心保证是：每个被选中的测试任务，至少与其社区内的一个训练任务共享一个能力单元。** 这确保了迁移评估是在有明确能力支持的前提下进行的，而非随机的任务划分。

### 3. 实验设计

*   **数据集与Benchmark**：
    *   覆盖四个长周期智能体领域，共2,605个源任务，最终形成 **528个训练任务和267个测试任务** 的官方评估拆分。
        *   **网络研究**：基于BrowseComp-Plus。
        *   **算法推理**：基于LiveCodeBench。
        *   **软件工程**：基于SWE-Bench Verified。
        *   **知识工作**：基于GDPVal。
*   **对比方法**：
    *   **基线**：**Vanilla**（无演进状态）。
    *   **自动化自我演进方法**：
        *   **Memento**：基于嵌入相似度检索最相似的原始案例。
        *   **ReasoningBank**：将经验蒸馏为自然语言策略，并按策略相似度检索。
        *   **GEPA**：在训练集上进化出一个全局提示，广播到所有测试任务（无检索）。
    *   **诊断参考方法**：
        *   **Anchor Skill**：使用基准自带的能力图技能。它不是一种可部署的方法，因为其路由信息（测试任务的能力标签）来自构建者，而非从执行中自动提取。它用于测试**能力内容本身是否可迁移**。
*   **评估框架**：
    *   两个智能体支架（Scaffold）：**OpenClaw**（功能完整）和 **Nanobot**（轻量级）。
    *   三个骨干模型：**Qwen3.5-27B**、**Qwen3.5-397B** 和 **Gemma-4-31B**。
    *   每个设置下，任务、工具、超时、评分标准均相同，仅演进状态不同。

### 4. 资源与算力

*   论文中**未明确提及**使用的GPU型号、数量和训练时长等具体计算资源信息。仅提及评估使用了多个骨干模型和支架，但未描述其硬件环境。

### 5. 实验数量与充分性

*   **实验数量**：总共进行了 **6个（2个支架 × 3个骨干模型）** 不同的设置。每个方法在每个设置下都运行了 **3次独立重复**，并报告了均值和标准误。总计形成了 **72个主要结果单元格**（6种设置 × 4个方法 × 3个骨干模型 + 1个诊断参考）。
*   **充分性与公平性**：
    *   **是充分的**：涵盖了多种主流模型族（Qwen、Gemma）、不同模型规模（27B、397B）和两种设计迥异的支架，结论具有一定的泛化性。
    *   **是公平的**：在所有设置中，任务、工具、评分合同均为固定，且所有自动化方法都从完全相同的训练数据开始，经历相同的协议（用自己的支架-模型对生成训练轨迹）。这确保了观察到的差异仅来自方法本身。同时，详细的成本分析也增加了评估的全面性。
    *   **客观性**：所有实验使用固定的评估协议和官方提示，结果由程序（如代码测试）或固定的LLM裁判生成，减少了主观偏差。

### 6. 主要结论与发现

*   **能力内容是可迁移的**：诊断参考方法 **Anchor Skill 在所有设置和所有领域都取得了正向的迁移增益**。这意味着基准提供的程序性能力内容本身是有效的，并且能跨模型族迁移（构建模型的骨干与评估模型的骨干不同）。
*   **自动方法不稳定**：所有当前的自动化自我演进方法都**表现脆弱**。它们在某些设置下能提升性能，但在其他一些设置下甚至出现了**负迁移**（性能下降）。没有任何一种自动方法在所有设置中保持恒定正增益。
*   **瓶颈在于方法侧**：由于内容本身可迁移，自动方法的失败点在于**经验编码、索引和路由机制的缺陷**。例如，Memento的案例检索可能因表面相似性导致“灾难性不匹配”（如在SWE领域出现-36.3%的暴跌）。
*   **成本与准确率解耦**：演进并不总能降低成本。有时，即使任务解决了，方法也增加了交互轮次（如Memento在已解决任务上的开销增加了39.4%）。因此，报告准确性时**必须同时报告成本变化**。

### 7. 优点

*   **开创性的评估框架**：精准地定义和隔离了“过程性迁移”这一核心能力，这是现有基准的重大空白。
*   **信息泄露控制严谨**：通过设计，确保了训练和测试任务只有程序性重叠，没有实例或答案上的重叠，评估信号纯净。诊断参考方法（Anchor）也严格限制其信息来源。
*   **高度结构化的诊断能力**：通过能力图，当方法失败时，可以追溯到是训练侧缺少能力、提取失败还是路由失败，为方法开发提供了明确指引。
*   **跨维度评估**：并非只在单一模型或单一领域上测试，而是涵盖了多个模型、支架和领域，结论更可靠。同时，将准确率和成本联合报告，提供了更全面的视角。

### 8. 不足与局限

*   **领域覆盖有限**：仅覆盖了四个文本为主的智能体领域。扩展到多模态、具身智能体等领域是未来工作。
*   **提取器依赖**：能力提取步骤依赖于一个单一的LLM（Claude Sonnet 4.6），能力词汇表的质量取决于该模型。
*   **测试集规模**：每个领域内测试集任务数量为56-86个，虽然足以发现范式性模式，但对于更精细的领域内分析可能规模仍然偏小。
*   **基准污染风险**：评估用的骨干模型可能在预训练阶段就已接触过源基准数据。论文指出，因为所有方法共享相同的设置，这种污染会影响绝对分数，但**不会影响方法之间的相对比较**，这是合理的。
*   **特定采样偏差**：测试任务被优先选择存在提升空间（headroom）的任务。因此，观察到的迁移增益大小（Δm）是针对这个特定支持拆分（supported split）的，而不是随机任务样本上的期望增益。这影响了绝对增益值的泛化性，但方法间的比较仍然有效。
