## 摘要
编码代理是使用大语言模型（LLM）自主解决软件工程任务的系统，它们依赖于代理脚手架：一种位于开发者与大语言模型之间的中间件层，负责编排系统提示、工具执行、上下文管理和迭代推理循环。尽管这些脚手架以极快的速度演化，但尚无研究检验这种演化如何随时间影响代理质量（即有效性和效率）。实践者经常在脚手架更新后报告质量回归，但总是将其归因于底层模型而非脚手架本身。在本文中，我们通过进行首次受控纵向研究来弥补这一空白，该研究分离了脚手架的贡献。与先前固定脚手架而改变模型的工作不同，我们固定模型而仅改变脚手架，评估了35个连续版本以衡量其对代理有效性和效率的影响。我们首先实证研究了五个主要开源脚手架（即Codex、Qwen Code、Gemini、OpenCode和OpenHands）的开发和发布演化，揭示了超过每天两个版本的极高发布速度以及数月内数千个问题。然后，我们对Qwen Code CLI的35个连续版本进行了受控深度研究，在保持底层LLM不变的情况下，针对50个分层的SWE-bench Verified任务评估每个版本。我们将由此产生的质量波动追溯到特定的开发模式和架构组件，并通过具体定性证据说明我们的发现，将单个拉取请求与测量的质量变化联系起来。

## Abstract
Coding agents, autonomous systems that use large language models (LLMs) to resolve software engineering tasks, rely on agentic scaffolding: a middleware layer in between a developer and a large language model that orchestrates system prompts, tool execution, context management, and iterative reasoning loops. While these scaffoldings evolve at extreme velocities, no study has examined how this evolution affects agent quality (i.e., effectiveness and efficiency) over time. Practitioners regularly report quality regressions after scaffolding updates, yet consistently attribute them to the underlying model rather than the scaffolding itself. In this paper, we address this gap by conducting the first controlled longitudinal study that isolates the scaffolding's contribution. Unlike prior work that fixes the scaffolding and varies the model, we fix the model and vary only the scaffolding, evaluating 35 sequential releases to measure their impact on agent effectiveness and efficiency. We first empirically study the development and release evolution of five major open-source scaffoldings (i.e., Codex, Qwen Code, Gemini, OpenCode, and OpenHands), revealing extreme release velocities exceeding two releases per day and thousands of issues within months. We then perform a controlled deep dive into 35 sequential releases of the Qwen Code CLI, evaluating each against 50 stratified SWE-bench Verified tasks while holding the underlying LLM constant. We trace the resulting quality fluctuations to specific development patterns and architectural components, and illustrate our findings with concrete qualitative evidence linking individual pull requests to measured quality shifts.

#### 6. 论文的主要结论与发现
- **RQ0（景观分析）**：代理脚手架呈现“超活跃”开发状态，平均每周10-18个版本，每天13-34个提交，PR审核中位数不到4小时，远超传统项目（VS Code、GitHub CLI）。同时，积累数千个issue且解决率偏低（如OpenCode仅54%），形成不断增长的积压。
- **RQ1（质量演化）**：
  - 解决率在35个版本间波动但无统计显著趋势（Spearman ρ=0.208, p=0.231），早期版本（如v0.0.14）达到峰值39%，后期未见改善。
  - 令牌消耗显著增加（ρ=0.743, p<0.001），后期版本比早期多消耗约70%令牌（从391K到668K），工具调用也增加。
  - 任务归一化分析证实增加源于脚手架本身，而非任务难度。
  - 失败的任务消耗2.7倍令牌和1.8倍工具调用，说明脚手架增加了无效循环。
  - 系统提示增大8%、推理轮次增加18%是令牌膨胀的驱动因素。
- **RQ2（发布级开发模式）**：
  - **特征密集型发布**与解决率正相关（feat_churn_ratio ρ=0.438），但同时也增加令牌消耗和工具调用。
  - **修复密集型发布**增加资源消耗但不提升解决率（fix_churn_ratio与令牌消耗正相关）。
  - **清理代码（高删除量）**与效率提升相关（delta分析中总删除量增加对应令牌消耗下降）。
  - **合并更大PR**（mean_pr_size）与更低的工具调用和令牌消耗相关（增量分析）。
- **RQ3（架构敏感性）**：
  - **LLM Provider层**和**Context Management**是高回归风险区域：Provider层的添加操作在绝对分析中有益，但增量分析显示一次性大量修改带来效率下降；Context Management的扩展与令牌效率负相关。
  - **Extensibility层**与更好的令牌效率相关（fix_churn增加对应效率提升）。
  - **Security层**的修复相关于效率改进（安全修复通常削减代码路径）。
  - **UI层**的绝对分析和增量分析结果相反（长期扩展增加成本，短期重构提升效果）。
- 总体结论：脚手架演化并不保证质量提升，复杂度增加伴随资源消耗翻倍而有效性未改善；需要建立针对代理的非功能性回归测试（Agentic QA）。

#### 7. 优点
- **方法创新**：首次通过固定模型、变化脚手架实现纵向受控实验，明确分离scaffolding贡献，消除了以往模型混淆效应。

- **数据公开**：提供了完整的复现包（35个版本、评估脚本、原始结果）。
- **结构性分析**：从项目级和架构级两个层次解释质量波动，提供了可操作的建议（如警惕LLM Provider和Context Management改动，优先考虑Extensibility和Security变化）。

#### 8. 不足与局限
- **外部有效性**：仅研究一个脚手架（Qwen Code CLI）和一个模型（Qwen3-Next-80B），难以推广到其他脚手架（如Codex、OpenHands）或其他模型。作者承认这一点并呼吁复现。

- **架构映射的主观性**：文件和组件映射依赖人工审查，存在一定偏差风险。
- **无对比方法**：未与基线scaffolding（如固定早期版本）或替代设计进行比较，因此无法量化“最佳版本”与“最差版本”间的差距绝对值。
