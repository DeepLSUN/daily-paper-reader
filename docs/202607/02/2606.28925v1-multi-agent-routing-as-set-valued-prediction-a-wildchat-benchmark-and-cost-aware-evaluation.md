## 摘要
基于自然语言提示的工具和智能体路由本质上是一个集值预测问题：单个查询可能需要多个智能体，而过度选择会增加执行成本。本文引入的基准测试源自WildChat，包含3000个提示，覆盖固定的12个智能体目录，在固定架构下使用AI辅助启发式标签，并通过受控重平衡进行多标签评估。评估协议结合了集合级指标（精确率、召回率、F1、Jaccard和精确匹配）、延迟、面向执行的能力覆盖模拟，以及基于有序智能体成本等级的约束加权路由设置。比较的方法包括最近邻匹配、线性多标签分类、依赖感知基线、微调编码器、通过加权智能体路由（WAR）的确定性加权后评分以及零样本LLM基线。结果表明，有监督的路由器显著优于最近邻和零样本LLM路由。微调编码器在无约束情况下取得了最强的集合准确率，而线性多标签模型提供了最实用的基线。在约束设置下，加权路由层在强有监督评分器之上应用时提高了效用，其中编码器+WAR的提升最大。总体而言，该基准测试和评估协议支持对固定目录多智能体路由中准确率-成本权衡的可重复研究。

## Abstract
Tool and agent routing from natural-language prompts is naturally a set-valued prediction problem: a single query may require multiple agents, while over-selection increases execution cost. The benchmark introduced here is derived from WildChat and contains 3,000 prompts over a fixed 12-agent catalog, with AI-assisted heuristic labels under a fixed schema and controlled rebalancing for multi-label evaluation. The evaluation protocol combines set-level metrics (Precision, Recall, F1, Jaccard, and Exact Match), latency, an execution-oriented capability-coverage simulation, and a constrained weighted-routing setting based on ordinal agent-cost tiers. Compared methods include nearest-neighbor matching, linear multilabel classification, dependency-aware baselines, a fine-tuned encoder, deterministic weighted post-scoring via Weighted Agent Routing (WAR), and a zero-shot LLM baseline. Results show that supervised routers substantially outperform nearest-neighbor and zero-shot LLM routing. The fine-tuned encoder achieves the strongest unconstrained set accuracy, while the linear multilabel model provides the strongest practical baseline. In the constrained setting, the weighted routing layer improves utility when applied on top of strong supervised scorers, with the largest gain observed for Encoder+WAR. Overall, the benchmark and evaluation protocol support reproducible study of accuracy-cost trade-offs in fixed-catalog multi-agent routing.

## 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

### 2.1 核心思想
- 将路由建模为：输入提示 $x$，输出代理集 $\hat{S}(x) \subseteq A$（$A$为固定$M$个智能体目录）。每个智能体有成本等级 $c(a_i) \in \{1,2,3\}$。目标是最大化预测集与标注集 $G(x)$ 的重叠，同时控制总成本。
- 采用**先评分后选择**的流水线：所有方法共享提示嵌入（基于MPNet的句子编码器），产生每智能体分数，再通过阈值化（含top-1后备）生成动态大小的预测集。

### 2.2 关键技术细节与基线方法
论文比较了多种方法，可分为几类：

1. **朴素基线**：Majority（输出最频繁的智能体）；KNN（基于余弦相似度的近邻评分）。
2. **监督多标签分类**：
   - **线性ML**：每智能体一个One-vs-Rest线性SVM，输入是提示嵌入，输出分数。
   - **依赖感知基线**：Classifier Chains（CC）和ML-kNN，显式建模标签共现依赖。
3. **微调编码器**：对MPNet骨干+线性多标签头做端到端微调（3 epoch，batch size 8，lr 2e-5，weight decay 1e-2），输出sigmoid概率。
4. **零样本LLM**：GPT-4o受限JSON模式，要求选择1-3个智能体。
5. **成本感知后处理——加权智能体路由（WAR）**：在分数基础上引入成本惩罚 $\tilde{s}_i(x) = s_i(x) - \lambda c(a_i)$，再阈值化。参数 $(t, \lambda)$ 在开发集上通过效用函数选择。

### 2.3 公式/算法流程（文字描述）
1. **嵌入**：提示文本 $x$ 经编码器 $f$ 得嵌入 $p$；每个智能体构建文本画像（角色、能力等）得嵌入 $v_i$。均$\ell2$归一化。
2. **评分**：各方法产生分数向量 $s_i(x)$（如KNN用点积，线性SVM用决策值，微调编码器用sigmoid概率）。
3. **集合构建**：$\hat{S}(x) = \{a_i \mid s_i(x) \ge t\}$，若为空则取分数最高者。
4. **WAR调整**：对分数做减法惩罚，再阈值化。

## 3. 实验设计：数据集 / 场景 / 基准 / 对比方法

### 3.1 数据集与基准
- **来源**：从WildChat（1M条真实ChatGPT交互日志）中筛选3,000条提示，固定12个智能体目录（覆盖数据分析、API调用、数据库查询、绘图等常见任务）。
- **标注**：AI辅助启发式标签（基于GPT-4生成，人工校验规则），每个提示标注1-3个智能体，平均1.5个。
- **重平衡**：控制每智能体出现次数约375次，集合大小分布60%单标签、30%双标签、10%三标签。划分为2400训练/300开发/300测试。

### 3.2 对比方法

### 3.3 评估指标
- **集合级**：Precision、Recall、F1、Jaccard、Exact Match (EM)、平均预测集大小。
- **执行模拟**：Task Success (完全覆盖)、Coverage、Cost (成本等级求和)、Extra (多余智能体)、Utility = Coverage - 0.1*Cost - 0.05*Extra。

- 开发集上扫描WAR的惩罚$\lambda \in \{0,0.02,0.05,0.10,0.15\}$，按效用选择。

## 6. 论文的主要结论与发现

1. **微调编码器最优**：在无约束集合预测中，Encoder达到F1 89.59±0.33、Jaccard 85.52±0.30、EM 73.11±0.19，显著领先所有基线。
2. **线性ML是强实用基线**：线性SVM（F1 71.79）在简单性和可重训练性上优势明显，远优于KNN和零样本LLM。
3. **依赖感知基线表现中等**：CC和ML-kNN提高了召回但整体F1低于线性ML，表明该基准中**语义表征质量比标签依赖建模更重要**。
4. **零样本LLM路由不可靠**：GPT-4o F1仅41.51，远低于监督方法，说明仅靠通用语言能力不足以胜任固定目录下的精准选择。
5. **WAR有效提升成本-效用权衡**：WAR在强基础评分器上效果显著。Encoder+WAR将Success从74.33%提升至92.11%，Utility从0.654提升至0.670（代价是平均成本增加）。ML+WAR则主要通过降低成本提升效用（0.465→0.480）。
6. **阈值选择敏感**：$t=0.6$为通用最优，过低过选，过高欠选。WAR的最佳惩罚系数因基础模型而异（强模型需较小$\lambda$）。

## 8. 不足与局限：包括实验覆盖、偏差风险、应用限制

1. **标注不确定性**：提示到智能体映射不是唯一确定的，不同部署可能接受不同集合。论文虽验证了协议一致性，但某些边界情况（语义重叠、多智能体冗余）仍存在歧义。
