## 摘要
大型语言模型（LLM）越来越多地被用于自动化数据处理工作流，然而编码智能体通常生成的脚本并不会自动物化为持久、可编辑的平台工件。我们将这一脱节称为“NL2Pipeline鸿沟”。为弥合这一鸿沟，我们引入了DataFlow-Harness，该平台通过类型化的增量变异（而非自由格式脚本）来指导LLM智能体构建平台原生的有向无环图（DAG）。该平台结合了用于过程指导的DataFlow-Skills、暴露实时算子注册表和当前管道状态的模型上下文协议（MCP）层，以及将对话式创作与可视化DAG编辑器同步的DataFlow-WebUI。在一个包含12个任务的数据工程基准测试中，DataFlow-Harness实现了93.3%的端到端通过率。相较于Vanilla Claude Code，其衡量货币成本降低了72.5%，生成延迟降低了49.9%；其通过率比上下文感知的Claude Code基线低0.9个百分点，而成本降低了42.8%。逐任务分析表明，当构建依赖于隐式过程知识时，Skills最为有用。这些结果表明，实时平台基础化能够生成持久、可编辑的工作流工件，其可靠性接近脚本生成基线，且构建成本和延迟更低。

## Abstract
Large language models (LLMs) are increasingly used to automate data-processing workflows, yet coding agents typically produce scripts that are not automatically materialized as persistent, editable platform artifacts. We call this disconnect the \textit{NL2Pipeline gap}. To bridge it, we introduce \textsc{DataFlow-Harness}, a platform that guides an LLM agent to construct platform-native directed acyclic graphs (DAGs) through typed, incremental mutations rather than free-form scripts. The platform combines \textsc{DataFlow-Skills} for procedural guidance, a Model Context Protocol (MCP) layer that exposes the live operator registry and current pipeline state, and \textsc{DataFlow-WebUI}, which synchronizes conversational authoring with a visual DAG editor. On a 12-task data-engineering benchmark, \textsc{DataFlow-Harness} achieves a 93.3\% observed end-to-end pass rate. Relative to Vanilla Claude Code, it reduces measured monetary cost by 72.5\% and generation latency by 49.9\%; its observed pass rate is within 0.9 percentage points of the Context-Aware Claude Code baseline while its cost is 42.8\% lower. Per-task analysis indicates that Skills are most useful when construction depends on implicit procedural knowledge. These results show that live platform grounding can produce persistent, editable workflow artifacts with an observed reliability close to script-generation baselines and with lower measured construction cost and latency.

## 1. 核心问题与整体含义

- **研究动机**：大型语言模型（LLM）越来越多地用于自动化数据处理工作流，但现有编码智能体直接生成自由格式脚本，这些脚本难以自动物化为持久、可编辑的平台工件（如可视化DAG、可复用管道），导致“NL2Pipeline鸿沟”——用户用自然语言表达工作流需求，但生产环境需要结构化、持久的管道资产。
- **整体含义**：弥合这一鸿沟不能仅靠提升代码生成准确率，而需要让智能体在平台语义中接地（grounded），生成与宿主平台集成的原生工件。

## 2. 方法论

### 核心思想
- 将工作流构建从自由格式脚本转变为**类型化的增量变异**，驱动LLM智能体构建平台原生的有向无环图（DAG），而非一次性生成完整代码。
- 通过三个解耦组件实现：DataFlow-Skills（过程知识）、Model Context Protocol（MCP）层（实时算子注册表与管道状态）、DataFlow-WebUI（对话式界面与可视化DAG编辑器同步）。

### 关键技术细节
- **Data Pipeline Backend**：管道表示为 \( P = (D, O, E, S, R) \)，包含数据源、算子实例、有向边、输入输出模式、运行时状态。智能体通过类型化变异（添加/删除算子、更新参数、连接边）与后端交互，变异只有通过无环检查和模式兼容性检查后才提交。
- **MCP Tools层**：遵循“请求-验证-提交”协议。智能体通过MCP工具调用获取实时管道状态和执行变异；验证引擎检查DAG无环性和算子间架构兼容性；已验证的变更提交到后端并通过WebSocket同步到所有客户端。
- **DataFlow-Skills**：将两类领域知识注入Claude Code推理上下文：**过程蓝图**（schema推断、算子选择、参数配置、服务验证等推荐序列）和**组合约束**（算子兼容规则如模态匹配、字段流约定），引导正确选择与组合算子。
- **DataFlow-WebUI**：提供对话界面（用户自然语言描述需求，智能体通过MCP调用执行）和可视化DAG编辑器（用户可直接检查和编辑，手动变更立即同步到后端）。

## 3. 实验设计

### 数据集/场景与Benchmark
- 构建了一个**12个任务**的数据工程基准测试，涵盖六类场景：QA生成、审查治理、长文档处理、多维评分、模式规范化、低质量过滤。每个任务提供自然语言目标、输入数据样本和任务特定验收标准。
- **补充评估**：教科书到VQA提取任务（FlipVQA-Miner设置），评估正确性和覆盖率。
- **下游训练效用**：数学推理管道（Prompt 1：数学数据清洗与合成）和通用SFT管道（Prompt 2：从零生成指令微调数据）。

### 对比方法

## 6. 主要结论与发现

- **有效性（RQ1）**：DataFlow-Harness端到端通过率93.3%，比MCP-only高10.0个百分点，与Context-Aware CC（94.2%）相差仅0.9个百分点。结构化DAG合成可达到接近脚本基线的可靠性。
- **效率（RQ2）**：相比Vanilla CC，成本降低72.5%，延迟降低49.9%；相比Context-Aware CC，成本降低42.8%，延迟降低17.6%。主要驱动力是Token消耗大幅降低。
- **Skills作用（RQ3）**：过程指导在依赖隐式领域知识的任务（如QA生成）上提升最大（从18/30到29/30）；在简单变换/过滤任务上无提升；在非合成瓶颈任务（如数值约束）上无帮助。
- **下游训练效用（RQ4）**：数学管道中，DataFlow-Harness产生的数据在AIME24/25上显著优于Vanilla CC（例如1 epoch: AIME24@32 35.9 vs 25.1）；通用SFT管道中，代码基准平均提升2.3个百分点（63.8 vs 61.5），表明接地帮助生成更高质量的管道。
- **教材VQA任务**：DataFlow-Harness在精确度（0.972）和覆盖率（0.873）上均领先，说明有助于有效复用平台算子。
