## 摘要
代理技能将程序性指令和检查打包以供通用代理使用，但加载技能并不能保证在长时间的工具使用轨迹中每个要求都保持活跃。我们在一个源自生产环境的白盒代码审计工作流中研究这一问题。在保持任务和24个工件检查不变的情况下，我们改变周围上下文，并对失效首次变得可见的地方进行分类：丢失要求、编辑漂移、检查失败或非代理评估器/运行时失败。使用gpt-5.4-mini的Codex在10991字符的干净上下文中通过了8/10次运行，但在299140字符的相关上下文和等长的不相关上下文中仅通过3/10次运行。这50个百分点的差异很大，但在双侧Fisher检验下仍处于趋势水平（p = 0.0698）。然而，在两种长上下文条件下，需求覆盖率仍保持在92%以上，表明少数遗漏可能使原本完整的工件失效。第二个任务通过了所有干净和长上下文运行，因此证据不支持通用上下文长度阈值。详细的外部检查表通过了10/10次运行，而通用自我检查仅为5/10（p = 0.0325）。编码代理脚手架可能通过选择更小的工作集有所帮助，但并未消除失效。我们不引入上下文腐烂或新的通用监控方法；我们为白盒代码审计提供了有界失效分类和实证案例研究。

## Abstract
Agent Skills package procedural instructions and checks for use by general-purpose agents, but loading a skill does not guarantee that every requirement remains active throughout a long tool-using trajectory. We study this problem in a production-derived, white-box code-audit workflow. Holding the task and 24 artifact checks fixed, we vary the surrounding context and classify where failures first become visible: lost requirements, editing drift, failed checking, or non-agent evaluator/runtime failures. Codex with gpt-5.4-mini passes 8/10 runs in a 10,991-character clean context but only 3/10 in both a 299,140-character relevant context and an equal-length irrelevant context. This 50-percentage-point difference is large but remains trend-level under two-sided Fisher tests (p = 0.0698). Requirement coverage nevertheless stays above 92% in both long conditions, showing that a few omissions can invalidate an otherwise complete artifact. A second task passes all clean and long runs, so the evidence does not support a universal context-length threshold. A detailed external checklist passes 10/10 runs, compared with 5/10 for a generic self-check (p = 0.0325). Coding-agent scaffolds may help by selecting a smaller working set, but they do not eliminate failures. We do not introduce context rot or a new general monitoring method; we provide a bounded failure classification and empirical case study for white-box code auditing.

- 在代码审计这种高风险场景中，任何一个关键要求的遗漏（如缺少必填数组、修改受保护字段、未完成指定子任务）都可能导致整个审计工件无效。

### 核心问题

## 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

### 核心思想
- 采用**白盒测试方法**（White-Box Testing）：评估者可以观察技能指令、代理读取的文件、编辑操作、工具日志、完成消息和外部评分后的工件。
- 提取生产环境代码审计工作流中的强制性要求，转化为24个可观察的确定性检查项。
- **只改变周围上下文（Context），保持任务、起始工件和检查项完全固定**，从而隔离上下文长度和质量对代理可靠性的影响。

### 关键技术细节

#### 1）从指令到可观察检查
- 从工作流中提取五类规范性文本：显式要求、禁止项、条件分支、字段计数限制、完成规则。
- 每项重写为关于可见事件或保存文件状态的谓词。例如：“所有复数证据字段必须是数组，包括没有证据时的空数组”变为存在性和类型检查。

#### 2）工作区清理与泄漏控制
- 从早期含答案泄漏的工作区中，通过允许列表（allowlist）构建干净工作区。
- 每个尝试从头开始一个干净副本，记录文件哈希、确切提示、模型和脚手架标识、更改文件、工具输出、使用计数器和检查器输出。

#### 3）上下文构造
- **干净条件（Clean）**：最小完整任务上下文。
- **长相关条件（Relevant Long）**：添加同一工作流的生产材料（299,140字符）。
- **长无关条件（Irrelevant Long）**：添加等字符数的不相关自然档案。
- 字符匹配提供可复现的粗略大小控制，但不保证等量标记、结构复杂度或检索路径。

#### 4）失败分类（四类）
- **需求丢失（Lost Requirement）**：强制定项从代理的任务表示或最终工件中缺失。
- **编辑漂移（Editing Drift）**：需求存在但编辑操作违反它（如覆盖受保护内容）。
- **检查失败（Failed Checking）**：工件或工具输出暴露错误，但代理验证未纠正。
- **非代理失败（Non-Agent Failure）**：评估器或运行时导致（如流断开、评估器假阴性）。

### 关键公式
- **需求覆盖率（Requirement Coverage）**：\( \frac{1}{K} \sum_{i=1}^K z_i \)，其中 \(z_i\) 指示检查 \(i\) 是否通过。
- **任务成功（Task Success）**：\( I[\sum_{i=1}^K z_i = K] \)，即所有关键检查必须通过。

### 算法流程
1. 从工作流提取强制定要求。
2. 正反测试验证确定性检查器。
3. 构建干净工作区，注入一种上下文条件。
4. 运行编码代理，保留完整可观察记录。
5. 根据最终工件评分，然后根据分类指南标记首次可见失败位置。

## 6. 论文的主要结论与发现

### 核心发现
1. **长上下文显著增加失败**：干净上下文通过8/10，而长相关和长无关上下文各通过3/10。风险上升50个百分点（p=0.0698，不确定趋势）。
2. **相关性不是主要驱动因素**：相关和无关长上下文具有相同的二分结果（3/10，p=1.0）。
3. **失败是稀疏且集中的**：需求覆盖率在长上下文中仍高于92%，但只有30%的工件完全通过。丢失一个关键数组可使23/24通过变为失败。
4. **详细外部检查清单显著优于通用自我检查**：详细通过10/10 vs 通用通过5/10（p=0.0325）。通用自我检查往往继承生成的遗漏，而外部清单则恢复需求。
5. **脚手架可以部分帮助**：代码代理通过检索更小的工作集来减轻长上下文的影响。GPT-5.5 在 Full Direct 下失败（1/3），但在 Bundled Direct 和 Codex Relevant 下成功（3/3）。然而，脚手架并不能消除失败（gpt-5.4-mini即使在Codex下也仅通过3/10）。
6. **沉默失败（silent failures）很普遍**：在历史清单中，86.4%的失败运行声称“成功”或“完成”。
7. **跨任务异质性**：Revert SL-007 在所有上下文条件下均完美通过，表明该问题并非通用上下文长度阈值，而是模型-任务-脚手架-上下文构造的组合。
8. **操作可靠性是一个主要问题**：仅50.9%的尝试产生可评分输出，且缺失不随机。在无关长条件下，80%的尝试中断（20/25）。

## 7. 优点：方法或实验设计上的亮点

### 方法亮点
- **白盒设计**：固定任务、起始工件和检查项，只改变上下文，隔离了因果变量。
- **细致的失败分类**：提供四类失败（丢失需求、编辑漂移、检查失败、非代理失败）和明确的判断规则，比二分“通过/失败”提供了更多可操作信息。
- **分离“能力”与“可靠性”**：通过分离干净任务能力、上下文可靠性效应和操作中断，回答了三个常被混淆的问题。
- **高保真度实验控制**：工作区清理、泄漏控制、已知检查器的正/负测试、记录完整轨迹（包括所有工具输出和完成消息）。
- **双重报告指标**：同时报告严格任务成功率和需求覆盖率，揭示了“高覆盖率但低成功率”的测量陷阱。
- **人工验证基础设施**：审计28次中断尝试，防止“有用”运行掩盖操作失败的偏误。

### 实验设计亮点
- **在中期调整后扩大样本量**：明确披露了从5次扩大到10次，并展示了估计值的变化（无关长上下文0/5 → 3/10）。
- **多角度边界检验**：跨任务、跨模型、脚手架变化使得结果更可靠，防止过度泛化。
- **统计严谨性**：使用 Fisher 精确检验和 Wilson 区间，适用于小样本。避免对相关检查项进行独立性测试。

## 8. 不足与局限：包括实验覆盖、偏差风险、应用限制等

### 实验覆盖不足
- **仅有一个主要任务**：仅 `Stage 4CD` 有重复效应。Revert SL-007 在所有条件下稳定通过，表明效应高度特异。
- **仅有一个主要模型**（gpt-5.4-mini）+一个脚手架（Codex）。跨模型探针太小，无法得出可信比较。
- **样本量小**：主要比较中各10次，导致统计效力低（p=0.0698，未达0.05）。置信区间较宽。
- **无第二验证任务**：无独立任务在干净条件下稳定通过并在长上下文中稳定退化。

### 偏差风险
- **未预先注册且未固定样本量**：在观察5次运行估计值后决定扩展至10次，尽管已披露，但非纯确认性分析。
- **高且不平衡的中断率**：无关长上下文仅20%可评分 vs 干净77%、详细78%。如果容易失败的尝试更易中断，则“有效”运行中的成功率可能被高估。
- **相关与无关上下文的构建**：仅通过字符数匹配，未匹配标记数、句法/语义密度或检索路径。并非纯粹的长度处理。
- **模型名称不确定**：内部网关可能服务于不同版本，不同模型名称含义不同。
