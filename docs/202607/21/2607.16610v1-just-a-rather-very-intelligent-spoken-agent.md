## 摘要
长视界AI代理正变得越来越强大，但它们与用户的交互仍然出奇地薄弱。在大多数工作流程中，用户给出初始指令后，只会收到选择性的文本更新，并且对代理正在做什么或何时介入失去清晰的认识。这在当前的代理生态系统中留下了一个缺失的部分：一个始终在线的贾维斯式中介，使代理能够持续与用户保持联系。这样的中介应支持与用户的实时语音交互，在不中断工作进程的情况下回答问题，主动报告进度或困惑，并在有用时将用户指导注入到代理的执行中。在这项工作中，我们引入了JarvisBench，这是一个衡量长视界代理工作流中中介双重价值的基准。JarvisBench包含两个互补的轨道：一个代理协作轨道，衡量中介是否改善了下游任务完成情况；以及一个用户交互轨道，衡量中介是否使正在进行的执行更易理解、响应更快且对用户更可访问。我们用模块化参考Jarvis原型实例化了该基准，并在OpenClaw中执行的34个纯文本WildClaw任务上进行了评估。使用GPT-5.5、Claude Opus 4.7、基于Gemini和基于GPT的工作代理的初步结果表明，贾维斯式中介可以为用户问题提供基于轨迹的响应，并在适当时刻注入稀疏用户指导时提高任务性能。结果还表明，有效性强烈依赖于中介的LLM大脑，凸显了这一缺失中间层的潜力以及需要更广泛的社区努力。演示页面 https://cchen1436.github.io/jarvis

## Abstract
Long-horizon AI agents are becoming increasingly capable, yet their interaction with users remains surprisingly thin. In most workflows, users give an initial instruction, receive only selective textual updates, and lose a clear sense of what the agent is doing or when to step in. This leaves a missing part in the current agent ecosystem: an always-on Jarvis-style mediator that keeps the agent continuously reachable to the user. Such a mediator should support real-time spoken interaction with the user, answer questions without interrupting the worker, proactively report progress or confusion, and inject user guidance back into the agent's execution when useful. In this work, we introduce JarvisBench, a benchmark for measuring the dual value of mediation in long-horizon agent workflows. JarvisBench contains two complementary tracks: an agent-collaboration track that measures whether mediation improves downstream task completion, and a user-interaction track that measures whether mediation makes ongoing execution more understandable, responsive, and accessible to users. We instantiate the benchmark with a modular reference Jarvis prototype and evaluate it on 34 text-only WildClaw tasks executed in OpenClaw. Preliminary results with GPT-5.5, Claude Opus 4.7, Gemini-based, and GPT-based worker agents suggest that Jarvis-style mediation can provide trace-grounded responses to user questions and improve task performance when sparse user guidance is injected at appropriate moments. The results also show that effectiveness depends strongly on the mediator's LLM brain, highlighting both the promise of this missing middle layer and the need for broader community effort. - **问题**：当前长时程AI代理虽能力增强，但与用户的交互模式仍停留在**薄弱的单次任务启动+少量文本更新**，用户无法实时了解代理的进展、困难或偏离意图，也难以适时干预。
- **缺失层**：缺少一个**始终在线、支持语音交互的“贾维斯”式中介**，该中介应能持续监控工人代理的轨迹，主动向用户汇报进度/困惑，回答用户问题，并将用户简洁的指导注入回工人代理的执行中。
- **研究目标**：提出 **JarvisBench** 基准，系统衡量这种中介在长时程代理工作流中的双重价值——**改善任务完成**与**增强用户实时理解与可控性**。同时搭建参考原型Jarvis，验证可行性。

### 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：设计一个**插拔式的中介层**（Jarvis），位于用户和工人代理之间。中介不独立求解任务，而是：
  - 观察工人代理的ReAct轨迹（推理摘要、工具调用、观察、错误、下一步动作）。
  - 识别失败敏感检查点（重复工具失败、进展停滞、模糊观察、危险不可逆操作、低置信度终结）。
  - 向用户（专家或非专家）发起交互，如主动报告、回答问题。
  - 将用户反馈（简洁修正或确认）注入回工人代理的上下文中，继续执行。

- **关键技术细节**：
  - **JarvisBench双轨设计**：
    - **代理协作轨道**（Track 1）：评估中介是否提高下游任务完成率。对比同一工人代理是否带Jarvis。Jarvis仅触发检查点并传递专家指导，不解任务。
    - **用户交互轨道**（Track 2）：评估中介是否使工人执行过程更可理解、响应更快。固定工人代理，变化Jarvis大脑；非专家用户提问（进度类/主题类），评价回答的轨迹扎实度、任务一致性。
  - **参考Jarvis原型**：
    - **级联全双工语音接口**：流式ASR → 语义VAD（SoulX-Duplug） → LLM大脑 → 流式TTS（Chatterbox）。支持始终监听、主动说话。
    - **轨迹监控与上下文管理**：分离对话上下文和代理执行上下文；控制状态记录Jarvis当前角色（回答问题、等待专家、准备注入指令）。
    - **交互检查点**：合作暂停机制。在检查点时Jarvis记录工人状态，暂缓执行，完成交互后恢复并注入指导。Track 1检查点是失败敏感的；Track 2是定时或随机提问点。

### 6. 论文的主要结论与发现

- **中介改善任务完成**：所有工人代理在有Jarvis中介时得分更高。最大提升：Claude Opus 4.7从64.01%升至75.79%（+11.78%），Gemini-3.1-Pro提升+11.64%。干预次数很少（最多8次），表明改进源于关键检查点的稀疏专家指导，而非代理替换。
- **中介提升用户理解**：非专家用户可以通过Jarvis获得基于轨迹的进度解释和任务上下文回答。Jarvis大脑能力强烈影响回答质量：GPT-5.4在主题问题上最佳（Topic 3.91，73.9%≥4），GPT-OSS-120B在进度解释上最佳（Progress 4.07，93.8%≥4）。
- **跨模型失败模式**：最难失败点不是普通推理，而是精确性验证（精确标识符、输出格式、硬约束、用户文件状态、安全边界）。中介通过强制稀疏验证可缓解部分失败。
- **中介大脑重要性**：弱LLM可能引入错误或不适时干预，强闭源模型带来更稳定收益。

### 7. 优点

- **问题定义新颖**：识别了当前代理生态系统中缺失的“中介”层，并提供了系统化评估基准。
- **双轨设计全面**：同时评估中介对任务完成和用户体验的价值，分开衡量。
- **模块化与可替换性**：Jarvis原型支持替换LLM大脑、语音模块、工人代理、监控策略等，便于社区贡献和改进。
- **真实环境与任务**：使用WildClawBench中的真实工具（浏览器、shell、文件系统、邮件、日历），任务设计面向实际场景。
- **用户模拟器增强可复现**：GPT驱动模拟非专家与专家用户，降低人工成本，便于复现比较。
- **定性案例充实**：附录提供具体轨迹片段，直观展示中介如何纠正覆盖文件、强制执行约束、验证外部源等。
