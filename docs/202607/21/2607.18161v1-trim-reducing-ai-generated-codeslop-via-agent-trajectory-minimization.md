## 摘要
编程智能体越来越多地被用于加速代码生成，用于修复错误、构建应用程序和原型设计等下游任务。然而，尽管它们作为编程助手具有价值，智能体生成的代码往往比相应的人类编写的实现更大、更冗长。在这项工作中，我们表明原因在于智能体自身的搜索过程：在迭代寻找通过的解决方案时，智能体会积累推测性编辑、放弃的假设以及最终补丁中保留的临时更改。对于单个补丁来说，这似乎无害，但随着智能体负责代码库中越来越大的部分，问题会累积——曾经最小化且维护良好的代码库逐渐积累冗余，其速度超过了清理速度，从而进入一个更难以维护的状态。考虑到问题的严重性，我们迈出了一步来缓解这一问题。首先，我们正式将此现象定义为“代码垃圾”——AI生成代码中常见残留的、功能上不必要的编辑。然后，我们提出了算法TRIM（轨迹引导的冗余识别与最小化）。TRIM不直接最小化代码垃圾，而是最小化智能体轨迹。正如我们经验性地展示，这种间接最小化代码垃圾的技术非常有效：TRIM在各种智能体框架上将代码垃圾减少了17.9%-32.9%，且性能退化可以忽略不计。TRIM也非常高效，其验证成本大约为Delta调试等算法基线的一半。

## Abstract
Coding agents are increasingly used to accelerate code generation in many downstream tasks, such as fixing bugs, building applications, and prototyping. However, despite their value as coding assistants, agent-generated code tends to be larger and more verbose than the corresponding human-written implementation. In this work, we show that the cause lies in the agent's own search process: while iterating toward a passing solution, an agent accumulates speculative edits, abandoned hypotheses, and temporary changes that persist into the final patch. This may seem harmless for a single patch, but the problem compounds as agents take responsibility for ever-larger portions of a codebase-a codebase that was once minimal and well-maintained slowly accumulates redundancy faster than it can be cleaned up, drifting to a state that is harder to maintain. Given the magnitude of this problem, we take a step towards alleviating this issue. First, we formally define this phenomenon as CodeSlop-the residual and functionally unnecessary edits commonly seen in AI-generated code. We then introduce our algorithm TRIM (Trajectory-guided Redundancy Identification and Minimization). Rather than minimizing CodeSlop directly, TRIM instead minimizes agent trajectories. As we show empirically, this indirect technique of minimizing CodeSlop is highly effective: TRIM cuts CodeSlop by 17.9%-32.9% across agentic scaffolds, with negligible performance regression. TRIM is also highly efficient, requiring roughly half the validation cost of algorithmic baselines such as Delta Debugging.

## 1. 核心问题与整体含义（研究动机与背景）

- **问题**：AI编码智能体（如SWE-Agent、Open Hands等）在修复缺陷或生成代码时，会经历多轮编辑-测试的搜索过程。即使最终补丁通过测试，其中往往残留了大量不必要的修改——包括推测性编辑、废弃的假设、临时变更等。这些残留称为“CodeSlop”（代码垃圾），导致补丁冗长、可读性差、维护成本高。
- **观察**：随着智能体承担代码库中越来越大的部分，冗余会加速积累，使代码库从原本简洁、良好维护的状态逐渐退化。
- **研究动机**：当前大多工作仅关注补丁是否通过测试，而忽略了补丁质量。本文首次形式化定义CodeSlop，并提出算法来系统性减少它，以提升AI生成代码的可维护性。

## 2. 论文提出的方法论

### 核心思想
- 不直接在最终补丁上搜索冗余（因为修改间相互依赖，搜索空间指数级），而是利用智能体**修复轨迹（repair trajectory）**中自然的时序层次结构——智能体在每次测试反馈之间执行一组编辑（称为编辑序列E_i）。这些编辑序列反映了智能体的探索过程，也隐含了依赖关系。
- 将CodeSlop识别转化为**层次化反事实搜索**：先尝试移除整个编辑序列，若补丁仍通过测试且变小则接受；再细化到文件级别；最后到单个编辑动作。

### 关键技术细节
1. **轨迹重构**：从原始轨迹中提取编辑序列和测试反馈，忽略非修改操作（如文件查看、目录导航）。形成简化轨迹 `Raj_R = ⟨(E1, FR1), (E2, FR2), ..., (Ek, FRk)⟩`，其中 `Ei = ⟨ei1, ei2, ..., ein⟩`。
2. **层次化反事实搜索**（Algorithm 1）：
   - 三个粒度依次执行：**编辑序列（Edit Sequence）** → **文件（File）** → **编辑动作（Edit Action）**。
   - 在每个粒度上，将当前存活编辑按该粒度分组，逆序遍历候选组，临时移除一组，如果补丁通过测试套件 `T_F` 且长度严格减小，则永久移除该组。
   - 重复直到无更多移除（若启用 oneMin 标志，则保证达到1-最小性；否则仅单遍）。
3. **验证**：每个反事实候选必须通过智能体原本使用的测试套件（`T_F`），确保行为等价。
4. **变体**：
   - **TRIM-G**：启用 oneMin，保证1-最小性，最坏复杂度 O(|EditAction|²)。
   - **TRIM-NG**：禁用 oneMin，每个粒度只扫一遍，复杂度 O(|EditAction|)，实际效果接近TRIM-G但验证成本更低。

- **TRIM有效减少CodeSlop**：在Live-kBench上减少17.9%-32.9%（全粒度TRIM-G），Oracle正确性几乎无退化（甚至略有提升）。

## 7. 优点

- **问题定义清晰**：首次将CodeSlop定义为**可移除功能冗余**，而非静态风格属性，精准刻画问题本质。
- **方法创新**：利用轨迹时序结构进行层次化反事实搜索，避免盲目组合搜索，高效利用依赖信息。
- **可靠且成本可控**：TRIM不会失败（确定性验证），而智能体最小化失败率高；成本-质量可调，用户可提前终止。
- **适用性广**：在安全修复（C内核）和通用仓库（Python）均有效，仅需适配轨迹预处理。
- **实验严谨**：使用隐藏Oracle进行保守评估，纳入多种基线并统计失败情况，结果有说服力。

## 8. 不足与局限

- **最小粒度局限**：原子编辑动作是TRIM的最小单元，若单个编辑内包含部分冗余，则无法消除（保守估计）。
- **依赖测试质量**：验证仅依靠智能体使用的测试套件，可能遗漏未测试的回归（虽然实验显示罕见）。
- **仅限于成功修复**：TRIM只处理已通过测试的补丁，不适用于未成功的探索。
- **评判偏差**：Live-kBench的语义等价性使用LLM评判，存在潜在偏差，可能影响绝对增益估计。
- **基准覆盖有限**：仅评估两个基准（Linux内核和Python仓库）及四种智能体，在其他语言或更复杂系统的效果未知。
- **未讨论编辑之间复杂交互**：TRIM假设编辑可独立移除，但某些冗余可能只有在其他冗余移除后才可移除；TRIM-G通过多次迭代部分解决，但最坏情况仍可能不完整。
