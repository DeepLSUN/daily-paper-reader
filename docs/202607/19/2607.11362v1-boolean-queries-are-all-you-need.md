## 摘要
我们为基于LLM的搜索代理配备了布尔检索引擎，用于搜索TREC 2024 RAG赛道使用的MS MARCO V2.1去重片段集合。在86个主题的标准赛道子集上，在每主题100次模型调用的预算下，该代理实现了0.6863的NDCG@10，这一成绩超过了许多密集、稀疏和学习的稀疏第一级检索器。排名仅基于与查询匹配的语料子串密度，不需要监督学习、全局统计或术语权重。形式上，查询语言表达了正则语言的严格子集，文档得分基于其包含的匹配数量和长度。虽然这些结果更多是探索性的而非决定性的，因为它们基于单个在模型训练期间公开可用的测试集合，但结果表明简单的模式匹配可能足以用于代理搜索。

## Abstract
We equipped an LLM-based search agent with access to a Boolean retrieval engine to search the MS MARCO V2.1 deduped segment collection used by the TREC 2024 RAG track. Over a standard track subset of 86 topics, and operating under a budget of 100 model calls/topic, the agent achieved an NDCG@10 of 0.6863, which would place it above many dense, sparse, and learned-sparse first-stage retrievers. Ranking is based solely on the density of corpus substrings matching a query, with no requirement for supervised learning, global statistics, or term weights. Formally, the query language expresses a strict subset of the regular languages, with a document's score based on the number and length of matches it contains. Although the results are more exploratory than definitive, because they are based on a single test collection that was publicly available during model training, they suggest that simple pattern matching may be sufficient for agentic search.

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：在当前以密集检索、语义匹配为主导的智能搜索（agentic search）和检索增强生成（RAG）背景下，是否仅凭简单的布尔查询与模式匹配就能达到甚至超越复杂检索方法的性能？  
- **研究动机**：历史表明，布尔检索在专业用户手中效果显著，但普通用户难以有效构造查询。大语言模型（LLM）恰好可弥补这一短板，自动生成精确的布尔查询。  
- **整体含义**：挑战了“复杂语义匹配（如密集双编码器、交互式模型）是智能搜索必备”的主流观点，提出“简单模式匹配可能足以支撑代理搜索”，为低资源、无监督的检索方案提供了新视角。

## 2. 方法论：核心思想、关键技术、算法流程

- **核心思想**：将语料视为单个连续字符串，通过布尔查询找出所有“最短子串”匹配，基于匹配的长度和数量对文档进行排序，无需全局统计、词权重或监督学习。
- **关键技术**：
  - **最短子串排名（Shortest Substring Ranking, SSR）**：每个布尔查询的解是满足查询但无更短子串的连续文本片段。枚举所有最短子串，计算文档得分：  
    `score(S) = Σ_{i=1}^{|S|} 1/(C + qi - pi)`，其中 `C=42` 为平滑常数，`qi-pi` 为子串长度。越短的匹配贡献越大，匹配越多得分越高。
  - **Boolean 查询语言**：支持 `+`（AND）、空格（OR）、短语（引号）、嵌套括号，是正则语言的严格子集。
  - **Cottontail 搜索引擎**：实现“注解索引”，支持动态增量更新和事务性读写，能在 `O(m log n)` 时间内枚举最短子串（`n` 为最频繁词的出现次数，`m` 为解的数量）。
- **Vole 搜索代理**：由 LLM（gpt-5.5）、指令集和交互执行器组成。LLM 在每次调用时可选三种动作：① 执行新查询（返回顶部文档的 200 词片段）；② 请求下一结果；③ 查看全文。LLM 需对每个新文档做出 0-3 级相关性判断。交互限制：最多 50 个独立文档判断、最多 100 次模型调用。

## 3. 实验设计

- **数据集**：MS MARCO V2.1 deduped segment 集合（TREC 2024 RAG Track 所用语料）。共 301 个主题，其中 86 个主题有部分人工标注，全部主题均有 UMBRELA 工具（基于 gpt-4o）生成的 LLM 判断。
- **基准与对比方法**：与 WaterlooClarke 团队在 TREC 2024 RAG Track 提交的 11 种运行（run）对比，包括 BM25F、BM25F + 查询扩展、密集检索器、RankGPT 重排序、融合排名（uwc0/uwc1/uwc2）等。官方排名中 uwc1 为 top-3 方法。
- **评估指标**：NDCG@10（使用官方 UMBRELA 判断填充 Vole 检索出的未标注文档的“空洞”）。
- **实验规模**：仅在 86 个主题的子集上进行了单次实验，未进行多数据集或多任务验证。

## 4. 资源与算力

- **文中明确指出**：Vole 每主题平均消耗不到 74 次模型调用（上限 100 次），而对比方法 uwc1 平均需 194 次模型调用。  
- **未提及**：具体 GPU 型号、数量、训练时长等硬件细节。模型推理使用了 gpt-5.5（LLM）和 gpt-4o（UMBRELA），但无本地训练或微调过程。  
- **说明**：由于该方法完全免训练，算力成本主要来自 LLM API 调用，而非 GPU 训练。

## 5. 实验数量与充分性

- **实验组数**：单一实验场景（单集合、单任务），无消融实验、无跨数据集迁移测试、无超参数敏感性分析。  
- **充分性评估**：  
  - **优点**：与官方竞赛提交的 11 种方法（覆盖从简单到复杂的多种技术）进行了直接对比，对比体系完整。  
  - **不足**：仅报告了一个结果（NDCG@10 = 0.6863），未做统计显著性检验，未验证不同 LLM、不同索引策略或不同查询语言子集的影响。  
  - **客观性**：作者指出“结果更多是探索性的而非决定性的”，因为测试集在模型训练期间已公开（存在潜在的污染风险），且仅基于一个语料库。实验设计未能完全排除过拟合或偶然因素。

## 6. 主要结论与发现

- **主要结论**：基于布尔查询和最短子串排名的 LLM 驱动搜索代理（Vole）在 NDCG@10 上达到 0.6863，超过了多数第一级密集、稀疏和学习稀疏检索器（如 BM25F、密集检索器），仅低于使用了 LLM 重排序的顶级方法（如 uwc1 的 0.8199）。  
- **关键发现**：简单模式匹配（无需向量、全局统计或学习权重）在代理搜索中具有竞争力，甚至可能超越传统的第一阶段检索技术。  
- **额外发现**：LLM 生成的布尔查询有时会包含罕见短语或高频词组合，导致索引查询延迟（作者指出了一个优化问题，已修复）。

## 7. 优点

- **极简设计**：无需监督学习、全局统计、词权重或向量模型，仅靠子串匹配和长度打分，降低了部署和维护成本。  
- **动态更新能力**：Cottontail 的注解索引支持实时事务性更新，适合代理搜索中需要持续写入新信息（如模型判断、用户反馈）的场景。  
- **低模型调用成本**：Vole 平均每主题 74 次调用，优于多数需要二次重排序方法（如 uwc1 的 194 次）。  
- **历史意义重现**：重新验证了布尔检索在 LLM 辅助下的潜力，呼应了 1990 年代手工布尔查询仍能超越自动系统的历史结论。

## 8. 不足与局限

- **实验覆盖严重不足**：仅使用单一语料（MS MARCO V2.1 的一个子集）和单一任务（TREC RAG 2024），未在多个异构数据集（如法律、医学、新闻等领域）上验证，结果泛化性存疑。  
- **数据泄露风险**：测试集（MS MARCO V2.1）在模型训练期间公开可用，LLM 可能已“记住”文档内容，从而影响布尔查询生成和判断的公平性。  
- **无消融实验**：未分离各组件（如 SS 排名 vs. BM25、不同 LLM、不同 prompt 等）的贡献，无法确定性能提升的关键来源。  
- **仅依赖 LLM 判断**：最终评估使用了 gpt-4o 生成的 UMBRELA 判断，模型可能对自身生成的查询有偏好（self-bias），缺乏独立人工验证。  
- **未与全监督方法公平对比**：对比方法（如 uwc1）使用了大量 LLM 重排序调用（194 次），而 Vole 仅 74 次，但未控制总推理成本（token 数量或计算量）进行同等比较。  
- **工程优化未充分**：提出了查询延迟问题（如“make this issue known”类短语导致慢查询），但未系统评估不同查询模式下的效率与鲁棒性。
