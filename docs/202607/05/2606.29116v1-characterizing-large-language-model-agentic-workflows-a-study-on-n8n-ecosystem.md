## 摘要
大型语言模型（LLM）正迅速被应用于低代码和无代码自动化平台，非专业用户在这些平台上设计结合自然语言理解与外部服务和API的工作流。LLM代理是使用LLM作为核心“大脑”来推理、规划并自主执行复杂多步骤任务的LLM系统。在本文中，我们首次对低代码自动化平台中的LLM代理工作流进行了大规模实证研究。我们分析了超过6,000个公开可用的n8n工作流，并考察了其设计的四个方面：任务分布、结构与工具使用模式、可靠性机制以及自主性水平。我们的分析表明，LLM工作流不仅仅是提示响应管道。相反，LLM通常嵌入在更广泛的自动化结构中，涉及控制逻辑、外部工具、通信服务、存储系统和人工审查点。我们进一步发现，尽管许多工作流在LLM执行后包含轻量级后处理或路由逻辑，但显式的可靠性机制（如结构化回退路径、修复循环、特定故障警报和人工审批关口）仍然相对少见。这些结果揭示了LLM代理在实际自动化生态系统中的日益部署与可靠性、安全性和治理方面的工程支持有限之间的差距。总体而言，我们的研究为寻求理解和改进真实世界LLM代理工作流的研究人员、平台开发者和从业者提供了十项实证发现和五项研究启示。

## Abstract
Large Language Models (LLMs) are rapidly being adopted in low-code and no-code automation platforms, where non-expert users design workflows that combine natural language understanding with external services and APIs. LLM agents are LLM systems that use LLMs as a core "brain" to reason, plan, and autonomously execute complex, multi-step tasks. In this paper, we present the first large-scale empirical study of LLM agentic workflows in low-code automation platforms. We analyze more than 6,000 publicly available n8n workflows and examine four aspects of their design: task distribution, structural and tool use patterns, reliability mechanisms, and autonomy levels. Our analysis shows that LLM workflows are not merely prompt response pipelines. Instead, LLMs are commonly embedded within broader automation structures involving control logic, external tools, communication services, storage systems, and human review points. We further find that while many workflows include lightweight post-processing or routing logic after LLM execution, explicit reliability mechanisms such as structured fallback paths, repair loops, failure-specific alerts, and human approval gates remain relatively uncommon. These results reveal a gap between the increasing deployment of LLM agents in practical automation ecosystems and the limited engineering support for reliability, safety, and governance. Overall, our study provides ten empirical findings and five research takeaways for researchers, platform developers, and practitioners seeking to understand and improve real-world LLM agentic workflows.

## 6. 论文的主要结论与发现

- **10 项实证发现（Findings）**：
  1. 工作流拓扑复杂：仅 23.27% 为线性链，多数包含分支、合并、循环（18.92%）。
  2. LLM 主要作为中间处理模块，主导邻居模式为“逻辑节点 → LLM → 逻辑节点”（27.11%）。
  3. AI 依赖模式多样化：除了基础聊天模型（24.06%），工具调用（16.57%）、输出解析器（8.34%）、内存（5.85%）等组合普遍。
  4. 代理组合：最简配置（仅聊天模型）占 37.12%，但相当比例扩展了输出解析器、工具、内存等。
  5. 可靠性机制以本地设置为主：on error control（48.45%）、retry on fail（34.44%）、always output data（29.26%）。
  6. 只有 684 个工作流（占含错误处理工作流的 37.4%）的锚点能追溯到 LLM 输出。
  7. LLM 错误处理常通过中间解析器或逻辑节点间接实现。
  8. 工作流作者意识到错误处理，但多采用局部配置而非端到端恢复设计。
  9. 仅 167 个工作流（2.78%）包含显式人工审批路径。
  10. 逻辑门控自动化和直接自动化动作覆盖 68.32% 的工作流，表明 LLM 输出经常直接或经逻辑后到达外部服务。
- **5 项研究启示（Takeaways）**：LLM 输出应视为草案而非最终动作；外部动作需由人类决策把关；LLM 输入是预处理后的表示；LLM 输出需工作流级中介；下游检查不一定确保语义正确性。
- **总体结论**：LLM 代理工作流在 n8n 中实际部署广泛但可靠性工程支持滞后，显式回退、修复循环、人工审批等机制罕见，存在显著安全与治理缺口。

## 8. 不足与局限：实验覆盖、偏差风险、应用限制

- **静态分析的固有局限**：工作流 JSON 无法捕获运行时行为（如 LLM 实际输出、外部服务响应、用户意图），因此发现的可靠性机制是设计级证据而非运行时保障。
- **数据集偏差**：仅分析**公开**的 n8n 工作流模板，可能偏向演示、教学或可复用模板，大量生产级私有工作流未纳入，样本代表性受限。
- **分类误差风险**：部分 LLM 调用可能通过通用 HTTP 节点实现（而非显式 LLM 节点），导致漏检；控制节点参数不完整时可能误分类。论文已通过确定性规则和诊断报告部分缓解。
- **外部有效性限制**：结论仅适用于 n8n 生态系统，向其他平台（如 Zapier、Make）泛化需验证。不同平台的工作流设计范式和节点类型差异可能改变模式分布。
- **深度剖析不足**：未分析 LLM 输出质量、幻觉检测效果、用户反馈等；未量化可靠性机制的实际效果。例如，带输出解析器的工作流是否真正减少了错误传播。
- **计算资源缺失**：未报告 LLM 标注和数据分析所需的算力消耗，影响可复现性评估。
