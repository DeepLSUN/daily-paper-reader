## 摘要
LLM智能体日益作为基于智能体框架构建的源代码应用程序进行开发。这些智能体程序将传统宿主语言代码与框架定义的模型、提示词、工具、记忆以及多智能体编排逻辑语义相结合。因此，其行为不仅依赖于传统的控制流和数据流，还依赖于一类新的智能体依赖关系。此类依赖关系通常以框架诱导的语义形式表达，例如智能体构造函数、工具装饰器和智能体交接声明，使得现有静态分析或依赖追踪工具难以恢复。本文提出AgentFlow，这是首个用于从智能体程序中恢复和分析智能体依赖关系的静态分析框架。AgentFlow构建了智能体依赖图——一种与框架无关的图表示，将智能体、提示词、模型、能力、记忆状态和控制策略表示为类型化节点，并将其组件依赖、控制流和数据流依赖捕获为类型化边。基于ADG，AgentFlow支持一系列用于智能体治理和安全的分析，包括生成智能体物料清单和提示词到工具的风险检测。我们针对五个代表性智能体框架实现了AgentFlow，并在包含5,399个真实世界智能体程序的语料库AgentZoo上进行了评估。评估表明，与现有基于AST的智能体静态分析工具相比，AgentFlow恢复了更丰富的智能体实体和依赖关系，生成了更具依赖感知能力的智能体物料清单，并在真实世界智能体程序中发现了238个污点风格的提示词到工具风险。这些结果表明，ADG为理解、治理和保护新兴智能体软件提供了实用基础。

## Abstract
LLM agents are increasingly developed as source-code applications built on agent frameworks. These agent programs combine conventional host-language code with framework-defined semantics for models, prompts, tools, memory, and multi-agent orchestration logic. As a result, their behavior depends not only on traditional control and data flows, but also on a new class of agent dependencies. Such dependencies are often expressed as framework-induced semantics, such as agent constructors, tool decorators, and agent handoff declarations, making them difficult to recover with existing static analysis or dependency tracking tools. In this paper, we present AgentFlow, the first static analysis framework for recovering and analyzing agent dependencies from agent programs. AgentFlow constructs an Agent Dependency Graph (ADG), a framework-agnostic graph representation that represents agents, prompts, models, capabilities, memory states, and control policies as typed nodes, and captures their component-dependency, control-flow, and data-flow dependencies as typed edges. Built on ADGs, AgentFlow supports a range of analyses for agent governance and security, including Agent Bill of Materials (BOM) generation and prompt-to-tool risk detection. We implement AgentFlow for five representative agent frameworks and evaluate it on AgentZoo, a corpus of 5,399 real-world agent programs. Our evaluation shows that AgentFlow recovers richer agent entities and dependencies than existing AST-based agent static analysis tools, generates more dependency-aware Agent BOMs, and uncovers 238 taint-style prompt-to-tool risks in real-world agent programs. These results show that ADG provides a practical foundation for understanding, governing, and securing emerging agent software.

- **背景**：LLM 智能体正越来越多地作为基于框架的源代码应用程序开发。这些智能体程序融合了传统的宿主语言代码与框架定义的语义（模型、提示词、工具、记忆、多智能体编排逻辑），从而引入了一类新的**智能体依赖关系**。
- **问题**：现有静态分析或依赖追踪工具难以恢复这类智能体依赖，因为它们通常以框架诱导的语义形式出现（如构造函数、工具装饰器、智能体交接声明），且在不同框架间存在异构性。传统依赖抽象无法直接表示这些依赖，导致无法有效进行智能体程序的治理和安全分析。
- **含义**：需要一种系统化的方式来建模智能体依赖，以支持智能体物料清单生成、提示词到工具的污点风险检测等下游应用。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
提出 **AgentFlow**，首个静态分析框架，用于从智能体程序源代码中恢复并分析智能体依赖。其关键中间表示是**智能体依赖图**，一种框架无关的图表示，包含类型化节点和类型化边。

### 关键技术细节

- **智能体依赖图组成**：
  - **节点类型**：智能体、提示词、模型、能力（工具/MCP/技能）、记忆状态、控制策略。
  - **边类型**：三种依赖边：
    - 组件依赖边（ACDG）：静态绑定关系（如智能体与提示词、模型、能力、记忆、策略的绑定）。
    - 控制流边（ACFG）：可能的执行转移（智能体到能力、智能体到智能体，可附带守卫策略）。
    - 数据流边（ADFG）：信息传播（提示词 → 智能体、智能体 → 能力参数、能力返回、智能体间消息、内存读写等）。

- **分析流程**（三个阶段）：
  1. **事实提取前端**：识别并解析框架特定构造（如 @function_tool、Agent 构造函数、handoff 等），进行别名分析，输出标准化的事实（实体事实、组件绑定事实、控制流事实、数据流事实）。当前覆盖 5 个框架（OpenAI Agents SDK、LangChain/LangGraph、CrewAI、LlamaIndex、Semantic Kernel），共 143 个构造。
  2. **智能体内部依赖分析**：基于本地事实初始化每个智能体内部的组件绑定、控制流和数据流。
  3. **智能体间依赖分析**：处理跨智能体的转移（如 handoff）和数据传播（显式消息、共享内存状态），构建跨智能体的控制流和数据流边。

- **设计原则**：采用**过度近似**（over-approximate）策略，因为智能体行为部分由运行时 LLM 输出决定，静态分析只能捕获程序结构允许的依赖，而非单条确定执行轨迹。

- **RQ1**：AgentFlow 成功分析了 59/60 个项目（1 个超时），恢复的中位节点数 17，中位边数 28，远多于 Agent-Wiz（中位节点 4，无边）和 Agentic Radar（中位节点 2.5，中位边 2）。AgentFlow 能恢复能力类型（工具、MCP、技能）、记忆状态、控制策略以及数据流依赖，而基线工具仅部分覆盖。

## 7. 优点

1. **创新性**：首次提出框架无关的智能体依赖图中间表示，系统化建模智能体程序中隐藏的依赖关系。
2. **实用性强**：支持多种下游应用（BOM 生成、污点风险检测），并且已在 5 个流行框架上实现。
3. **大规模评估**：在 5,399 个真实项目上进行实验，结果具有统计意义。
4. **对比全面**：与多个现有工具（2 个 AST 分析工具、3 个 BOM 工具）进行定性和定量比较，突出了 ADG 的优势。
5. **人工验证**：对风险检测结果进行了人工审计，给出了精确率和假阴性估计，增加了可信度。
6. **设计合理**：采用过度近似策略适应智能体行为的概率性，同时保持静态分析的可行性。
7. **可扩展性**：分析时间和图大小在实践中可接受。

## 8. 不足与局限

1. **静态分析固有局限**：过度近似导致假阳性，无法捕获运行时实际依赖（论文承认）。
2. **框架覆盖有限**：目前仅支持 5 个 Python 框架，且仅针对 Python 语言。Agent-Wiz 支持 10 个框架，AgentFlow 的覆盖需扩展。
3. **用户自定义构造处理不足**：对于不遵循框架标准 API 或自定义编排的逻辑，效果较差（论文提到 9 个漏报原因与自定义包装等有关）。
4. **语义注册表需持续更新**：框架快速演进，需维护语义规则。
5. **仅捕获高层级语义**：ADG 不替代底层程序依赖分析，对于需要深度追踪工具实现内部或库调用的情况，需与指针分析等结合。
6. **BOM 评估缺乏标准化基准**：虽然对比了三个 BOM 工具，但缺乏统一标准的 Agent BOM 定义，比较主要依赖论文定义的 schema。
7. **假阴性估计有限**：仅人工检查了 100 个非报告案例，可能未覆盖所有漏报模式。
8. **无消融实验**：未分析各组件（如别名分析、不同类型边）对最终效果的贡献。
