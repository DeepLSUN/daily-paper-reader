## 摘要
LLM和智能体可以从自然语言需求生成网络爬虫，但由于依赖错误、选择器失效、模式不匹配和页面结构异构，直接生成仍然不可靠。我们提出了一个受限可验证的智能体框架，将LLM的输出从自由格式代码转变为类型化的JSON收集器配置，结合了六种收集器分类、模板和效用函数约束、静态Airflow DAG执行、基于规则的质量检查以及结构化反馈纠正。对138个任务的实验表明，该分类支持基于描述的需求类型化，同时确认稳定的实例化需要完成超出初始描述的源、字段和执行约束。在80个独立源验证的任务上，该框架以零执行阶段的LLM令牌和最低的平均挂钟时间运行，用适中的一次性质量换取了可重用、确定且可验证的执行路径，适用于重复的定时收集。这些结果使该框架成为重复开放网络数据收集的可重用、低成本且可验证的执行路径。

## Abstract
LLMs and agents can generate web scrapers from natural-language requirements, but direct generation remains unreliable because of dependency errors, broken selectors, schema mismatches, and heterogeneous page structures. We propose a constrained, verifiable agent framework that shifts LLM output from free-form code to typed JSON collector configurations, combining a six-type collector taxonomy, template and utility-function constraints, static Airflow DAG execution, rule-based quality checking, and structured feedback correction. Experiments on 138 tasks show that the taxonomy supports description-based requirement typing, while confirming that stable instantiation requires completing source, field, and execution constraints beyond the initial description. On 80 independently source-verified tasks, the framework runs with zero execution-stage LLM tokens and the lowest average wall-clock time, trading moderate one-shot quality for a reusable, deterministic, and verifiable execution path suited to repeated scheduled collection. These results position the framework as a reusable, low-cost, and verifiable execution path for repeated open-web data collection.

- **核心问题**：LLM 和智能体可从自然语言需求生成网络爬虫代码，但由于依赖错误、选择器失效、输出模式不匹配、网页结构异构等，直接生成在开放网络数据收集场景中高度不可靠。传统手动编写爬虫成本高、可复用性差。如何使自动化收集过程变得**可验证、可审计、低成本**成为关键挑战。
- **整体含义**：论文主张不应追求完美的智能体推理，而应使收集管线**可验证**，即每一阶段的输入输出均可被检查、度量和审计。通过将 LLM 输出从自由代码转为受约束的类型化 JSON 配置，并引入静态 DAG 执行和规则质量门控，实现“可控、显式的失败”优于“不可控、静默的成功”，从而为重复调度收集提供低成本、确定性、可验证的执行路径。

## 2. 方法论

### 核心思想
- **约束生成**：将 LLM 的任务从“生成自由格式代码”转变为“生成类型化 JSON 收集器配置”，并施加四层约束：任务类型约束、配置结构约束、工具复用约束、执行反馈约束。
- **可验证闭环**：形成“生成 → 验证 → 评估 → 反馈 → 重新生成”的闭环，每个环节的输出可被规则检查。
- **静态执行**：执行阶段完全脱离 LLM，通过预构建的通用收集器操作符（generic collector operator）和静态 Airflow DAG 运行，避免 LLM 引入运行时不确定性和错误。

### 关键技术细节
1. **六种收集器类型分类法**：Search、List、Detail、API、Interactive、File，每种具有明确的边界和组合规则（例如 Search+Detail）。
2. **需求理解智能体**：接收用户自然语言描述 \( d \)，通过主动探测目标网站（主页、导航、sitemap、样本页等）获取源上下文 \( x \)，输出完整任务表示 \( R = (d, s, x, f, c, o) \)（s: 源标识, f: 字段集, c: 约束, o: 输出格式）。输出受预定义 JSON Schema 约束。
3. **收集器实例化智能体**：基于模板槽填充（template slot-filling），从需求表示 \( R \) 选择模板 \( T_k \)，填充槽位 \( \theta_k \in \Theta_k \)，生成 JSON 配置 \( M = \text{Instantiate}(T_k, \theta_k) \)，并满足校验函数 \( \text{Check}(M) = \text{True} \)。模板硬编码了 Airflow 结构、错误处理、持久化等；智能体只填充 URL、选择器、字段映射、清洗规则。提示词包含系统提示、少样本示例、需求表示和槽填充指令。
4. **Airflow 验证与批量执行**：验证阶段使用小样本参数（有限页数、强制存档原始页面），批量阶段使用生产参数。静态 DAG 保证 Scheduler 解析稳定。
5. **质量检查模块**：基于规则的确定性检查（无需 LLM），计算五维综合得分 \( S = w_s S_s + w_c S_c + w_r S_r + w_a S_a + w_e S_e \)（权重: 结构 0.35, 内容 0.25, 相关性 0.20, 附件 0.10, 执行 0.10）。输出质量报告 \( Q \)，包含分项得分、通过/失败、错误标签、原始页面对比（区分源缺陷与解析错误）、纠正提示。
6. **反馈纠正模块**：将质量报告 \( Q \) 转为结构化反馈约束 \( \Gamma_t \)，包括要修复的槽、禁止假设、候选纠正方向。维护一个失败黑名单 \( B_t \) 用于单调剪枝。纠正优先级：解析错误 > 模式错误 > 运行时错误 > 内容噪声 > 低相关性。核心算法为状态机（算法1），输出 \( \Gamma_t \) 或要求人工介入。

## 6. 主要结论与发现

1. **需求类型分类**：基于自然语言描述的六类型识别在 138 任务上精确匹配率为 0.7101，Macro-F1 为 0.7793（规则基线为 0.7562）。仅凭描述不足以可靠分类，特别是 Detail 和 List 类型混淆较多，需通过主动探测提供结构先验。
2. **收集器实例化**：Ours 在平均质量（0.5369）、通过率（0.5507）、类型匹配率（0.8551）上均优于无约束 LLM 生成（0.4876, 0.5217, 0.1884）和规则模板基线（0.3184, 0.0942, 0.7536），证实类型化槽填充和 Schema 约束提升了对齐度。但质量仍低于完全指定任务的水平，说明仍需扩展描述通过探测完成。
3. **质量-成本权衡**：在 80 个源验证任务上，运行时 LLM 方法（Schema-constrained LLM 质量 0.7051，通过率 0.8125；Crawl4AI+LLM 质量 0.6996，通过率 0.8000）的一次性质量优于 Ours（质量 0.5631，通过率 0.5000），但 Ours 执行阶段耗用 0 个 LLM token，平均挂钟时间最低（5.052s vs 14.257s-20.194s）。框架适合重复调度场景，LLM 成本一次性摊销。
4. **反馈纠正有效性**：在注入缺陷的任务上，反馈纠正将平均质量从 0.5087 提升至 0.9007，通过率从 0.0072 提升至 1.0000，平均纠正轮次 1.99，验证了修复机制的有效性（已知缺陷类型下）。
5. **工程验证**：所有 Ours 生成的配置均复用共享模板和工具函数（复用率 1.0000 vs 无约束生成 0）；80 任务上运行成功率为 80.0%（注意通过率 50.0% 要求 S≥0.60）；静态 DAG 完全通过 Airflow 2.8.0 测试。
6. **核心发现**：可控、约束化的配置生成和确定性执行路径，以适度牺牲一次性质量换取了零运行时 LLM 成本、可验证性和可调度性，适合重复开放网络数据收集。

## 7. 优点

- **创新性设计**：将 LLM 从“代码生成器”转变为“配置生成器”，并通过四层约束（类型、结构、工具复用、反馈）实现可验证闭环，思想独特。
- **极低的执行成本**：执行阶段完全不需要 LLM，token 消耗为 0，时间成本显著低于运行时 LLM 方法，适合大规模、重复调度。
- **高度可验证性**：输出为类型化 JSON 配置，可被规则检查；执行通过静态 DAG 进行，每一步结果可审计；质量检查基于规则而非 LLM 主观判断，结果可复现。
- **结构化反馈而非自然语言反馈**：纠正模块将质量报告转为结构化约束和黑名单，避免 LLM 自修复中的错误检出困难问题。
- **模型无关且工程兼容**：框架可使用任何指令微调 LLM，且与主流工作流引擎 Airflow 兼容，易于集成到现有生产环境。
- **系统性实验验证**：从需求理解到执行、质量检查、反馈纠正进行全链路实验，设计控制变量和多基线对比。

## 8. 不足与局限

- **数据集规模有限**：138 个任务、80 个验证任务，数量较少，且为自建数据集，缺乏广泛认可的公开基准（如 MacroBench、WebArena 等），外推性受限。
- **缺少长期调度实验**：未进行持续重复调度（如每日运行）的稳定性、容错性验证；未提供统计显著性检验（如 t 检验）。
- **反馈纠正实验的局限性**：仅测试了人为注入的特定缺陷类型，不能证明框架具备通用自修复能力；实际场景中缺陷类型可能更复杂。
- **对网站变化的敏感性**：框架依赖配置中的选择器和字段映射，网站结构变化后配置会失效，需重新生成。论文未深入讨论动态适应机制。
- **一次生成质量较低**：在一次性收集任务（非重复调度）中，Ours 质量（0.5631）明显低于运行时 LLM 方法（>0.7），不适合对质量要求高的单次提取。
- **未处理认证与反爬**：框架假设公开可访问数据，不绕过认证和 CAPTCHA，限制了应用范围。
- **未进行大规模多任务调度实验**：仅验证了单个任务配置的 DAG 执行，未测试同时调度数百/数千个任务时的系统性能。
- **基线方法选择偏重 LLM 方案**：未与经典非 LLM 爬虫框架（如 Scrapy + 手工规则）直接比较，虽然传统方法成本高但可能质量更稳定。
