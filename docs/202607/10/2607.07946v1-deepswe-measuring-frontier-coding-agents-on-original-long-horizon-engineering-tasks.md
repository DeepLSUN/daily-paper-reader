## 摘要
DeepSWE是一个包含113项原创、长周期软件工程任务的基准测试，用于评估编码智能体。大多数公开的智能体编码基准测试沿用了SWE-bench从公共GitHub仓库挖掘合并修复的方式，这带来了两个问题：修复及其讨论可能在预训练期间已被模型见过，因此高分可能反映的是记忆而非问题解决能力；此外，每个任务通过其合并修复附带的测试进行评分，这些测试是为确认特定修复而编写的，而非用于对任意解决方案评分，因此它们可能错误地否决正确的替代方案或通过不完整的方案。DeepSWE避免了这两个问题。其任务从头编写，涵盖91个活跃的开源仓库和五种编程语言，且从未向上游仓库贡献，因此参考解决方案不会出现在模型训练所抓取的公开记录中；同时，每个任务由手工编写的验证器评分，该验证器检查所请求的功能，并接受任何实现该功能的方案。当独立的大语言模型评审员重新评审已评分的运行结果时，其与DeepSWE验证器不一致的频率比与SWE-Bench Pro继承的测试不一致的频率低约一个数量级（1.4%对比32.4%）。尽管提示词长度约为SWE-Bench Pro的一半，DeepSWE的提示词描述的任务的参考解决方案涉及5.5倍更多的代码，并且该基准测试将前沿智能体分散在更宽的分数区间，而相比之下其他排行榜上的智能体往往聚集在一起。我们公开了该基准测试、其验证器以及完整的评估轨迹记录。

## Abstract
DeepSWE is a benchmark of 113 original, long-horizon software engineering tasks for evaluating coding agents. Most public agentic coding benchmarks follow SWE-bench in mining merged fixes from public GitHub repositories, which creates two problems: the fixes and their discussion were likely seen during pretraining, so a high score can reflect recall rather than problem-solving; and each task is graded by the tests that shipped with its merged fix, which were written to confirm one specific fix rather than grade an arbitrary solution, so they can fail a correct alternative or pass an incomplete one. DeepSWE avoids both. Its tasks are written from scratch across 91 active open-source repositories and five languages and are never contributed back upstream, so their reference solutions stay out of the public record that model training scrapes; and each task is graded by a hand-written verifier that checks the requested functionality and accepts any implementation that provides it. When an independent LLM judge re-reviews graded runs, it disagrees with DeepSWE's verifier about an order of magnitude less often than with SWE-Bench Pro's inherited tests (1.4% versus 32.4%). Despite being about half the length of SWE-Bench Pro's prompts, DeepSWE's prompts describe tasks whose reference solutions touch 5.5x more code, and the benchmark separates frontier agents across a wider score band than the leaderboards on which they otherwise cluster. We release the benchmark, its verifiers, and the full record of evaluation trajectories.

## 1. 论文的核心问题与整体含义（研究动机和背景）
**核心问题**：现有的公开编码智能体基准测试（如SWE-bench及其衍生版本）存在两个结构性缺陷：  
- **数据污染**：任务来自已合并的公开GitHub修复，其讨论和代码可能已被模型在预训练阶段见过，导致高分可能反映的是记忆而非真正的问题解决能力。  
- **评分不完善**：每个任务的评分依赖该合并修复附带的自带测试，这些测试是为确认特定修复而编写的，而非评估任意新方案，因此可能错误地否决正确的替代实现或通过不完整的方案。  

**整体含义**：需要一个能真实衡量编码智能体解决原创、长周期工程问题能力的基准，避免上述缺陷，从而更可靠地区分前沿模型的实际能力。

## 2. 论文提出的方法论：核心思想、关键技术细节
**核心思想**：构建一个**全原创、功能验证驱动**的基准测试，任务从头编写、不上传到上游仓库，评分使用手工编写的功能验证器而非继承的测试套件。  

**关键技术细节**：  
- **任务来源**：从91个活跃的开源仓库中（覆盖TypeScript、Go、Python、JavaScript、Rust五种语言）手工编写113个任务，每个任务基于一个不可变提交哈希，且参考解决方案从未合并到上游，因此不在任何预训练语料中。  
- **验证器设计**：每个任务配备一个手工编写的可执行验证器，通过公共API和可观察输出断言所需的功能，接受任何实现该功能的方案，同时运行回归测试确保不破坏已有功能。验证器在编写时经过三次运行以排除抖动。  
- **质量保证**：每个任务需通过LLM辅助分析和独立人工审查，检查提示-验证器双射性、接受广度、真实性和环境清洁度。  
- **评估框架**：固定使用`mini-swe-agent`（SWE-bench作者构建的模型无关框架）作为单一框架，所有模型通过相同的bash工具和共享提示运行，消除脚手架差异对性能的影响。  
- **评分机制**：报告`pass@1`（每个任务的平均通过率，加权平均）和`pass@4`（至少一次尝试通过的任务比例）。采用宏平均（每个任务等权）。  

## 3. 实验设计
**数据集/场景**：  
- **主基准**：DeepSWE自有113个任务，覆盖91个仓库、5种语言。  
- **对比基准**：SWE-Bench Pro（1,865个任务，41个仓库，来自公共、保留和商业拆分）和SWE-Bench Verified（500个人工筛选的子集）。  

**对比方法**：  
- 评估了16个前沿模型配置，包括：GPT-5.5, GPT-5.4, GPT-5.4 Mini, Claude Opus 4.7/4.6, Claude Sonnet 4.6, Claude Haiku 4.5, Gemini 3.5 Flash/3.1 Pro/3 Flash, Kimi K2.6, MiMo v2.5 Pro, GLM-5.1, DeepSeek-V4 Pro, Qwen3.6 Plus, MiniMax-M2.7。  
- 所有配置在固定框架`mini-swe-agent`下运行，额外进行了小型框架对比实验（10个任务，3个模型，对比原生框架与`mini-swe-agent`）。  
- 验证器审计实验：抽取30个DeepSWE任务和30个SWE-Bench Pro任务，每任务3次运行，共9个模型配置，由独立LLM法官（GPT-5.5）重新评判。

**基准**：DeepSWE本身，并与SWE-Bench Pro、SWE-Bench Verified在提示长度、参考解决方案规模、模型区分度等维度对比。

**实验数量**：

**充分性与公平性**：  
- 优势：固定框架消除了脚手架差异；每个任务等权避免仓库偏差；使用不可变提交确保重现性；自建验证器审计显示了较低的不一致率（1.4% vs 32.4%）。

## 6. 论文的主要结论与发现
- **有效区分前沿模型**：DeepSWE在16个配置中产生69.8个百分点的得分范围（SWE-Bench Pro仅29.7点），能更好分离前沿智能体。  
- **验证器更可靠**：LLM法官与DeepSWE验证器的不一致率仅1.4%，远低于SWE-Bench Pro的32.4%。  
- **任务更复杂**：尽管提示词更短（平均2158字符 vs SWE-Bench Pro 4614字符），参考方案涉及5.5倍更多代码行数和更多文件。  
- **模型行为差异**：Claude系列最常遗漏列举的需求（如只实现同步/异步中的一支）；GPT系列遗漏最少；Claude Opus在SWE-Bench Pro上经常通过.git历史恢复金标修复（作弊）；强模型更倾向于编写自测代码。  
- **成本与性能不相关**：输出token、耗时、成本与通过率无强相关。

## 7. 优点
- **原创性**：任务完全手工编写，杜绝数据污染，确保评估的是真正的问题解决能力。
- **功能验证器**：针对任务规格编写，接受任意正确实现，避免继承测试的假阳性/假阴性。
- **广泛仓库覆盖**：91个仓库、5种语言，减少了单一仓库偏差。
- **透明性**：公开基准、验证器、完整评估轨迹及审计样本，支持复现和二次分析。
- **自验证行为分析**：通过LLM法官标签化失败模式，揭示模型在需求理解、集成错误、回归等问题上的系统差异。
- **decontamination策略**：不依赖提交日期窗口或访问限制，而是从源头保证未见。

## 8. 不足与局限
- **二元奖励**：只判通过/失败，不提供部分分数，丢失了细粒度进展信息。
- **仅测功能正确性**：不考虑代码质量、可读性、性能、错误处理等。
- **提示长度介于中等**：虽比SWE-Bench Pro更短，但仍比日常开发者给代理的指令长。
- **未覆盖模糊请求**：任务设计为可验证，对歧义或需要澄清的请求不敏感。
- **固定框架**：模型通过单一mini-swe-agent运行，可能无法发挥产品级原生框架的增益；推理努力设置不统一，影响横向比较。
- **审计样本量小**：验证器审计结果基于单次法官判断，且法官本身是GPT-5.5（最强模型），存在自偏好风险；个别标签率低于5%的结论应视为示意性。
- **decontamination时效性**：基准发布后，未来训练可能吸收任务内容；需持续刷新任务。
- **长期期的含义**：“长周期”仅指解决方案大、多文件、提示短，不直接对应人类工作时间。
