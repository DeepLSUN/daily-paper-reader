## 摘要
基于大型语言模型（LLM）的多智能体系统通过协调推理和行动实现复杂问题求解，但其分布式结构也引入了诊断系统级故障的新挑战。当执行失败时，由于长程交互和紧密耦合的智能体行为，难以确定哪个智能体应承担责任以及轨迹在何时首次变得不可逆转地错误。本文研究基于LLM的多智能体系统中的故障定位问题，提出AgentLocate框架，该框架将故障归因于特定智能体和最早的关键步骤。AgentLocate将基于LLM的判断机制与独立评估者的多视角验证相结合，通过置信度感知策略聚合评估结果。产生的反馈进一步用于通过轻量级微调来适配判断器，从而提高归因质量。我们在两个互补基准上评估AgentLocate，涵盖不同任务、智能体配置和轨迹长度。实验结果表明，AgentLocate在识别责任智能体和故障步骤方面始终优于现有故障定位方法，同时在令牌使用和运行时间方面保持高效。

## Abstract
Large language model (LLM) based multi-agent systems enable complex problem solving through coordinated reasoning and action, but their distributed structure also introduces new challenges in diagnosing system-level failures. When an execution fails, identifying which agent is responsible and at what point the trajectory first becomes irreversibly misdirected is difficult due to long-horizon interactions and tightly coupled agent behaviors. In this paper, we study the problem of failure localization in LLM-based multi-agent systems and present AgentLocate, a framework that attributes failures to both a specific agent and the earliest decisive step. AgentLocate combines an LLM-based judging mechanism with multi-perspective verification by independent evaluators, whose assessments are aggregated using a confidence-aware strategy. The resulting feedback is further used to adapt the judge through lightweight fine-tuning, improving attribution quality. We evaluate AgentLocate on two complementary benchmarks covering diverse tasks, agent configurations, and trajectory lengths. Experimental results show that AgentLocate consistently outperforms existing failure localization methods in identifying both responsible agents and failure steps, while remaining efficient in terms of token usage and running time.

## 2. 方法论：核心思想、关键技术细节与算法流程

- **核心思想**：将故障定位视为一个“产生假设 → 多视角验证 → 自适应改进”的可迭代过程，而非一次性判断。
- **整体框架（AgentLocate）** 包含三个阶段：

### 2.1 决定性故障假设生成（Decisive Failure Hypothesis Generation）
- 使用一个基于LLM的Judge，给定失败轨迹τ和用户查询q，通过推理协议（prompt）生成假设 `(î, ˆt)`，即责任智能体及其最早关键错误步骤。
- 支持两种协议：**All-at-once**（一次性查看完整轨迹）和**Step-by-step**（逐步检查前缀，一旦满足反事实反转条件即输出）。

### 2.2 评估器验证（Evaluator Verification）
- 多个独立评估器（Evaluator）分别重新分析同一轨迹，基于Judge的假设（î, ˆt）产生自己的预测 (i_h, t_h)，附带理由r_h和置信度c_h。
- 评估器使用不同风格的prompt（base、concise、evidence），鼓励多样化推理。
- 通过置信度加权投票聚合：`A(i,t)=∑c_h * I{(i_h,t_h)=(i,t)}`，选择总支持最高的 `(i_eval, t_eval)` 作为验证输出。

### 2.3 自适应Judge改进（Adaptive Judge Improvement）
- 将Judge的假设、各评估器的输出及聚合结果打包成训练实例 `z_q`，收集所有失败轨迹构成微调数据集D。

- 微调后的Judge `Judge_ft` 用于最终推断：`(i_final, t_final) = Judge_ft(p, τ, q)`。

- **算法流程伪代码**：遍历所有失败查询q，调用Judge生成假设；各Evaluator独立评估并输出；计算聚合结果；构造实例放入D；最后用D微调Judge。

## 6. 主要结论与发现

1. **AgentLocate在责任智能体和步骤定位上均显著优于所有基线**，在两个基准上均取得最高准确率（Algorithm-Generated all-at-once下agent-level达69.05%，step-level达38.10%）。

## 7. 优点

- **方法新颖**：将故障定位建模为可验证的迭代过程，结合LLM Judge与多评估器，并通过微调自适应改进，避免一次性判断的脆弱性。
- **设计合理**：置信度加权投票有效聚合多视角结果；不同的评估器prompt风格促进互补推理；LoRA微调高效利用验证反馈。
- **实验全面**：覆盖多个数据集、模型、协议、消融、效率分析，对比了大量基线，结果可信。
- **实用性强**：在令牌消耗和运行时间方面高效，适合在真实调试流程中多次运行。
- **通用性**：不仅适用于任务级错误，还扩展到Aegis-Bench的错误模式分类（虽提升有限），展示了跨场景适应性。

## 8. 不足与局限

- **步骤级精度在长轨迹上仍较低**：Hand-Crafted子集（平均50步）上的step-level准确率最高仅37.5%（step-by-step），说明长时间依赖的因果定位仍然困难。
- **依赖LLM作为判断器**：性能受底层LLM推理能力影响；若Judge或Evaluator本身存在偏差，可能传播到最终结果。实验使用7B/8B模型，更大模型可能带来更好结果，但未探索。
- **对错误模式识别的提升有限**：在Aegis-Bench的pair-level准确率上虽然最高，但与基线差距不大，表明框架更擅长“谁”和“何时”，而非“何种错误”。
- **训练数据需求**：需要收集失败的轨迹并构造验证实例；如果可用失败数据少，微调效果可能受限。论文只在数据充足的情况下验证。

- **评估器设计固定为三种prompt**：未深入探究最佳评估器组合或自适应选择策略，可能不是最优配置。
