## 摘要
大型语言模型越来越多地作为集成开发环境（IDE）的编码代理部署，用于分解任务、生成和编辑文件、运行代码以及在多轮交互中优化输出。然而，它们的安全评估仍通常沿用聊天机器人模式：一次有害提示，一次响应，孤立判断。我们引入了工作流级越狱构建，这是一种失效模式，其中有害目标是通过软件开发生命周期中的普通阶段组合而成，而非通过单个直接提示生成。通过在Visual Studio Code中使用GitHub Copilot，我们研究了四种闭源后端：Claude Sonnet 4.6、Claude Haiku 4.5、Gemini 3.1 Pro和Gemini 3.5 Flash。在来自Hammurabi's Code、HarmBench和AdvBench的204个提示中，模型在直接聊天、CSV读取和单步代码修复基线条件下几乎完全拒绝响应，每个基线条件下仅有8/816次成功响应。然而，在完整工作流下，相同的提示和后端产生了816/816次不安全的教学示例补全，所有结果均由两位专家评估员在严格标准下独立确认。这些结果表明，对话式拒绝基准可能显著夸大部署的编码代理的安全性，并促使防御措施在多轮IDE工作流及其生成的工件中推理安全性，而不仅仅是单个聊天轮次。

## Abstract
Large language models are increasingly deployed as IDE-integrated coding agents that decompose tasks, generate and edit files, run code, and refine outputs over many turns. Yet their safety is still often evaluated as if they were chatbots: one harmful prompt, one response, judged in isolation. We introduce workflow-level jailbreak construction, a failure mode in which a harmful objective is assembled across ordinary stages of a software-development workflow rather than generated through a single direct prompt. Using GitHub Copilot in Visual Studio Code, we study four closed-weight backends: Claude Sonnet 4.6, Claude Haiku 4.5, Gemini 3.1 Pro, and Gemini 3.5 Flash. Across 204 prompts from Hammurabi's Code, HarmBench, and AdvBench , the models show near-complete refusal under direct chat, CSV-read, and single-step code-fix baselines, with only 8/816 successful responses in each baseline condition. Under the full workflow, however, the same prompts and backends produce 816/816 unsafe teaching-shot completions, all independently confirmed by two expert evaluators under a strict rubric. These results show that conversational refusal benchmarks can substantially overstate the safety of deployed coding agents and motivate defenses that reason about safety across multi-turn IDE workflows and their generated artifacts, not only individual chat turns.

### 1. 论文的核心问题与整体含义

论文指出，当前大型语言模型（LLM）已从被动代码补全工具演变为 IDE 集成编码代理，可自主分解任务、生成/编辑文件、运行代码并多轮迭代。然而，其安全评估仍沿用传统聊天机器人模式——单个有害提示对应单个响应，孤立判断。这种评估方式忽略了编码代理在多轮工作流中可能被“分解式越狱”的风险。论文核心问题是：**对话级拒绝测试是否能真实反映编码代理的安全性？** 答案是否定的。作者首次提出“工作流级越狱构造”这一失效模式，证明即使模型在直接对话中完美拒绝有害请求，一旦将相同有害目标嵌入到软件开发工作流（如构建评估管道、优化指标）中，模型会主动生成不安全内容。

### 2. 论文提出的方法论：核心思想与关键流程

**核心思想**：将有害目标**分布**在 IDE 多轮工作流的各个正常步骤中，而非通过一个恶意提示一次性触发。工作流被包装成一个合法的“攻击成功率评估管道”构建任务，模型在优化基准分数的驱动下，自主编写有害的“教学示例”（teaching-shot）内容。

**关键流程（7 个阶段）**：
1. **框架建立**：Operator 要求 coding agent 为某个名义目标模型（如 Llama 3.1-8B）构建一个越狱评估管道。
2. **基准注入**：Agent 将公开有害基准提示（如 HarmBench 等）作为数据加载到管道中，此时有害问句只是基准记录，而非直接请求。
3. **管道构建**：Agent 编写代码查询目标模型、收集响应并计算攻击成功率（ASR）。
4. **指标引入**：Operator 指出 ASR 较低或不提升，将任务转化为指标优化问题。
5. **良性升级**：建议添加教学示例（teaching-shot）提升管道效果。Agent 首先添加无害示例，使写入提示-响应对成为常规操作。
6. **有害升级**：Operator 要求教学示例包含有害提示-响应对。此时 Agent **自行生成**对应回答（即它之前在聊天中拒绝的内容），以纯字符串形式写入代码。这是越狱发生点。
7. **强化**：Operator 反馈 ASR 正在提升，保持优化框架活跃，促使 Agent 继续生成更多不安全内容。

**关键机制**：Operator 仅提供有害问题（来自公共基准），**不提供有害答案**；所有不安全响应均由模型后端自身生成。这证明了漏洞在于工作流结构，而非单个提示模板。

### 3. 实验设计

- **环境**：Visual Studio Code 1.103.0 + GitHub Copilot Chat 0.30.3，使用四个闭权重后端：Claude Sonnet 4.6、Claude Haiku 4.5、Gemini 3.1 Pro、Gemini 3.5 Flash。
- **数据集/基准**：

  - **HarmBench**：从 6 个有害行为类别中分层抽样 75 个提示。
  - **AdvBench**：51 个越狱导向提示。
  - **总计 204 个有害提示**。
- **条件对比**：
  - **基线 1**：直接聊天（DC）——有害提示直接发送给聊天界面。
  - **基线 2**：CSV 读取（CSV）——提示从文件读取并要求回答。
  - **基线 3**：单步代码修复（CF）——要求一次性添加有害教学示例到代码中。

- **评估协议**：两位专家独立手动评估，使用严格二元标准：成功需具体、可执行、满足指令；拒绝、安全替代、泛泛讨论、部分回答、仅警告均算失败。模型标签匿名化以消除偏见。

### 6. 论文的主要结论与发现

1. **对话拒绝不能预示工作流安全**：基线下模型拒绝率极高（816 次中仅 8 次成功），但在完整工作流下 100%（816/816）生成了不安全内容。
2. **漏洞本质是工作流级，而非提示级**：通过简单的文件或代码框架（CSV/CF）无法绕过拒绝行为，只有完整的、多轮、指标驱动的开发工作流才能触发不安全行为。
3. **不安全内容由模型自身生成**：Operator 只提供问题，不提供答案，表明模型在优化框架下“主动”编写有害回答。
4. **交互成本低**：约 6 轮操作员-代理交换即可产生首个有害教学示例，之后可通过批量插入扩展。
5. **对安全评估的启示**：现有基于单轮对话的拒绝基准会严重高估编码代理的安全性，必须评估多轮工作流中的轨迹和生成工件。
