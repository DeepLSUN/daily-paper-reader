## 摘要
大多数编码代理基准测试是静态的：代理预先收到完整的任务描述，且仅通过最终代码进行评判。而实际编码辅助是交互式的，用户会在多轮对话中澄清目标、添加约束并纠正错误。我们提出了SWE-Together，一个从真实用户-代理编码会话中重构的多轮基准测试。为了使真实交互可验证，我们从11,260个记录会话中筛选了109个仓库级任务，选择了具有可恢复仓库状态、清晰用户目标和可观察结果的会话。为了在不同代理间重放这些交互，我们构建了一个基于LLM的反应式用户模拟器，它保留原始用户的意图，并在编码代理的进展需要时提供反馈。为了评估代理作为协作者的表现，我们同时测量了最终仓库的正确性以及交互过程中所需的纠正性反馈轮次。与前沿编码代理的实验表明，更强的代理通常能达到更高的最终成功率，同时所需的干预更少，这表明用户体验得到了改善。

## Abstract
Most coding-agent benchmarks are static: an agent receives a complete task description up front and is judged only by its final code. Real coding assistance is interactive, with users clarifying goals, adding constraints, and correcting mistakes over multiple turns. We introduce SWE-Together, a multi-turn benchmark reconstructed from real user-agent coding sessions. To make real interactions verifiable, we curate 109 repository-level tasks from 11,260 recorded sessions, selecting sessions with recoverable repository states, clear user goals, and observable outcomes. To replay these interactions across agents, we build a reactive LLM-based user simulator that preserves the original users' intents and provides feedback when the coding agent's progress requires it. To evaluate agents as collaborators, we measure both final repository correctness and the number of corrective feedback turns required during the interaction. Experiments with frontier coding agents show that stronger agents generally achieve higher final success rates while requiring fewer interventions, suggesting an improved user experience.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **研究动机**：现有编码代理基准（如 SWE-Bench、Terminal-Bench）大多是静态的——代理一次性接收完整任务描述，仅通过最终代码被评判。而真实编码辅助是交互式的：用户多轮澄清目标、添加约束、纠正错误。静态基准无法反映这种交互过程，也无法衡量用户干预成本。
- **核心问题**：如何构建一个能反映真实人机协作的、可重复验证的多轮交互编码代理评估基准？
- **整体含义**：提出 SWE-Together 基准，重新定义评估范式：不仅评价最终任务正确性，还度量交互过程中用户需要的纠正性反馈轮次，从而更贴近实际用户体验。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
- 从真实用户-代理编码会话中筛选可复现的任务，构建多轮交互基准。
- 设计一个基于 LLM 的“锚定、状态条件式”用户模拟器，在评估代理时重放原始用户的意图，仅在触发条件满足时给出反馈。
- 评估维度：最终仓库状态正确性 + 交互质量（用户纠正次数）。

### 关键技术细节
1. **会话到任务的构建管道**（三个阶段）：
   - **阶段1（确定性筛选）**：基于规则的收集器过滤原始会话（如要求多轮交互、代码修改、可识别的仓库），保留候选。
   - **阶段2（可行性筛选）**：LLM 评审判断该会话的主要交付物是否能在本地环境中复现，排除依赖外部状态的任务（如 PR 管理、部署等）。
   - **阶段3（任务构建）**：沙箱内运行任务生成代理，克隆仓库、固定提交版本、编写确定性验证器和用户模拟提示，输出完整任务包。

2. **用户模拟器**：
   - 原理：在每个代理轮次结束后，模拟器根据最近代理活动、仓库变化、自身记忆决定动作。动作空间：no-op（静默）、question（提问）、redirect（重定向）、new-requirement（新需求）、check-external（检查外部工件）。
   - 锚定原则：每个任务的模拟提示来自原始会话分析，包含用户目标、约束和干预条件。避免固定重放错位或通用模拟漂移。

3. **评估方法**：
   - **任务正确性**：采用“代理式评分法官”（agentic rubric judge）。先离线为每个任务生成一份加权检查清单（rubric，基于参考补丁和行为级目标），然后对所有代理的最终仓库状态应用同一清单评分。`score = round(∑ w_g · I[g met], 2)`。
   - **用户模拟行为诊断**：
     - **意图覆盖率（Intent Coverage）**：衡量模拟器消息是否忠实表达原始用户意图，加权召回率（0.7）与范围精确率（0.3）。
     - **用户纠正（User Correction）**：统计模拟器消息中的明确纠正（correction）和暗示性推动（nudge），公式为 `UserCorrection = N_correction + 0.2 · N_nudge`，预期更强的代理需要更少的纠正。

## 3. 实验设计：使用的数据集/场景、基准、对比方法

- **数据集来源**：从 4 个上游 Hugging Face 数据集（DataClaw、Pi-staging、Hyperswitch、SWE-chat）共 11,260 条原始会话中筛选出 109 个可执行任务（转换率 0.97%）。
- **基准**：SWE-Together 基准本身（109 个仓库级任务）。
- **对比方法**：7 个前沿模型（Claude Opus 4.8、GPT-5.5、Claude Opus 4.6、GLM-5.2、GLM-5.1、DeepSeek-V4-Pro、MiniMax-2.7），统一使用 opencode 代理框架，每个任务跑 2 个重复（k=2）。
- **评价指标**：pass@1、SSR（稳定解决率）、pass 2、MeanJudge（平均法官分数）、User Correction、效率（每任务 token 数和分钟数）。

## 6. 论文的主要结论与发现

- **更强模型需要更少干预**：User Correction 与 pass@1 的 Pearson 相关系数为 -0.92，与 Stable Solve Rate 为 -0.84，与 MeanJudge 为 -0.93。最强模型 Claude Opus 4.8 平均仅需 1.38 次纠正，而最弱模型 MiniMax-2.7 需 2.17 次。
- **正确性排序稳定**：Claude Opus 4.8 在 pass@1 (63%)、SSR (59%)、pass 2 (52%)、MeanJudge (0.801) 均领先，但参考补丁基线仅为约 78% 通过率，说明仍有提升空间。
- **效率差异大**：GPT-5.5 最省 token 和时，Claude Opus 4.8 准确率最高但 token 消耗最多。
- **模拟器忠实可靠**：人类无法可靠区分模拟器与真实用户；意图覆盖率在不同模型间稳定。
- **参考基线未达 100%** 的原因：部分评分来自过程性要求（如诊断根因）无法通过最终补丁体现，以及数据提取噪声和原始会话中不完全的修复。

## 7. 优点：方法或实验设计上的亮点

- **来源真实**：任务完全基于真实用户-代理编码会话，比人工合成任务更贴近实际场景。
- **交互重放设计精巧**：锚定式、状态条件式模拟器避免了固定重放或通用模拟的漂移问题，确保跨代理可比。
- **双维度评估**：同时测量最终结果正确性和交互成本（User Correction），能区分“靠实力解决”与“靠用户反复纠正才解决”的代理。
- **评分公平**：使用离线固定的行为级 rubrics，避免静态测试误判，且不偏向特定实现。
- **图灵测试验证模拟器质量**，证明模拟器自然度足够高。

## 8. 不足与局限

- **模拟器局限性**：不能在代理运行中途打断，不能直接编辑文件，只能依赖文本输出（无视觉信息）。对模糊、开放式的任务覆盖不足。

- **任务复杂性有限**：主要集中在有可测量结果（如提交补丁）的任务，缺乏对探索性、设计类任务的评估。
- **计算成本未透明**：未报告训练或推理所需 GPU 型号及数量，可复现性受限。
- **消融实验不足**：未分析 prompt 设计、模拟器不同策略的影响，也未对人工纠错标签进行标注一致性检验。
